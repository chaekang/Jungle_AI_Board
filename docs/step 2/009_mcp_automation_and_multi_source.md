# 009_mcp_automation_and_multi_source

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- MCP 연동 코드는 `apps/fastapi-api/app/routers`, `apps/fastapi-api/app/services`, `apps/fastapi-api/app/schemas`에 나눠 둔다.
- Nest에서 MCP 호출을 감싸야 하면 `apps/nest-api/src/mcp` 같은 도메인 폴더를 만든다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 목적

이 문서는 2차 구현에서 MCP 연동을
"수동 조회 + 짧은 캐시" 수준에서
"자동 갱신 + 복수 소스 관리" 수준으로 올리는 작업을 다룬다.

관련 1차 문서:

- [docs/step 1/010_mcp_integration.md](../step%201/010_mcp_integration.md)

## 이 단계가 끝나면 되는 것

- 스케줄러 기반 자동 동기화가 된다
- 복수 외부 소스를 같은 내부 모델에 매핑할 수 있다
- 실패 시 재시도와 상태 추적이 가능해진다

## 이번 단계에서 구현하는 것

- 동기화 잡 스케줄링
- 외부 소스별 어댑터 구조
- 동기화 이력 저장
- 실패 재시도 또는 수동 재실행
- 충돌 해결 규칙

## 권장 구현 순서

1. 외부 소스 어댑터 인터페이스 정리
2. 동기화 이력 테이블 추가
3. 스케줄러 잡 추가
4. 재시도 / 알림 / 실패 상태 처리
5. 복수 소스 우선순위 정책 정리

## 중요한 포인트

### 1. 소스 추가보다 내부 표준 모델이 먼저다

여러 소스를 붙이면 가장 먼저 흔들리는 것이 내부 데이터 기준이다.
따라서 2차에서는 반드시 아래를 먼저 고정한다.

- 내부 표준 극장 키
- 내부 표준 층 / 구역 표현
- 데이터 신뢰 우선순위

### 2. 자동화에는 관측 포인트가 필요하다

동기화 자동화는 "돌아간다"보다
"어디서 실패했는지 보인다"가 더 중요하다.

## 체크리스트

- [ ] 스케줄러 기반 동기화 구현
- [ ] 외부 소스 어댑터 구조 분리
- [ ] 동기화 이력 저장
- [ ] 실패 재실행 경로 추가
- [ ] 충돌 해결 규칙 문서화

## 완료 기준

- 수동 갱신에 덜 의존한다
- 외부 데이터 품질을 더 안정적으로 유지할 수 있다
- 소스가 늘어나도 구조가 크게 흔들리지 않는다
