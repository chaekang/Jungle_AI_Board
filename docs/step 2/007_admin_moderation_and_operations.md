# 007_admin_moderation_and_operations

## 목적

이 문서는 2차 구현에서 운영자 관점 기능을 붙이는 작업 가이드다.

1차와 2차 초반까지는 사용자 기능 중심이었다면,
이 단계부터는 신고, 검수, 정리, 관리 기능을 붙여 서비스 운영성을 높인다.

관련 1차 문서:

- [docs/step 1/004_seat_review_crud.md](../step%201/004_seat_review_crud.md)
- [docs/step 1/005_comments.md](../step%201/005_comments.md)

## 이 단계가 끝나면 되는 것

- 운영자가 후기와 댓글을 검토할 수 있다
- 신고 / 숨김 / 복구 흐름이 생긴다
- 태그와 메타데이터를 정리할 수 있다

## 이번 단계에서 구현하는 것

- 관리자 권한 모델
- 신고 기능
- 운영자 검수 상태 변경
- 숨김 / 복구 / 강제 삭제 흐름
- 간단한 운영 대시보드

## 권장 구현 순서

1. 관리자 역할 정의
2. 신고 데이터 모델 설계
3. 후기 / 댓글 신고 API 추가
4. 운영자 검수 API와 화면 추가
5. 정리 대상 목록 화면 구성

## 중요한 포인트

### 1. 소프트 삭제를 운영 기능과 연결한다

1차에서 넣은 `deletedAt`은 2차 운영 기능과 연결될 때 진짜 가치가 생긴다.

예:

- 신고 누적으로 숨김
- 운영자 복구
- 강제 삭제 기록 보존

### 2. 운영 기능은 일반 사용자 흐름과 분리한다

관리자 전용 API, 관리자 전용 화면, 감사 로그를 나누는 편이 좋다.

## 체크리스트

- [ ] 관리자 권한 정의
- [ ] 신고 테이블 및 API 구현
- [ ] 검수 / 숨김 / 복구 플로우 구현
- [ ] 운영 대시보드 추가
- [ ] 감사 로그 범위 정리

## 완료 기준

- 문제가 있는 후기 / 댓글을 운영자가 제어할 수 있다
- 소프트 삭제 구조를 운영 기능과 연결할 수 있다
- 서비스 품질 유지가 수작업만으로도 가능해진다
