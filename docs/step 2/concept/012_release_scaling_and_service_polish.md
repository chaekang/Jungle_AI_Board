# 012_release_scaling_and_service_polish_concepts

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- Nest 새 기능은 `apps/nest-api/src/<domain>` 형태의 도메인 폴더에 둔다.
- React 새 기능은 `apps/web-react/src/features/<feature>` 형태의 기능 폴더에 둔다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 이 문서의 역할

이 문서는 `012_release_scaling_and_service_polish` 단계에서 필요한
배포 마감, 성능, 스케일링, 서비스 완성도 개념을 정리한 문서다.

2차 마지막 단계는 새 기능을 더 붙이는 것이 아니라
지금까지 만든 기능을 실제 서비스처럼 다듬는 단계다.

## 핵심 개념

### `Production Readiness`

정의:

서비스가 실제 사용자 환경에서
안정적으로 동작할 준비가 되었는지를 판단하는 기준이다.

포함 요소:

- 보안 설정
- 환경 변수 관리
- 에러 처리
- 운영 절차
- 성능

### `Scalability`

정의:

사용자 수, 데이터 양, 요청 수가 늘어나도
시스템이 감당할 수 있는 능력이다.

언제 보는가:

- 검색 요청 증가
- 후기 데이터 증가
- RAG 문서 증가
- 외부 연동 호출 증가

### `Performance Bottleneck`

정의:

전체 응답 속도나 처리량을 제한하는 병목 구간이다.

예:

- DB 쿼리
- 검색 쿼리
- 임베딩 조회
- 외부 API 호출
- 프론트 렌더링

### `Service Polish`

정의:

기능 자체는 돌아가지만
사용자 경험과 운영 경험이 아직 거친 부분을 다듬는 작업이다.

예:

- 로딩 메시지
- 실패 안내
- 빈 상태 화면
- 용어 일관성
- 데모 시나리오 정리

### `Deployment Checklist`

정의:

배포 전에 반드시 확인해야 하는 항목 목록이다.

예:

- 시크릿 설정
- DB 마이그레이션
- CORS / cookie 설정
- 로그 레벨
- 관리자 계정

## 면접 포인트

"프로젝트 후반에는 기능 추가보다 운영성과 일관성이 더 중요합니다. 실제로는 release polish 단계에서 보안 설정, 실패 경험, 배포 체크리스트를 정리해야 서비스 완성도가 크게 올라갑니다."

## 한 줄 정리

2차 마감 단계의 핵심은
"기능을 더 많이 넣는 것"이 아니라
"이미 만든 기능을 실제 서비스 수준으로 정리하는 것"이다.
