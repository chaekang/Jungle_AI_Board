# 002_extended_domain_modeling_concepts

## 이 문서의 역할

이 문서는 `002_extended_domain_modeling` 단계에서 필요한
확장 도메인 모델링 개념을 정리한 문서다.
1차에서 극장, 작품, 공연, 좌석, 후기 중심으로 최소 구조를 만들었다면,
2차에서는 배우, 배역, 캐스트, 후기 포커스 같은 더 세밀한 도메인 개념을 모델에 반영한다.

## 왜 2차에서 하는가

1차에서는 CRUD와 검색이 먼저 돌아야 하므로
도메인을 지나치게 잘게 쪼개면 구현이 무거워진다.
하지만 RAG, 추천, 캐스트 기반 검색까지 생각하면
2차에서는 더 정밀한 도메인 구조가 필요하다.

## 핵심 개념

### `Domain Modeling`

정의:

서비스가 다루는 실제 세계의 개체와 관계를
DB와 코드 구조로 옮기는 작업이다.

언제 쓰는가:

- 테이블을 늘릴지 말지 결정할 때
- 자유 텍스트를 구조화할지 고민할 때
- 검색과 추천에 쓸 데이터 기준을 정할 때

어떻게 적용하는가:

- "어떤 질문에 답해야 하는가"를 먼저 본다
- 그 질문을 안정적으로 답하려면 어떤 엔티티가 필요한지 정한다
- 너무 이른 추상화와 너무 늦은 구조화를 모두 피한다

장점:

- 기능 추가 시 재사용성이 높아진다
- 검색 / 추천 / AI 품질이 좋아진다

문제 / 주의점:

- 초기에 과도하게 세분화하면 개발 속도가 급격히 느려진다
- 반대로 너무 느슨하면 나중에 데이터 정제가 어려워진다

### `Character`, `Actor`, `Cast`

정의:

- `Character`: 극 중 배역
- `Actor`: 실제 배우
- `Cast`: 특정 공연 또는 회차에서 어떤 배우가 어떤 배역을 맡는지에 대한 연결 정보

언제 쓰는가:

- "어떤 배우가 나온 회차 후기만 보고 싶다"
- "특정 배역 체감이 좋은 좌석을 찾고 싶다"
- 공연 메타데이터와 후기 내용을 더 정밀하게 연결하고 싶다

어떻게 사용하는가:

- `characters`, `actors`, `performance_casts` 같은 테이블로 분리한다
- 표시 이름과 정규화 키를 나눈다
- 공연 회차와 캐스트 연결을 후기 메타데이터에 연결한다

장점:

- 배우 / 배역 조건 검색이 가능해진다
- 후기의 맥락 정보가 풍부해진다
- 추천 이유가 더 정교해진다

문제 / 주의점:

- 데이터 수집 비용이 커진다
- 시즌별 캐스트 변경이 잦아 관리가 어렵다

### `Normalization`

정의:

사람이 입력하거나 외부에서 수집한 흔들리는 값을
내부에서 비교 가능한 표준 형태로 바꾸는 작업이다.

언제 쓰는가:

- 배우 이름 표기가 제각각일 때
- 배역 이름에 띄어쓰기나 별칭 차이가 있을 때

어떻게 적용하는가:

- 표시용 이름과 정규화 키를 분리한다
- 동의어 매핑 표를 둔다
- 외부 수집 데이터에도 같은 규칙을 적용한다

장점:

- 검색 정확도가 높아진다
- 중복 데이터가 줄어든다

문제 / 주의점:

- 과도한 정규화는 표시 품질을 해칠 수 있다
- 관리 도구가 없으면 운영자가 수작업으로 고생한다

### `Metadata Enrichment`

정의:

후기 본문만으로는 부족한 맥락을
별도 필드와 관계 데이터로 보강하는 작업이다.

예:

- 배우 / 배역 정보
- 시야 방해 포인트
- 특정 장면 체감
- 회차별 차이

장점:

- RAG 문서 품질 향상
- 추천 점수 계산에 활용 가능
- 검색 필터의 세밀도 향상

## 면접 포인트

"1차에서는 리뷰 CRUD를 우선했기 때문에 배우 / 배역을 자유 텍스트에 일부 남겨도 괜찮았습니다. 하지만 2차에서 검색과 RAG 정확도를 높이려면 actor, character, cast를 구조화해야 장기적으로 유지보수가 쉬워집니다."
