# 004_advanced_comments_and_interactions_concepts

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- Nest 댓글 도메인 코드는 `apps/nest-api/src/comments` 같은 도메인 폴더에 둔다.
- React 댓글 전용 API, 훅, 컴포넌트는 `apps/web-react/src/features/comments`에 둔다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 이 문서의 역할

이 문서는 `004_advanced_comments_and_interactions` 단계에서 다룰
대댓글, 멘션, 좋아요, 실시간성 같은 상호작용 개념을 정리한 문서다.

1차 댓글은 "질문과 답변이 가능한 최소 구조"였다면,
2차 댓글은 "사용자 참여가 쌓이는 커뮤니티 구조"에 가깝다.

## 핵심 개념

### `Threaded Comment`

정의:

댓글이 부모-자식 관계를 가지며
대화 흐름을 계층적으로 표현하는 구조다.

언제 쓰는가:

- 특정 댓글에 직접 답글을 달고 싶을 때
- 긴 토론을 하나의 평면 목록으로 두기 어려울 때

어떻게 구현하는가:

- `parent_comment_id`를 둔다
- 응답 시 트리 형태로 묶거나 2단 구조로 반환한다
- 깊이 제한을 둔다

장점:

- 대화 맥락이 명확해진다
- 질문-답변 구조가 자연스러워진다

문제 / 주의점:

- UI가 복잡해진다
- 정렬과 페이징이 어려워진다
- 무한 중첩은 유지보수 비용이 크다

### `Mention`

정의:

댓글 본문에서 특정 사용자를 지목하는 기능이다.

언제 쓰는가:

- 질문에 답한 사람을 다시 호출할 때
- 특정 사용자에게 답변 맥락을 연결할 때

어떻게 사용하는가:

- `@nickname` 같은 패턴 파싱
- 사용자 식별자 연결
- 렌더링 시 링크 처리
- 알림 시스템과 연결 가능

장점:

- 상호작용 밀도가 올라간다
- 대화 참여자를 더 명확히 부를 수 있다

문제 / 주의점:

- 닉네임 변경 시 연결 정책이 필요하다
- 무분별한 멘션은 스팸이 될 수 있다

### `Reaction` / `Like`

정의:

좋아요나 간단한 반응을 통해
텍스트를 쓰지 않고도 피드백을 남기는 기능이다.

언제 쓰는가:

- 유용한 댓글을 빠르게 표시하고 싶을 때
- 인기 댓글, 공감 댓글을 잡고 싶을 때

장점:

- 참여 장벽이 낮다
- 랭킹 신호로도 쓸 수 있다

문제 / 주의점:

- 인기 편향이 강해질 수 있다
- 다중 클릭 방지, 취소 정책이 필요하다

### `Realtime Update`

정의:

새 댓글, 좋아요, 상태 변경을
새로고침 없이 즉시 반영하는 구조다.

방법:

- 짧은 주기 폴링
- SSE
- WebSocket

장점:

- 대화형 서비스 느낌이 강해진다

문제 / 주의점:

- 인프라와 상태 관리가 복잡해진다
- 꼭 실시간이 필요한지 먼저 따져야 한다
