# 010_agent_orchestration_and_memory_concepts

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- Agent API와 실행 로직은 `apps/fastapi-api/app/routers`, `apps/fastapi-api/app/services`, `apps/fastapi-api/app/schemas`에 나눠 둔다.
- React 에이전트 화면이 필요하면 `apps/web-react/src/features/agent`에 둔다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 이 문서의 역할

이 문서는 `010_agent_orchestration_and_memory` 단계에서 필요한
에이전트 오케스트레이션, 대화 상태, 사용자 선호 기억 개념을 정리한 문서다.

1차 에이전트가 좌석 추천 흐름을 한 번 실행하는 구조였다면,
2차 에이전트는 여러 조건을 대화로 조정하고
도구 호출 순서를 안정적으로 제어하는 구조로 올라간다.

## 핵심 개념

### `Orchestration`

정의:

여러 도구와 단계가 있는 시스템에서
무엇을 어떤 순서로 호출할지 제어하는 작업이다.

에이전트에서 왜 중요한가:

- 검색, RAG, MCP, 랭킹이 순서 없이 섞이면 결과가 불안정해진다
- 비용과 응답 속도도 제어해야 한다

### `Conversation State`

정의:

현재 대화에서 사용자가 준 조건과 맥락을 저장한 상태다.

예:

- 예산
- 선호 층
- 선호 방향
- 동반 인원
- 배우 선호

장점:

- 사용자가 조건을 조금씩 수정해도 흐름을 이어갈 수 있다

### `Memory`

정의:

이전 대화나 사용자 선호를 다시 활용하기 위해 저장하는 정보다.

종류:

- 단기 메모리: 현재 세션에서만 유지
- 장기 메모리: 사용자 계정 단위로 저장

문제 / 주의점:

- 너무 많이 기억하면 오히려 오판할 수 있다
- 개인화는 편하지만 설명 가능성이 떨어질 수 있다

### `Fallback`

정의:

도구 호출 실패, 검색 실패, 조건 충돌 시
완전히 실패하지 않고 대체 응답을 주는 전략이다.

예:

- 조건 완화 제안
- 추가 질문
- 일반 후기 추천

### `Structured Reasoning Output`

정의:

에이전트가 최종 문장만 내는 것이 아니라
근거, 제외 사유, 사용 조건을 구조 데이터로 함께 남기는 방식이다.

왜 중요한가:

- 디버깅이 쉬워진다
- 추천 이유를 UI에 따로 노출할 수 있다

## 면접 포인트

"에이전트 품질은 모델 성능만으로 결정되지 않고 orchestration에 크게 좌우됩니다. 어떤 순서로 조건을 파싱하고, 어떤 도구를 호출하고, 실패 시 어떻게 fallback하는지가 실제 사용자 경험을 결정합니다."

## 한 줄 정리

2차 에이전트 고도화의 핵심은
"더 똑똑한 모델"보다
"더 안정적인 도구 조합과 대화 상태 관리"다.
