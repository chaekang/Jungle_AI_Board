# 008_rag_quality_and_evaluation_concepts

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- RAG API와 실험 코드는 `apps/fastapi-api/app/routers`, `apps/fastapi-api/app/services`, `apps/fastapi-api/app/schemas`에 나눠 둔다.
- Nest에서 RAG 호출을 감싸야 하면 `apps/nest-api/src/rag` 같은 도메인 폴더를 만든다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 이 문서의 역할

이 문서는 `008_rag_quality_and_evaluation` 단계에서 필요한
RAG 품질 개선과 평가 개념을 정리한 문서다.

1차 RAG가 "답변이 나온다"에 집중했다면,
2차 RAG는 "왜 잘 나오고 왜 틀리는지 측정할 수 있다"에 초점을 둔다.

## 핵심 개념

### `Chunking`

정의:

긴 문서를 검색과 임베딩에 적합한 작은 단위로 나누는 작업이다.

왜 중요한가:

- 너무 크면 검색 정확도가 떨어진다
- 너무 작으면 문맥이 끊긴다

어떻게 조정하는가:

- 본문 길이
- 메타데이터 포함 여부
- 후기 단위 / 문단 단위 / 섹션 단위

### `Embedding Refresh`

정의:

원본 문서가 바뀌었거나 메타데이터가 추가됐을 때
임베딩을 다시 생성하는 과정이다.

언제 필요한가:

- 후기 수정
- 태그 정리
- 캐스트 정보 추가
- 청킹 전략 변경

### `Evaluation Set`

정의:

RAG 품질을 반복적으로 점검하기 위해
질문과 기대 결과를 모아 둔 테스트 집합이다.

예:

- 특정 배우가 잘 보이는 좌석 질문
- 사이드 좌석 장단점 질문
- 음향 우선 질문

장점:

- 모델이나 검색 전략을 바꿨을 때 회귀를 확인할 수 있다

### `Retrieval Failure Analysis`

정의:

답변 품질이 낮을 때
검색 단계에서 무엇이 잘못됐는지 분석하는 과정이다.

실패 원인 예:

- 문서 누락
- 메타데이터 부족
- 질의 정규화 실패
- 랭킹 기준 부적절

### `Observability for RAG`

정의:

RAG가 어떤 문서를 검색했고,
어떤 점수와 조건으로 답변했는지 추적 가능한 상태를 뜻한다.

왜 중요한가:

- LLM 결과만 보고는 디버깅이 어렵다
- 검색 문제인지 프롬프트 문제인지 분리해야 한다

## 면접 포인트

"RAG 품질은 프롬프트만 바꿔서 해결되지 않습니다. 문서 품질, 청킹, 메타데이터, 검색 전략, 평가셋이 같이 있어야 반복 개선이 가능합니다."

## 한 줄 정리

2차 RAG 품질 개선은
"답변을 더 그럴듯하게 만드는 일"이 아니라
"왜 그 답변이 나왔는지 개선 가능한 구조를 만드는 일"이다.
