# 009_mcp_automation_and_multi_source_concepts

## 이 문서의 역할

이 문서는 `009_mcp_automation_and_multi_source` 단계에서 필요한
자동 동기화, 복수 소스 통합, 어댑터 구조 개념을 정리한 문서다.

1차 MCP는 보조 조회 기능에 가깝지만,
2차 MCP는 외부 데이터를 운영 가능한 방식으로 받아오는 파이프라인에 가깝다.

## 핵심 개념

### `Scheduler`

정의:

정해진 주기에 따라 작업을 실행하는 구조다.

언제 쓰는가:

- 극장 정보 갱신
- 좌석 메타데이터 동기화
- 임베딩 재생성 배치

장점:

- 수동 작업을 줄일 수 있다

문제 / 주의점:

- 실패 시 추적이 필요하다
- 너무 잦으면 외부 API 제한에 걸릴 수 있다

### `Adapter Pattern`

정의:

외부 소스마다 다른 응답 형식을
내부 표준 인터페이스로 감싸는 구조다.

왜 중요한가:

- 외부 소스가 늘어나도 내부 코드를 덜 흔들 수 있다
- 테스트가 쉬워진다
- 특정 소스 교체 비용이 줄어든다

### `Source of Truth`

정의:

충돌하는 데이터가 있을 때
무엇을 최종 기준으로 삼을지 정한 원천 데이터 기준이다.

MCP에서 왜 중요한가:

- 외부 소스끼리 값이 다를 수 있다
- 내부 DB와 외부 데이터가 충돌할 수 있다

### `Sync Job History`

정의:

동기화 작업이 언제 실행됐고,
성공했는지 실패했는지, 몇 건을 반영했는지 남기는 기록이다.

장점:

- 운영 추적 가능
- 재시도 판단 가능
- 문제 구간 파악 가능

### `Conflict Resolution`

정의:

서로 다른 데이터 소스가 상충할 때
어떤 값을 채택할지 정하는 규칙이다.

예:

- 최신 값 우선
- 신뢰도 높은 소스 우선
- 내부 수동 수정 우선

## 면접 포인트

"복수 외부 소스를 붙일 때 핵심은 API 호출 코드가 아니라 내부 표준 모델과 source of truth를 먼저 정하는 것입니다. 그렇지 않으면 소스가 늘수록 데이터 충돌이 커집니다."

## 한 줄 정리

2차 MCP 자동화는
"외부 데이터 조회 기능"을
"운영 가능한 동기화 시스템"으로 바꾸는 단계다.
