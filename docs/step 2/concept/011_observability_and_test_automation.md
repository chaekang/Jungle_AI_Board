# 011_observability_and_test_automation_concepts

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- 테스트나 관측 코드를 추가할 때도 기능별 소스는 `apps/web-react/src/features/*`와 `apps/nest-api/src/<domain>` 위치를 유지한다.
- 공통 테스트 헬퍼가 필요할 때만 앱별 테스트 유틸 폴더로 분리한다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 이 문서의 역할

이 문서는 `011_observability_and_test_automation` 단계에서 필요한
관측 가능성, 통합 테스트, 회귀 테스트 개념을 정리한 문서다.

기능이 많아질수록 "만드는 능력"보다
"깨졌을 때 빨리 알아차리는 능력"이 중요해진다.

## 핵심 개념

### `Observability`

정의:

시스템 내부 상태를
로그, 메트릭, 트레이스 같은 신호를 통해 이해할 수 있는 능력이다.

왜 중요한가:

- 장애 원인을 빨리 파악할 수 있다
- AI 파이프라인 문제도 추적할 수 있다

### `Integration Test`

정의:

여러 구성 요소가 함께 동작하는지 확인하는 테스트다.

예:

- 인증 후 후기 작성
- 댓글 작성 후 조회
- 태그 연결 후 검색

장점:

- 실제 사용자 흐름에 가깝다
- 모듈 간 연결 오류를 빨리 잡는다

### `E2E Test`

정의:

브라우저나 실제 사용자 행동에 가까운 방식으로
처음부터 끝까지 흐름을 검증하는 테스트다.

언제 중요한가:

- 로그인
- 후기 작성
- 검색
- 에이전트 요청

### `Regression Test`

정의:

이전에 잘 되던 기능이
새 변경 후에도 계속 잘 되는지 확인하는 테스트다.

AI 기능에서는 왜 중요한가:

- 프롬프트, 검색 전략, 모델 변경으로 품질이 쉽게 흔들린다

### `Correlation ID` / `Request ID`

정의:

하나의 요청 흐름을 여러 로그 사이에서 추적하기 위한 식별자다.

왜 중요한가:

- 프론트 요청
- 백엔드 처리
- 외부 도구 호출
- RAG 검색

이 흐름을 한 묶음으로 볼 수 있다.

## 면접 포인트

"기능이 늘어날수록 단위 테스트만으로는 부족하고, 핵심 사용자 시나리오 기준 통합 테스트와 E2E가 중요해집니다. AI 기능은 정답이 고정되지 않는 경우가 많아서 regression 질문셋을 따로 두는 게 효과적입니다."

## 한 줄 정리

관측 가능성과 테스트 자동화는
"잘 만들었는지 확인하는 단계"가 아니라
"계속 망가지지 않게 유지하는 기반"이다.
