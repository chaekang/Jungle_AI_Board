# 010_agent_orchestration_and_memory

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- Agent API와 실행 로직은 `apps/fastapi-api/app/routers`, `apps/fastapi-api/app/services`, `apps/fastapi-api/app/schemas`에 나눠 둔다.
- React 에이전트 화면이 필요하면 `apps/web-react/src/features/agent`에 둔다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

"단발성 추천"에서 "대화형 보조 시스템"으로 올리는 작업을 다룬다.

## 이 단계가 끝나면 되는 것

- 여러 조건을 대화로 보정할 수 있다
- 추천 과정의 근거를 더 명확히 보여줄 수 있다
- 에이전트가 도구 호출 순서를 안정적으로 제어할 수 있다

## 이번 단계에서 구현하는 것

- 대화형 상태 관리
- 추천 근거 구조화
- 도구 호출 순서 개선
- 사용자 선호 메모리
- 실패 시 fallback 응답 전략

## 권장 구현 순서

1. 사용자 입력 상태 모델 정의
2. 대화형 슬롯 또는 조건 누적 구조 설계
3. 도구 호출 순서 규칙화
4. 추천 근거와 주의사항 포맷 정의
5. 선호 저장 또는 최근 조건 기억 기능 추가

## 중요한 포인트

### 1. 기억은 짧고 명확해야 한다

2차에서는 장기 기억보다 아래 정도가 현실적이다.

- 최근 좌석 선호
- 선호 층 / 방향
- 예산 범위
- 동반 인원 특성

### 2. 근거를 문장과 구조 데이터로 같이 둔다

에이전트 답변은 자연어만 있으면 디버깅이 어렵다.
가능하면 아래를 같이 보관한다.

- 사용한 조건
- 조회한 데이터 소스
- 추천 이유
- 제외 이유

## 체크리스트

- [ ] 대화형 상태 모델 정의
- [ ] 사용자 선호 기억 구조 추가
- [ ] 추천 근거 포맷 확정
- [ ] fallback 응답 전략 구현
- [ ] 도구 호출 순서 검증

## 완료 기준

- 사용자가 조건을 여러 번 바꿔도 흐름이 안정적이다
- 추천 이유를 설명할 수 있다
- 실패했을 때도 납득 가능한 대안 응답을 줄 수 있다
