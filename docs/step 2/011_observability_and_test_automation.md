# 011_observability_and_test_automation

## 목적

이 문서는 2차 구현에서 테스트, 로그, 모니터링을 정리해
기능이 많아진 상태에서도 안정적으로 유지할 수 있게 만드는 작업을 다룬다.

관련 1차 문서:

- [docs/step 1/008_core_feature_integration_check.md](../step%201/008_core_feature_integration_check.md)
- [docs/step 1/012_final_integration_and_demo.md](../step%201/012_final_integration_and_demo.md)

## 이 단계가 끝나면 되는 것

- 핵심 흐름 자동 테스트가 생긴다
- 에러와 느린 구간을 추적할 수 있다
- AI 기능 품질 회귀를 더 빨리 발견할 수 있다

## 이번 단계에서 구현하는 것

- API 통합 테스트
- 핵심 프론트 플로우 E2E
- 구조화 로그
- 에러 추적 포인트
- RAG / Agent 회귀 테스트용 샘플 질문

## 권장 구현 순서

1. 핵심 사용자 시나리오 정리
2. 백엔드 통합 테스트 추가
3. 프론트 핵심 E2E 추가
4. 로그 포맷과 요청 ID 정리
5. AI 응답 회귀 확인 시나리오 추가

## 중요한 포인트

### 1. 모든 기능을 다 자동화하려 하지 않는다

우선순위는 아래처럼 잡는 편이 좋다.

- 인증
- 후기 작성 / 조회
- 댓글 상호작용
- 검색
- RAG 질문 2~3개
- 에이전트 추천 2~3개

### 2. 로그는 운영과 개발이 같이 읽을 수 있어야 한다

최소한 아래는 남기는 편이 좋다.

- 요청 ID
- 사용자 ID
- 주요 파라미터
- 실패 이유
- 외부 도구 호출 여부

## 체크리스트

- [ ] 백엔드 통합 테스트 추가
- [ ] 프론트 E2E 추가
- [ ] 요청 ID 기반 로그 정리
- [ ] AI 기능 회귀 질문셋 추가
- [ ] 장애 추적 포인트 문서화

## 완료 기준

- 주요 회귀를 수동 테스트 전에 잡을 수 있다
- 장애 원인을 더 빠르게 파악할 수 있다
- 기능이 늘어나도 배포 불안이 줄어든다
