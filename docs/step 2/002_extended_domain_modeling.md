# 002_extended_domain_modeling

## 목적

이 문서는 2차 구현에서
1차의 핵심 테이블 위에 배역, 배우, 캐스트, 문서 메타데이터 같은
확장 도메인 모델을 붙이는 작업 가이드다.

관련 1차 문서:

- [docs/step 1/001_project_setup_and_db_design.md](../step%201/001_project_setup_and_db_design.md)
- [docs/step 1/003_theater_performance_seat_metadata.md](../step%201/003_theater_performance_seat_metadata.md)

## 이 단계가 끝나면 되는 것

- `characters`, `actors`, `performance_casts` 구조가 들어간다
- 공연별 캐스트 정보를 후기와 더 정확히 연결할 수 있다
- RAG와 추천에서 배역 / 배우 조건을 더 정밀하게 쓸 수 있다

## 이번 단계에서 구현하는 것

- `characters`
- `actors`
- `performance_casts`
- 후기와 캐스트 조건 연결용 필드 또는 연결 테이블
- 배우 / 배역 이름 정규화 규칙

## 이번 단계에서 구현하지 않는 것

- 복잡한 공연 편성 캘린더 자동 수집
- 외부 예매처 전체와의 실시간 동기화

## 권장 구현 순서

1. 1차 스키마와 충돌 없는 관계 설계
2. `musicals`, `performances`와 캐스트 연결
3. 후기에서 배우 / 배역 체감 데이터를 어디까지 구조화할지 결정
4. 검색과 RAG에 쓸 메타데이터 필드 추가
5. 샘플 데이터와 시드 정리

## 중요한 포인트

### 1. 이름 중복과 표기 흔들림을 먼저 잡는다

배우 / 배역 데이터는 문자열 흔들림이 심하다.

예:

- 본명 / 활동명
- 띄어쓰기 차이
- 시즌별 다른 표기

따라서 2차에서는 표시용 이름과 정규화 키를 같이 관리하는 편이 좋다.

### 2. 후기와 캐스트의 연결 강도를 정한다

후기가 특정 공연 회차를 가리키면
캐스트 데이터와 연결할 수 있는 정보가 강해진다.

최소한 아래 중 하나는 정해야 한다.

- 후기에서 `performance_id`를 사실상 필수에 가깝게 올릴지
- 캐스트 자유 메모를 계속 허용할지
- 구조화된 캐스트 연결을 우선할지

## 체크리스트

- [ ] 캐스트 관련 테이블 추가
- [ ] 배우 / 배역 이름 정규화 규칙 확정
- [ ] 후기와 캐스트 연결 규칙 확정
- [ ] 시드 데이터 준비
- [ ] 검색 / RAG 메타데이터 반영 포인트 정의

## 완료 기준

- 특정 배우 / 배역 기준 후기를 더 정확히 묶을 수 있다
- 후기 데이터와 공연 메타데이터가 더 촘촘하게 연결된다
- 이후 추천과 RAG 정확도를 높일 기반이 생긴다
