# 008_rag_quality_and_evaluation

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- RAG API와 실험 코드는 `apps/fastapi-api/app/routers`, `apps/fastapi-api/app/services`, `apps/fastapi-api/app/schemas`에 나눠 둔다.
- Nest에서 RAG 호출을 감싸야 하면 `apps/nest-api/src/rag` 같은 도메인 폴더를 만든다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 목적

이 문서는 2차 구현에서 RAG를
"일단 돌아가는 답변"에서 "품질을 측정하고 조정할 수 있는 구조"로 올리는 작업을 다룬다.

관련 1차 문서:

- [docs/step 1/009_rag.md](../step%201/009_rag.md)

## 이 단계가 끝나면 되는 것

- 문서 분할과 메타데이터 전략이 정교해진다
- 검색 실패 원인을 추적할 수 있다
- RAG 품질을 반복 개선할 수 있다

## 이번 단계에서 구현하는 것

- 문서 청킹 전략 개선
- 메타데이터 강화
- 임베딩 재생성 파이프라인
- 질의 유형별 평가셋
- RAG 결과 로그

## 권장 구현 순서

1. 현재 RAG 실패 사례 정리
2. 청킹 전략 재설계
3. 메타데이터 필드 강화
4. 임베딩 재생성 스크립트 정리
5. 질문-기대답변 평가셋 구축

## 중요한 포인트

### 1. RAG 품질은 프롬프트만으로 해결되지 않는다

보통 아래가 같이 맞아야 한다.

- 문서 품질
- 청킹 방식
- 메타데이터
- 검색 전략
- 프롬프트

### 2. 실패 사례를 축적해야 개선이 된다

예를 들면 아래 질문을 별도로 관리한다.

- 특정 배우 중심 추천 질문
- 좌우 시야 비교 질문
- 음향 우선 질문
- 가족 동반 / 키 작은 사람 같은 조건 질문

## 체크리스트

- [ ] 청킹 전략 재정리
- [ ] 메타데이터 확장
- [ ] 임베딩 재생성 흐름 정리
- [ ] 평가 질문셋 작성
- [ ] RAG 결과 로그 수집 포인트 추가

## 완료 기준

- 질문 유형별로 품질을 비교할 수 있다
- RAG 실패 원인을 설명할 수 있다
- 이후 에이전트 답변 품질도 같이 개선할 수 있다
