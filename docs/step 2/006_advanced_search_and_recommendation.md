# 006_advanced_search_and_recommendation

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- Nest 검색 도메인 코드는 `apps/nest-api/src/search` 같은 도메인 폴더에 둔다.
- React 검색/필터 UI는 `apps/web-react/src/features/search`에 둔다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 목적

이 문서는 2차 구현에서 검색과 필터를
"조건 조회"에서 "가중치 기반 추천과 랭킹"으로 확장하는 작업을 다룬다.

관련 1차 문서:

- [docs/step 1/007_search_filter_sort_pagination.md](../step%201/007_search_filter_sort_pagination.md)
- [docs/step 1/006_tags.md](../step%201/006_tags.md)

## 이 단계가 끝나면 되는 것

- 조건 일치뿐 아니라 우선순위 기반 정렬이 된다
- 추천 점수 계산 로직을 만들 수 있다
- 검색 결과 품질을 더 세밀하게 조정할 수 있다

## 이번 단계에서 구현하는 것

- 가중치 기반 정렬 점수
- 복합 필터 조합 최적화
- 검색어 전처리 개선
- 추천 정렬 옵션 추가

## 권장 구현 순서

1. 현재 검색 실패 사례 수집
2. 가중치 후보 정의
3. 정렬 점수 계산 로직 설계
4. API 응답에 추천 점수 또는 정렬 기준 반영
5. `apps/web-react/src/features/search`에서 추천순 노출

## 추천 점수 예시

- 시야 점수
- 음향 점수
- 태그 일치도
- 최근성
- 이미지 포함 여부
- 캐스트 일치도

## 중요한 포인트

### 1. 추천과 검색을 섞되 구분은 유지한다

사용자는 "정확히 찾기"와 "추천받기"를 다르게 기대한다.
따라서 2차에서는 아래를 분리하는 편이 좋다.

- 검색 결과
- 추천 결과
- 추천 이유

### 2. 점수는 설명 가능해야 한다

나중에 에이전트 답변까지 생각하면
"왜 이 후기가 위에 왔는지"를 설명할 수 있어야 한다.

## 체크리스트

- [ ] 복합 필터 성능 점검
- [ ] 추천순 정렬 옵션 추가
- [ ] 추천 이유 노출 필드 정의
- [ ] 실패 검색 사례 재검증

## 완료 기준

- 같은 조건에서도 더 그럴듯한 결과가 위로 올라온다
- 추천 결과의 이유를 사용자에게 설명할 수 있다
- 이후 RAG와 에이전트 품질 개선에 재사용할 수 있다
