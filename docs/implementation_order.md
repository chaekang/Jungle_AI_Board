# 뮤지컬 좌석 후기 게시판 구현 순서 정리

## 1. 전체 구현 순서

이 프로젝트는 AI 기능부터 바로 붙이기보다,
먼저 좌석 후기 게시판의 핵심 데이터와 사용자 흐름을 안정적으로 만든 뒤
그 위에 RAG, MCP, Agent를 확장하는 순서가 가장 안전하다.

추천 구현 순서는 다음과 같다.

1. 프로젝트 기본 세팅 / DB 설계
2. 사용자 인증
3. 극장 / 공연 / 좌석 메타데이터 구조 설계
4. 좌석 후기 CRUD
5. 댓글 기능
6. 태그 기능
7. 검색 / 필터 / 정렬 / 페이징
8. 기본 기능 통합 점검
9. RAG 기능
10. MCP 연동 기능
11. 좌석 선택 도우미 Agent
12. 전체 통합 테스트 / 시연 시나리오 정리

---

## 2. 프로젝트 기본 세팅 / DB 설계

가장 먼저 애플리케이션 기본 구조와 데이터 모델을 잡아야 한다.

이 단계에서 해야 할 일은 다음과 같다.

1. 프로젝트 생성
2. DB 연결
3. 환경변수 설정
4. 공통 폴더 구조 정리
5. 공통 예외 처리 / 응답 형식 초안 정리
6. RAG 확장을 고려한 데이터 구조 설계

초기부터 고려할 주요 테이블은 다음과 같다.

- `users`
- `theaters`
- `musicals`
- `performances`
- `seat_reviews`
- `comments`
- `tags`
- `seat_review_tags`
- 임베딩 또는 검색용 문서 테이블

좌석 후기는 단순 텍스트만 저장하는 것이 아니라,
층, 구역, 열, 번호 같은 구조화된 좌석 정보와 함께 저장되어야
나중에 검색, 필터, RAG 정확도가 좋아진다.
여기에 배역, 배우, 무대 좌우 체감 같은 관람 포인트를 담을 수 있어야
"어느 쪽이 더 잘 보이냐" 같은 질문에도 대응할 수 있다.

---

## 3. 사용자 인증

후기 작성자, 댓글 작성자, 수정 권한, 삭제 권한이 모두 사용자와 연결되므로
인증 기능은 초반에 먼저 구현하는 것이 좋다.

구현 순서는 다음과 같다.

1. 회원가입
2. 로그인
3. JWT 발급
4. 인증 Guard 구현
5. 현재 로그인 사용자 조회
6. 작성자 본인 여부 확인 로직 정리

이 단계가 완료되어야 후기와 댓글의 작성 및 수정 권한을 안정적으로 제어할 수 있다.

---

## 4. 극장 / 공연 / 좌석 메타데이터 구조 설계

일반 게시판과 달리 이 프로젝트는 좌석 위치 데이터가 핵심이므로,
후기 CRUD 전에 극장과 좌석 메타데이터 구조를 먼저 정리하는 편이 좋다.

우선 구현할 항목은 다음과 같다.

1. 극장 엔티티 설계
2. 공연 또는 작품 엔티티 설계
3. 공연 회차 또는 시즌 단위 데이터 필요 여부 결정
4. 좌석 위치 필드 규칙 정하기
   - 층
   - 구역
   - 열
   - 번호
5. 시야 / 음향 / 좌석 편의성 / 표정 / 무대 체감 평가 필드 정의
6. 배역 / 배우 / 무대 좌우 체감 메모 필드 정의
7. 시야 방해 여부와 메모 필드 정의

이 단계를 먼저 잡아두면 후기 작성 API와 검색 API를 훨씬 깔끔하게 설계할 수 있다.

---

## 5. 좌석 후기 CRUD

이제 서비스의 핵심인 좌석 후기 기능을 구현한다.

구현 순서는 다음과 같다.

1. 후기 작성
2. 후기 목록 조회
3. 후기 상세 조회
4. 후기 수정
5. 후기 삭제
6. 작성자만 수정 / 삭제 가능하도록 권한 체크

후기 작성 시 필수 입력값은 아래 정도로 제한하는 것이 현실적이다.

- 극장
- 공연명 또는 작품
- 층
- 구역
- 열
- 번호
- 후기 본문
- 시야 평가
- 음향 평가
- 좌석 편의성 평가
- 배우 표정 체감
- 무대 전체 체감

이 정도만 필수로 받아도 사용자는 충분히 후기를 남길 수 있다.

아래 항목은 선택 입력 또는 추후 AI 추론 대상으로 두는 편이 좋다.

- 특정 배역 / 배우 체감 메모
- 무대 왼쪽 / 오른쪽 / 중앙 체감 메모
- 시야 방해 여부
- 시야 방해 상세 메모
- 편의성 보충 메모

즉, 초반 후기 작성 UX는 `짧고 쉽게` 만들고,
세부 메타데이터는 후기 본문을 바탕으로 AI가 보조적으로 추론하는 구조가 적합하다.

이 기능이 안정적이어야 이후 댓글, 태그, 검색, RAG가 모두 자연스럽게 연결된다.

## 6. 댓글 기능

댓글은 후기 상세 페이지 경험을 완성하는 핵심 기능이다.
특정 좌석에 대한 추가 질문과 보충 설명이 이 영역에 쌓이기 때문이다.

구현 순서는 다음과 같다.

1. 댓글 작성
2. 후기별 댓글 조회
3. 댓글 수정
4. 댓글 삭제
5. 댓글 작성자 권한 체크

처음에는 대댓글 없이 일반 댓글만 구현하는 것이 범위 관리에 유리하다.

---

## 7. 태그 기능

태그는 후기 검색성과 분류 품질을 높여주므로
후기 CRUD와 댓글 기능이 잡힌 뒤 구현하는 것이 좋다.

구현 순서는 다음과 같다.

1. 태그 테이블 설계
2. 후기 작성 시 태그 연결
3. 후기 수정 시 태그 변경
4. 중복 태그 방지
5. 후기 상세 조회 시 태그 포함
6. 태그별 후기 목록 조회

초기에 유용한 태그 예시는 다음과 같다.

- 공연 태그
- 극장 태그
- 좌석 특성 태그
  - `시야좋음`
  - `시야안좋음`
  - `음향좋음`
  - `음향안좋음`
  - `배우잘보임`
  - `전체무대잘보임`
  - `난간시야방해`
  - `가성비`
- 관람 목적 태그
  - `표정위주`
  - `전체무대위주`
  - `첫관람추천`
  - `회전추천`

## 8. 검색 / 필터 / 정렬 / 페이징

좌석 후기 서비스에서는 검색과 필터 품질이 매우 중요하므로
기본 CRUD 이후 비교적 빠르게 붙이는 것이 좋다.

구현 순서는 다음과 같다.

1. 후기 목록 페이징
2. 극장명 검색
3. 공연명 검색
4. 좌석 위치 검색
   - 층
   - 구역
   - 열
   - 번호
5. 태그 검색
6. 시야 방해 여부 필터
7. 음향 평가 필터
8. 좌석 편의성 필터
9. 배역 / 배우 키워드 검색
10. 정렬 옵션 추가
   - 최신순
   - 인기순
   - 평점순

예시 API 형태는 다음과 같다.

```http
GET /seat-reviews?page=1&limit=10
GET /seat-reviews/search?theater=블루스퀘어&floor=2&row=3
GET /seat-reviews/search?tag=배우잘보임&hasObstruction=false
GET /seat-reviews/search?musical=어쩌면해피엔딩&character=올리버&side=left
GET /seat-reviews/search?theater=샤롯데&comfort=comfortable
```

이 단계가 끝나면 사용자가 원하는 좌석 경험을 직접 찾아보는 기본 UX가 완성된다.

## 9. 기본 기능 통합 점검

AI 기능을 붙이기 전에 기본 기능을 한 번 정리해야 한다.

확인해야 할 항목은 다음과 같다.

- 회원가입 / 로그인 정상 동작
- JWT 인증 정상 동작
- 후기 작성 / 조회 / 수정 / 삭제 정상 동작
- 후기 수정 / 삭제 권한 체크 정상 동작
- 댓글 작성 / 조회 / 수정 / 삭제 정상 동작
- 댓글 권한 체크 정상 동작
- 태그 연결 정상 동작
- 검색 / 필터 / 정렬 / 페이징 정상 동작
- 좌석 편의성, 배역/배우, 좌우 방향 관련 조건 검색 정상 동작
- API 응답 형식 일관성 확인
- 예외 처리 정리

여기까지 되면 좌석 후기 게시판의 1차 완성본이라고 볼 수 있다.

## 10. RAG 기능

AI 기능 중에서는 RAG를 먼저 구현하는 것이 가장 자연스럽다.
누적된 좌석 후기 데이터를 직접 활용하는 기능이기 때문이다.

구현 순서는 다음과 같다.

1. RAG 대상으로 사용할 데이터 범위 결정
2. 후기 본문 + 구조화된 좌석 메타데이터를 문서화
3. 후기 생성 / 수정 시 임베딩 생성
4. 벡터 저장 구조 구성
   - pgvector 또는 별도 벡터 저장소
5. 질문 입력 API 구현
6. 사용자 질문에서 작품명 / 극장명 / 층 / 구역 / 열 / 배역명 / 배우명 / 편의성 관련 조건을 추출
7. 작품명이 있으면 해당 작품 후기만 우선 필터링
8. 배역명이나 배우명이 있으면 좌우 방향과 동선 관련 후기를 우선 검색
9. 좌석 편안함 질문이면 편의성 평가와 피로도 관련 후기를 우선 검색
10. 선택 입력되지 않은 세부 정보는 후기 본문에서 보조 추론
11. 작품명이 없으면 좌석 위치 중심으로 관련 후기 검색
12. 검색 결과를 LLM에 전달
13. 요약 답변과 근거 후기 반환

예시 기능은 다음과 같다.

- "블루스퀘어 2층 3열 괜찮아?"
- "샤롯데 1층 중앙 블록은 표정 보기 좋아?"
- "음향 좋은 구역 추천해줘"
- "이 극장 2층 앞열은 오래 앉아 있기 불편해?"
- "어쩌면 해피엔딩에서 올리버 보려면 왼쪽이 좋아, 오른쪽이 좋아?"

중요한 점은 단순 본문 검색이 아니라,
좌석 위치와 후기 평가값까지 함께 활용할 수 있어야 한다는 것이다.
다만 모든 세부 항목을 작성 시점에 강제로 입력받기보다,
핵심 필드만 받고 나머지는 본문에서 추론하는 방식이 후기 작성률 측면에서 더 유리하다.
또한 작품명이 포함된 질문과 포함되지 않은 질문의 검색 범위를 다르게 처리해야
사용자가 원하는 맥락에 맞는 답변을 만들 수 있다.
배역이나 배우가 포함된 질문은 일반 좌석 평가보다
좌우 방향, 동선, 시야 각도 관련 후기를 더 강하게 반영해야 한다.
좌석 편안함 질문은 시야나 음향보다
착석감, 다리 공간, 장시간 관람 피로도를 더 강하게 반영해야 한다.

## 11. MCP 연동 기능

MCP는 외부 극장 정보와 좌석 배치도를 가져오는 확장 단계로 보는 것이 좋다.
기본 데이터 모델과 후기 작성 흐름이 안정된 뒤에 붙여야 혼선이 적다.

구현 순서는 다음과 같다.

1. 어떤 외부 데이터를 연결할지 범위 결정
2. 극장 기본 정보 조회 도구 설계
3. 좌석 배치도 조회 도구 설계
4. 외부 데이터와 내부 극장 데이터 매핑 규칙 정의
5. 동기화 API 또는 관리자용 갱신 로직 구현
6. 후기 작성 화면에서 좌석 입력 보조 기능 연결

예시 MCP 활용은 다음과 같다.

- 극장 선택 시 층별 구역 정보 자동 제안
- 좌석 배치도 불러오기
- 극장별 메타데이터 갱신

즉, MCP는 후기 데이터 입력 정확도와 사용자 편의성을 높이는 단계다.

## 12. 좌석 선택 도우미 Agent

Agent 기능은 가장 마지막에 구현하는 것이 좋다.
이 기능은 후기 데이터, 검색, RAG, MCP를 모두 조합해서 동작하기 때문이다.

기본 흐름은 다음과 같다.

1. 사용자 요청 분석
2. 예산, 선호도, 배우 중심 여부, 배역 중심 여부, 전체 무대 선호 여부, 좌석 편안함 우선 여부 등 조건 추출
3. 관련 후기 검색 또는 RAG 질의 수행
4. 필요 시 극장 메타데이터 또는 좌석 배치도 정보 활용
5. 좌석 구역과 좌우 방향 후보 정리
6. 추천 이유와 주의사항 생성
7. 최종 추천 답변 반환

예시 기능은 다음과 같다.

- "예산은 중간 정도고 배우 표정을 잘 보고 싶어"
- "첫 관람이라 전체 무대가 잘 보이는 좌석 추천해줘"
- "시야 방해 적고 음향 좋은 자리 위주로 골라줘"
- "특정 배역 동선을 보기 좋은 왼쪽/오른쪽 블록을 추천해줘"
- "조금 비싸도 오래 앉아 있기 편한 좌석으로 추천해줘"

처음부터 완벽한 추천 엔진을 만들기보다,
후기 기반 추천과 이유 설명을 우선 제공하는 방식이 현실적이다.

## 13. 최종 결론

핵심은 좌석 후기 데이터 구조를 먼저 탄탄하게 만드는 것이다.

이 프로젝트의 AI 기능은 모두 후기 데이터 품질에 의존한다.
따라서 가장 좋은 구현 흐름은 다음과 같다.

```text
기본 좌석 후기 게시판 완성
→ 검색 / 필터 중심의 사용성 확보
→ RAG로 좌석 질문 답변 기능 추가
→ MCP로 외부 극장 정보와 좌석 배치도 연동
→ Agent로 사용자 조건 기반 좌석 추천 구현
```

이 순서로 진행하면 기능 간 의존성이 자연스럽고,
중간에 데이터 구조를 크게 뒤엎을 가능성도 줄일 수 있다.
