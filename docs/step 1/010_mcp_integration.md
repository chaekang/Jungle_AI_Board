# 010_mcp_integration

## 목적

이 문서는 [implementation_order.md](../implementation_order.md)의
`11. MCP 연동 기능`을 구현하기 위한 가이드다.

이 단계는 외부 극장 정보와 좌석 배치도 같은 보조 데이터를
현재 서비스에 안전하게 붙이는 작업이다.

관련 개념 문서:

- [docs/concept/010_mcp_integration.md](../concept/010_mcp_integration.md)

## 이 단계가 끝나면 되는 것

- 어떤 외부 데이터를 연결할지 범위가 정해진다
- 내부 극장 데이터와 외부 데이터 매핑 규칙이 정리된다
- 후기 작성 화면에서 좌석 입력 보조를 붙일 수 있다

## 권장 구현 범위

### 초기에 붙이기 좋은 것

- 극장 기본 정보 조회
- 층 / 구역 정보 조회
- 좌석 배치도 조회 또는 링크 연결

### 이번 구현 범위에서 제외하는 것

- 스케줄러 기반 자동 동기화
- 복수 외부 소스 통합

이 항목들은 현재 구현 순서 안에서 바로 다루지 않는다.
대신 짧은 TTL 캐시와 수동 갱신 포인트는 이번 단계에서 구현한다.

## 권장 구현 순서

1. 외부 데이터 범위 결정
2. 내부 모델과 매핑 규칙 정의
3. 조회 도구 인터페이스 설계
4. 짧은 TTL 캐시 + 관리자나 개발자 기준 수동 갱신 흐름 설계
5. 후기 작성 화면 보조 연결

## 중요한 포인트

### 1. 외부 데이터는 진실의 원천이 아니라 보조 수단이다

내부 서비스가 이미 가진 `theaters` 기준이 흔들리지 않게 해야 한다.
즉 외부 데이터는 내부 극장 레코드에 매핑되는 보조 정보로 다루는 편이 안전하다.

### 2. 실패했을 때 서비스가 멈추면 안 된다

MCP 조회 실패 시에도:

- 후기 조회는 된다
- 후기 작성 기본 입력은 된다

가 되도록 설계하는 것이 좋다.

### 3. 짧은 TTL 캐시를 같이 구현한다

매 요청마다 외부 데이터를 직접 조회하면
응답 속도와 실패 지점이 모두 불안정해질 수 있다.

이번 단계에서는 아래 정도를 캐시 대상으로 잡는다.

- 극장 기본 정보 캐시
- 층 / 구역 메타데이터 캐시
- 좌석 배치도 링크 또는 조회 결과 캐시

캐시 만료 시간은 너무 길지 않게
예를 들어 5분에서 30분 사이로 짧게 두는 편이 현실적이다.

### 4. 수동 갱신 포인트를 둔다

짧은 캐시만 믿기보다,
관리자용 또는 개발용 수동 갱신 경로가 있으면 디버깅이 훨씬 쉽다.

## 체크리스트

- [ ] 외부 데이터 범위 확정
- [ ] 내부-외부 매핑 규칙 확정
- [ ] 좌석 배치도 조회 방식 확정
- [ ] 캐시 대상과 TTL 정책 정의
- [ ] 실패 시 fallback 동작 정의
- [ ] 후기 작성 보조 UI 연결 포인트 정의

## 완료 기준

- 좌석 입력 정확도가 올라간다
- 외부 연동이 없어도 서비스 핵심 흐름은 유지된다
- 같은 외부 데이터 반복 조회 시 불필요한 호출이 줄어든다
