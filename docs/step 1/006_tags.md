# 006_tags

## 목적

이 문서는 [implementation_order.md](../implementation_order.md)의
`7. 태그 기능`을 구현하기 위한 단계별 가이드다.

태그는 검색성과 분류 품질을 올리는 가장 저렴한 구조화 수단이다.
본문 전체를 파싱하지 않아도 특정 후기를 빠르게 묶을 수 있다.

관련 개념 문서:

- [docs/concept/006_tags.md](../concept/006_tags.md)

## 이 단계가 끝나면 되는 것

- 태그 테이블이 정리된다
- 후기 작성 / 수정 시 태그를 연결할 수 있다
- 후기 상세 조회 시 태그를 같이 볼 수 있다
- 태그별 후기 목록 조회가 가능하다

## 권장 데이터 구조

- `tags`
- `seat_review_tags`

초기에는 N:M 조인 테이블로 가는 편이 가장 단순하다.

## 태그 전략

### 초기에 바로 유용한 태그

- 공연 태그
- 극장 태그
- 좌석 특성 태그
- 관람 목적 태그

예시:

- `시야좋음`
- `시야안좋음`
- `음향좋음`
- `배우잘보임`
- `전체무대잘보임`
- `난간시야방해`
- `가성비`
- `표정위주`
- `첫관람추천`

### 태그 입력 방식

초기에는 자유 입력보다 선택형 또는 제한된 자유 입력이 안전하다.

이유:

- 오탈자 방지
- 중복 태그 방지
- 검색 일관성 확보

## 권장 API 범위

- 후기 작성 / 수정 API에서 태그 ID 배열 또는 태그 문자열 배열 수용
- `GET /seat-reviews/:id`
- `GET /tags/:tagId/seat-reviews` 또는 검색 API 통합

## 구현 포인트

### 1. 중복 태그를 막는다

- 같은 이름 태그 중복 생성 방지
- 같은 후기-태그 연결 중복 방지

### 2. 태그 소유권은 따로 두지 않는다

초기에는 태그 자체의 작성자 개념 없이
후기와 연결되는 공용 분류 체계로 보는 편이 단순하다.

### 3. 본문과 태그의 역할을 분리한다

- 본문: 맥락이 풍부한 서술
- 태그: 빠른 분류와 필터

- 태그가 검색과 필터의 재료로 바로 쓸 수 있다
