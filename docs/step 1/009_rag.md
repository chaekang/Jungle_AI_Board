# 009_rag

## 목적

이 문서는 [implementation_order.md](../implementation_order.md)의
`10. RAG 기능`을 실제 구축 순서 중심으로 정리한 문서다.

이 단계의 핵심은 "후기 데이터가 AI가 읽기 좋은 문서"로 바뀌는 것이다.

관련 개념 문서:

- [docs/concept/009_rag.md](../concept/009_rag.md)

## 이 단계가 끝나면 되는 것

- 후기 + 메타데이터 기반 검색이 가능하다
- 질문에서 핵심 조건을 추출할 수 있다
- 관련 후기들을 LLM에 전달해 답변을 만들 수 있다
- 근거 후기와 함께 요약 답변을 반환할 수 있다

## 권장 구현 순서

1. RAG 대상 데이터 범위 확정
2. 후기 문서화 포맷 정의
3. 임베딩 생성 시점 결정
4. 벡터 저장소 결정
5. 질문 입력 API 구현
6. 조건 추출 규칙 구현
7. 조건 기반 필터 + 검색 결합
8. 답변 / 근거 포맷 정리

## 문서화 포맷 예시

하나의 후기를 아래처럼 하나의 검색 문서로 본다.

- 극장명
- 작품명
- 좌석 위치
- 평가값
- 본문
- 태그
- 배역 / 배우 관련 메모

즉 벡터화 전에 "사람이 읽을 수 있는 요약 문자열"을 안정적으로 만드는 것이 중요하다.

## 질문 처리 규칙 초안

### 작품명이 있을 때

- 해당 작품 후기를 우선 필터링

### 배우 / 배역명이 있을 때

- 좌우 방향, 동선, 표정 관련 후기를 더 강하게 반영

### 편의성 질문일 때

- `comfort_rating`
- 장시간 관람 피로도
- 다리 공간 / 착석감 메모

### 작품명이 없을 때

- 좌석 위치와 극장 중심으로 검색 범위를 좁힌다

## API 초안

- `POST /rag/questions`

응답 예시 구성:

- `answer`
- `reasons`
- `sources`
