# 011_seat_selection_agent

## 현재 코드 기준 업데이트

좌석 선택 Agent는 공식 구역과 비공식 설명 블록을 구분해서 말해야 한다.

- 공식 구역이 있는 공연장
  - 예: 두산아트센터 1층 `A/B/C`, 2층 `D/E/F`
  - 추천 결과에서 공식 구역명을 사용할 수 있다.
- 공식 구역이 없는 공연장
  - 예: 예스24스테이지
  - 리뷰 저장값에는 `seatSection`을 요구하지 않는다.
  - 사용자 설명에서는 복도 기준으로 `왼쪽블록 / 중앙블록 / 오른쪽블록` 같은 표현을 사용할 수 있다.

Agent 응답에서는 아래처럼 구분한다.

```text
공식 구역: 없음
설명용 블록: 중앙
추천 이유: 복도 기준 중앙 블록에 가까워 시야 왜곡이 적을 가능성이 큼
```

작품/시즌도 구분해야 한다.

- 사용자가 `베어더뮤지컬`이라고만 말하면 여러 시즌 리뷰를 함께 참고할 수 있다.
- 사용자가 `25시즌 베어더뮤지컬`이라고 말하면 `performance.seasonLabel = "25시즌"` 기준으로 좁힌다.
- `24-25시즌` 같은 값이 가능하므로 시즌은 숫자가 아니라 문자열로 처리한다.

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- Agent API와 실행 로직은 `apps/fastapi-api/app/routers`, `apps/fastapi-api/app/services`, `apps/fastapi-api/app/schemas`에 나눠 둔다.
- React 에이전트 화면이 필요하면 `apps/web-react/src/features/agent`에 둔다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 이 단계가 끝나면 되는 것

- 사용자 요청에서 조건을 읽어낼 수 있다
- 관련 후기와 메타데이터를 조합할 수 있다
- 추천 이유와 주의사항을 함께 반환할 수 있다

## 권장 입력 조건

- 예산
- 표정 위주 / 전체 무대 위주
- 배우 중심 / 배역 중심
- 시야 방해 민감도
- 음향 우선 여부
- 편의성 우선 여부

## 권장 출력 형태

- 추천 구역
- 추천 방향
- 추천 이유
- 주의사항
- 근거 후기

## 권장 구현 순서

1. 사용자 요청 분석 규칙 정의
2. 조건 스키마 정의
3. 검색 / RAG / MCP 호출 순서 정의
4. 추천 후보 정리 규칙 정의
5. 응답 포맷 정리

## 조합 흐름 예시

1. 사용자 질문 수신
2. 작품명 / 극장명 / 선호 조건 추출
3. 먼저 검색 또는 필터로 범위를 좁힘
4. 필요한 경우 RAG로 근거 후기 보강
5. 필요한 경우 MCP로 극장 구역 정보 확인
6. 후보 좌석 방향 정리
7. 이유와 주의사항 생성

## 중요한 포인트

### 1. 처음부터 완벽한 추천 엔진을 만들려 하지 않는다

초기 버전은 "근거 있는 추천 문장"만 안정적으로 나와도 충분하다.

### 2. 확신이 낮을 때는 범위를 제안하는 편이 낫다

예:

- 특정 좌석 1개 추천
- 특정 블록 / 구역 추천

중에서는 구역 추천이 더 안전한 경우가 많다.

### 3. 근거가 보이는 응답이 중요하다

추천만 주고 끝내기보다,
어떤 후기와 어떤 조건 때문에 그렇게 말했는지를 같이 보여주는 편이 좋다.
