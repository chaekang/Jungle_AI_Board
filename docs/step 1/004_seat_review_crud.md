# 004_seat_review_crud

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- Nest 후기 도메인 코드는 `apps/nest-api/src/seat-reviews` 같은 도메인 폴더에 둔다.
- React 후기 작성/조회 화면, 훅, 컴포넌트는 `apps/web-react/src/features/reviews`에 둔다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 목적

- `POST /seat-reviews`
- `GET /seat-reviews`
- `GET /seat-reviews/:id`
- `PATCH /seat-reviews/:id`
- `DELETE /seat-reviews/:id`

## 권장 구현 순서

1. DTO 정의
2. `SeatReviewsService.create()`
3. 목록 조회
4. 상세 조회
5. 수정
6. 삭제
7. 작성자 권한 체크

## 중요한 구현 포인트

### 1. 인증과 권한은 분리해서 본다

- 인증: 로그인한 사용자인가
- 권한: 그 후기의 작성자인가

즉 Guard만 붙이면 끝나는 일이 아니다.
수정과 삭제에서는 반드시 `authorId === currentUser.id` 체크가 들어가야 한다.

### 2. BigInt 직렬화 규칙을 지킨다

후기 ID, 작성자 ID가 `BigInt`라면
응답으로 보낼 때는 문자열 변환을 일관되게 적용해야 한다.

### 3. 응답용 변환 함수를 따로 둔다

DB 객체를 그대로 반환하면:

- 내부 필드가 섞일 수 있고
- 직렬화 에러가 생길 수 있고
- 나중에 응답 포맷이 흔들린다

`toPublicReview()` 같은 변환 함수를 두는 편이 좋다.

## 권장 서비스 로직 초안

### 작성

- 입력 검증
- FK 대상 존재 확인
- 현재 사용자 ID 연결
- 후기 저장

### 수정

- 대상 후기 조회
- 없으면 `404`
- 작성자 불일치면 `403`
- 수정 가능 필드만 반영

### 삭제

- 이번 단계에서 후기 삭제는 `deletedAt` 기반 소프트 삭제로 구현한다
- 목록, 상세, 검색에서 삭제된 후기를 제외하는 규칙까지 같이 맞춘다

## 완료 기준

- 로그인한 사용자는 후기를 작성할 수 있다
- 비로그인 사용자는 작성할 수 없다
- 작성자 본인만 수정 / 삭제 가능하다
- 목록 / 상세 조회 결과가 이후 댓글 / 태그 / 검색 확장에 무리가 없는 구조다
- 삭제 방식이 이후 댓글 / 검색 규칙과 충돌하지 않는다
