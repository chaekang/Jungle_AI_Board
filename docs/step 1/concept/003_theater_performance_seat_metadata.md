# 003 극장, 공연, 좌석 메타데이터 개념

## 문서 목적

극장과 작품, 공연 시즌, 좌석 구역처럼 서로 다른 현실 개념을 데이터 모델로 나누는 기준을 이해한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- Nest 메타데이터 API는 `apps/nest-api/src/metadata` 또는 도메인별 모듈에 둔다.
- React 메타데이터 선택 UI는 `apps/web-react/src/features/reviews`에서 사용한다.
- 좌석배치도 설정은 `apps/web-react/src/features/reviews/theater-seat-map-configs.ts`가 기준이다.
- 메타데이터는 화면 편의를 위한 문자열과 DB 정합성을 위한 식별자를 구분한다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `data modeling`

#### 개념

데이터 모델링은 현실의 대상과 관계를 저장 가능한 구조로 바꾸는 설계 과정이다.

여기서 중요한 건 `data modeling`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 극장, 공연, 좌석 메타데이터 개념에서 극장, 작품, 공연, 좌석, 후기, 댓글, 태그를 어떤 테이블로 나눌지 결정한다.
그래서 문서를 읽을 때도 먼저 "한 사실이 한 곳에 저장되는지, 관계가 쿼리 가능한 형태인지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

극장, 공연, 좌석 메타데이터 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `data modeling`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 ERD, Normalization, Primary Key를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/metadata`이다.
그 다음으로 `apps/web-react/src/features/reviews`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/features/reviews/theater-seat-map-configs.ts`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 작품과 공연을 분리하면 같은 작품의 다른 시즌과 극장을 표현할 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `data modeling`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// data modeling를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `data modeling`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| ERD | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Normalization | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Primary Key | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

처음 보이는 화면 입력값 그대로 테이블을 만들면 나중에 검색과 추천이 막힌다.
또 하나의 실수는 `data modeling`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 한 사실이 한 곳에 저장되는지, 관계가 쿼리 가능한 형태인지 확인한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`data modeling`는 극장, 공연, 좌석 메타데이터 개념에서 극장, 작품, 공연, 좌석, 후기, 댓글, 태그를 어떤 테이블로 나눌지 결정한다.
예시는 작품과 공연을 분리하면 같은 작품의 다른 시즌과 극장을 표현할 수 있다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 처음 보이는 화면 입력값 그대로 테이블을 만들면 나중에 검색과 추천이 막힌다는 점이다.
ERD, Normalization, Primary Key와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `data modeling`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `1:1`

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Cardinality | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |

#### 흔한 실수

자주 함께 쓰는 정보를 무리하게 1:1로 분리하면 JOIN만 늘어난다.
또 하나의 실수는 `1:1`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.

### `1:N`

#### 개념

1:N 관계는 한 행이 다른 테이블의 여러 행과 연결될 수 있는 관계다.

그래서 문서를 읽을 때도 먼저 "삭제나 수정 시 자식 데이터가 어떻게 처리될지 미리 정한다"라는 질문을 붙여 두면 좋다.

특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Foreign Key, Parent, Child를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

### `normalization`

#### 개념

또 하나의 실수는 `Foreign Key`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 삭제 정책과 참조 무결성이 서비스 규칙과 충돌하지 않는지 확인한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`Foreign Key`는 극장, 공연, 좌석 메타데이터 개념에서 후기가 존재하는 사용자, 극장, 작품을 참조하게 만들어 데이터 정합성을 지킨다.
예시는 SeatReview.theaterId는 Theater.id를 참조한다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 외래키 없이 숫자만 저장하면 없는 극장에 대한 후기가 생길 수 있다는 점이다.
Referential Integrity, Cascade, Join와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Foreign Key`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `Index`

#### 개념

인덱스는 테이블에서 특정 조건의 행을 빠르게 찾기 위해 따로 만든 탐색 구조다.

여기서 중요한 건 `Index`를 단어 뜻으로만 외우지 않는 것이다.

그 사이에서 `Index`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Query Plan, WHERE, ORDER BY를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

### `Scalability`

#### 개념

예시는 createdAt 월별 파티션은 오래된 후기와 최신 후기를 다르게 관리할 수 있게 한다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 파티션은 무조건 성능을 올리는 마법이 아니며 쿼리 조건과 맞아야 효과가 있다는 점이다.
Sharding, Index, Query Pruning와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Partition`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `Sharding`

#### 개념

샤딩은 데이터를 여러 데이터베이스나 서버에 나누어 저장하는 수평 분할 방식이다.

여기서 중요한 건 `Sharding`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 극장, 공연, 좌석 메타데이터 개념에서 서비스가 아주 커져 단일 DB가 감당하기 어려울 때 검토할 수 있다.
그래서 문서를 읽을 때도 먼저 "단일 DB 최적화, 캐싱, 읽기 복제 이후에도 필요한지 판단한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

극장, 공연, 좌석 메타데이터 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Sharding`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Partition, Distributed System, Consistency를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/metadata`이다.
그 다음으로 `apps/web-react/src/features/reviews`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/features/reviews/theater-seat-map-configs.ts`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 사용자 id 범위나 극장 id 기준으로 후기를 다른 샤드에 나눌 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Sharding`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Sharding를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Sharding`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Partition | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Distributed System | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Consistency | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

초기 프로젝트에 샤딩을 먼저 도입하면 복잡도만 크게 늘 수 있다.
또 하나의 실수는 `Sharding`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 단일 DB 최적화, 캐싱, 읽기 복제 이후에도 필요한지 판단한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?
