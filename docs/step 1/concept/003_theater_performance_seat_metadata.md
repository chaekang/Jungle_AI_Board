# 003_theater_performance_seat_metadata_concepts

## 이 문서의 역할

이 문서는 `003 극장 / 공연 / 좌석 메타데이터 구조 설계` 단계에서 중요한 데이터 구조 개념을 자세히 설명한다.
이 단계는 단순히 테이블 몇 개를 만드는 일이 아니라, 이후 후기 작성, 검색, 필터, RAG, Agent 추천이 기대는 공통 언어를 만드는 작업이다.

좌석 후기 서비스에서 가장 큰 실수 중 하나는 후기를 "그냥 텍스트 게시글"처럼 다루는 것이다.
하지만 이 서비스의 핵심 가치는 좌석 위치, 극장 구조, 공연 맥락, 관람 포인트를 구조화해서 저장하는 데 있다.
즉 메타데이터 설계는 편의 기능이 아니라 서비스 정체성과 직결된다.

## 왜 이 단계가 중요한가

예를 들어 사용자가 이런 질문을 한다고 생각해보자.

- "블루스퀘어 2층 3열 괜찮아?"
- "샤롯데 1층 중앙에서 표정 잘 보여?"
- "올리버 보려면 왼쪽이 좋아?"

이 질문들에 답하려면 후기 본문만으로는 부족하다.
최소한 다음이 구조화되어 있어야 한다.

- 극장
- 작품
- 공연 회차 또는 시즌
- 층 / 구역 / 열 / 번호
- 시야 / 음향 / 편의성 / 표정 / 무대 체감

즉 메타데이터 설계가 곧 검색 가능성과 AI 활용 가능성을 결정한다.

## 핵심 키워드

### `data modeling`

정의:

데이터 모델링은 어떤 정보를 어떤 단위로 저장하고, 각 정보가 서로 어떤 관계를 가지는지 설계하는 작업이다.

언제 중요한가:

- 스키마를 처음 설계할 때
- 기능 추가 전에 데이터 구조를 점검할 때
- RAG나 통계처럼 2차 활용 기능을 붙일 때

어떻게 적용하는가:

이 프로젝트에서는 후기 데이터를 최소 두 층으로 봐야 한다.

- 서술형 본문
- 구조화된 메타데이터

예를 들어 "2층 오른쪽 블록은 배우 동선 보기 좋음"은 본문으로도 쓸 수 있지만,
가능하면 `seat_floor=2`, `side_preference=right`, `expression_rating`, `character_note` 같은 구조도 함께 두는 편이 좋다.

장점:

- 검색성과 분석성이 좋아진다
- 데이터 품질이 올라간다
- AI가 더 안정적으로 활용할 수 있다

문제 / 주의점:

- 너무 많은 필드를 강제하면 후기 작성률이 떨어진다
- 너무 적게 구조화하면 나중에 검색과 RAG 정밀도가 떨어진다

면접 포인트:

"데이터 모델링은 저장 설계이면서 동시에 검색 설계라고 생각합니다. 특히 이 프로젝트처럼 좌석 후기의 2차 활용이 중요한 경우, 본문과 메타데이터의 균형이 핵심입니다."

### `ERD (1:1, 1:N, N:M)`

정의:

ERD는 엔티티 간 관계를 시각적으로 정리한 문서다.
관계의 기본 유형은 `1:1`, `1:N`, `N:M`으로 설명하는 경우가 많다.

이 프로젝트의 대표 관계:

- 극장 1개 : 공연 여러 개 -> `1:N`
- 작품 1개 : 공연 여러 개 -> `1:N`
- 후기 1개 : 댓글 여러 개 -> `1:N`
- 후기 여러 개 : 태그 여러 개 -> `N:M`

언제 중요해지는가:

- FK 설계할 때
- Prisma relation 작성할 때
- API 응답 구조를 정할 때

장점:

- 팀 안에서 데이터 구조를 빠르게 공유할 수 있다
- 구현 전 오류를 미리 찾기 좋다

문제 / 주의점:

- ERD만 예쁘고 실제 도메인 규칙이 없으면 소용없다
- 관계만 그리고 필드 수준 규칙을 빼먹는 경우가 많다

면접 포인트:

"저는 ERD를 단순 다이어그램보다 협업 도구로 봅니다. 특히 후기, 태그, 댓글, 공연처럼 관계가 많은 서비스에서는 API 설계 전 ERD가 큰 도움이 됩니다."

### `normalization`

정의:

정규화는 데이터를 중복 없이 더 일관되게 관리하기 위한 설계 원칙이다.

언제 쓰는가:

- 같은 정보가 여러 곳에 반복될 때
- 업데이트 불일치가 생길 가능성이 높을 때
- 관계형 구조를 안정적으로 만들고 싶을 때

이 프로젝트에서의 예:

- 극장 이름을 후기마다 문자열로 저장만 하지 않고 `theaters` 테이블로 분리
- 작품명을 후기 본문에만 두지 않고 `musicals` 테이블로 분리

장점:

- 데이터 정합성이 높아진다
- 수정 지점이 줄어든다
- 검색과 통계가 정확해진다

문제 / 주의점:

- 조회 시 조인이 많아질 수 있다
- 지나치게 세분화하면 구현 속도가 느려질 수 있다

면접 포인트:

"정규화는 만능이 아니라 trade-off라고 생각합니다. 이 프로젝트에서는 극장, 작품, 공연처럼 반복 사용되는 정보는 분리하고, 후기 맥락에 가까운 값은 일부 비정규화할 수도 있습니다."

### `Primary Key, Foreign Key, Index`

정의:

- `Primary Key`: 각 레코드를 고유하게 구분하는 키
- `Foreign Key`: 다른 테이블과 관계를 맺는 키
- `Index`: 특정 컬럼 기반 조회를 빠르게 만드는 보조 구조

언제 중요한가:

항상 중요하지만, 특히 검색 / 필터 / 정렬 요구가 있을 때 중요하다.

어떻게 적용하는가:

- `theaters.id`, `musicals.id`, `performances.id`, `seat_reviews.id`는 PK
- `seat_reviews.theater_id`, `seat_reviews.musical_id`, `seat_reviews.performance_id`는 FK
- `seat_floor`, `seat_section`, `created_at`, `theater_id`, `musical_id`는 인덱스 후보

장점:

- PK / FK는 구조를 안전하게 만든다
- Index는 검색 성능을 높인다

문제 / 주의점:

- 인덱스를 너무 많이 걸면 쓰기 성능이 저하될 수 있다
- FK 제약을 무시하면 단기 속도는 나도 장기 품질이 망가진다

면접 포인트:

"검색 서비스에서는 인덱스 설계가 기능 설계의 일부라고 생각합니다. 어떤 컬럼을 자주 필터링하고 정렬하는지를 먼저 보고 인덱스를 잡는 편입니다."

### `Join (inner, outer)`

정의:

조인은 여러 테이블의 데이터를 관계를 기반으로 함께 읽는 방식이다.

언제 쓰는가:

- 후기 상세에 작성자, 극장, 작품 정보를 함께 보여줄 때
- 특정 극장의 특정 작품 후기만 가져올 때
- 태그와 후기를 함께 조회할 때

`inner join`과 `outer join` 차이:

- `inner join`: 양쪽 모두 값이 있어야 결과 포함
- `outer join`: 기준 테이블 값을 유지하며 연결이 없더라도 결과 포함 가능

왜 중요한가:

메타데이터 구조를 잘 나눠 놓으면 결국 조회 시 조인을 통해 다시 묶어야 한다.
즉 정규화와 조인은 세트로 이해해야 한다.

면접 포인트:

"정규화가 쓰기 정합성을 만든다면, 조인은 읽기 복잡도를 가져옵니다. 그래서 둘을 항상 같이 봐야 한다고 생각합니다."

### `api design`

정의:

API 설계는 클라이언트가 어떤 입력을 보내고 어떤 응답을 받을지 계약을 정하는 일이다.

언제 중요한가:

메타데이터 설계는 DB에서 끝나지 않는다.
후기 작성 API, 후기 조회 API, 검색 API 구조까지 직접 영향을 준다.

예를 들어:

- 후기 작성 시 `theaterId`를 받을지, `theaterName`을 받을지
- `seatFloor`, `seatSection`, `seatRow`, `seatNumber`를 어떻게 나눌지

를 미리 정해야 프론트와 백엔드가 같은 말을 쓰게 된다.

면접 포인트:

"데이터 모델링과 API 설계는 분리된 작업이 아니라고 생각합니다. DB 컬럼 이름이 곧 DTO와 응답 모델에 반영되는 경우가 많기 때문입니다."

### `Scalability`, `Partition`, `Sharding`

정의:

- `Scalability`: 데이터와 트래픽이 늘어도 시스템이 감당할 수 있는 능력
- `Partition`: 한 테이블을 논리적으로 나누는 전략
- `Sharding`: 여러 DB로 물리적으로 분산하는 전략

언제 고려하는가:

초기 MVP에서는 당장 필요 없을 수 있다.
하지만 후기 수, 검색 수, 로그 수가 커질수록 검토하게 된다.

이 프로젝트에서 현실적인 관점:

지금 당장 샤딩을 설계할 필요는 없다.
대신 나중에 샤딩이 필요해져도 너무 힘들지 않은 기본 키 구조와 조회 패턴을 가지는 것이 중요하다.

면접 포인트:

"확장성은 초반부터 과하게 설계하기보다, 나중에 확장 가능한 키 구조와 조회 패턴을 유지하는 수준이 더 현실적이라고 봅니다."

## 면접처럼 정리하는 한 문단 요약

이 단계의 핵심은 좌석 후기를 일반 게시글이 아니라 검색과 AI 활용이 가능한 구조화된 데이터로 보는 것이다.
그래서 데이터 모델링, ERD, 정규화, PK/FK, 인덱스, 조인을 함께 생각해야 하고, DB 스키마와 API 설계를 따로 떼어 보면 안 된다.
실무적으로는 "어떤 값을 입력받을까"가 아니라 "나중에 어떤 질문에 답할 수 있어야 할까"라는 관점에서 설계하는 것이 중요하다.
