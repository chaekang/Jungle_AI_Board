# 009 RAG와 비동기 처리 개념

## 문서 목적

AI 추천이나 RAG를 붙일 때 즉시 응답 API와 오래 걸리는 백그라운드 작업을 어떻게 나눌지 이해한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- RAG 관련 API는 추천 도메인 또는 별도 AI 도메인으로 분리한다.
- 임베딩 생성과 인덱싱은 비동기 작업으로 분리하는 편이 안전하다.
- 캐시와 큐는 실패와 재시도 전략까지 함께 설계한다.
- 테스트는 모델 품질뿐 아니라 데이터 흐름과 장애 대응도 확인한다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `asynchronous processing`

#### 개념

비동기 처리는 오래 걸리는 작업을 요청 흐름과 분리해 나중에 완료하도록 만드는 방식이다.

여기서 중요한 건 `asynchronous processing`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 RAG와 비동기 처리 개념에서 RAG 인덱싱, AI 추천 생성, 대량 데이터 처리에 필요하다.
그래서 문서를 읽을 때도 먼저 "작업 생성, 진행 상태, 완료, 실패, 재시도 흐름이 있는지 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

RAG와 비동기 처리 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `asynchronous processing`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Job Queue, Worker, Event를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `RAG 관련 API는 추천 도메인 또는 별도 AI 도메인으로 분리한다.`이다.
그 다음으로 `임베딩 생성과 인덱싱은 비동기 작업으로 분리하는 편이 안전하다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `캐시와 큐는 실패와 재시도 전략까지 함께 설계한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 사용자가 요청하면 작업 id를 받고, 서버는 큐에서 추천 생성을 진행할 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `asynchronous processing`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// asynchronous processing를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `asynchronous processing`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Job Queue | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Worker | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Event | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

비동기 작업의 상태 조회와 실패 처리를 설계하지 않으면 사용자는 결과를 기다릴 방법이 없다.
또 하나의 실수는 `asynchronous processing`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 작업 생성, 진행 상태, 완료, 실패, 재시도 흐름이 있는지 본다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`asynchronous processing`는 RAG와 비동기 처리 개념에서 RAG 인덱싱, AI 추천 생성, 대량 데이터 처리에 필요하다.
예시는 사용자가 요청하면 작업 id를 받고, 서버는 큐에서 추천 생성을 진행할 수 있다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 비동기 작업의 상태 조회와 실패 처리를 설계하지 않으면 사용자는 결과를 기다릴 방법이 없다는 점이다.
Job Queue, Worker, Event와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `asynchronous processing`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `RabbitMQ`

주의점은 캐시 무효화 전략 없이 저장하면 오래된 데이터가 계속 보인다는 점이다.
Redis, TTL, Invalidation와 비교하면 책임 범위가 더 선명해진다.

### `Redis`

#### 개념

Redis는 메모리 기반 key-value 저장소로 캐시, 세션, 큐 보조 저장소로 자주 쓰인다.

여기서 중요한 건 `Redis`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 RAG와 비동기 처리 개념에서 rate limit 카운터, 세션 저장, 자주 쓰는 검색 결과 캐시에 활용할 수 있다.
그래서 문서를 읽을 때도 먼저 "데이터가 사라져도 복구 가능한지, 만료 정책이 있는지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

RAG와 비동기 처리 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Redis`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Cache, TTL, Pub/Sub를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `RAG 관련 API는 추천 도메인 또는 별도 AI 도메인으로 분리한다.`이다.
그 다음으로 `임베딩 생성과 인덱싱은 비동기 작업으로 분리하는 편이 안전하다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `캐시와 큐는 실패와 재시도 전략까지 함께 설계한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 사용자별 로그인 실패 횟수를 Redis에 TTL과 함께 저장할 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Redis`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Redis를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Redis`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Cache | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| TTL | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Pub/Sub | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

Redis가 빠르다고 영구 보존이 필요한 핵심 데이터를 무조건 맡기면 위험하다.
또 하나의 실수는 `Redis`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 데이터가 사라져도 복구 가능한지, 만료 정책이 있는지 확인한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// api design를 구현할 때 머릿속에 두는 최소 흐름
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `api design`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |

#### 흔한 실수

프론트 화면 구조 그대로 API를 만들면 다른 화면이나 기능에서 재사용하기 어렵다.
또 하나의 실수는 `api design`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 요청과 응답이 화면 하나가 아니라 도메인 자원 중심으로 설계됐는지 본다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`api design`는 RAG와 비동기 처리 개념에서 후기 목록 검색 조건, 좌석 필터, 태그 선택, 댓글 작성 규칙을 엔드포인트로 표현한다.
예시는 검색어와 극장 필터는 query parameter로 표현하면 목록 조회 의미가 자연스럽다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 프론트 화면 구조 그대로 API를 만들면 다른 화면이나 기능에서 재사용하기 어렵다는 점이다.
REST, DTO, Pagination와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `api design`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `testing`

#### 개념

테스트는 코드가 기대한 동작을 계속 만족하는지 자동 또는 수동으로 확인하는 활동이다.

여기서 중요한 건 `testing`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 RAG와 비동기 처리 개념에서 인증, 후기 CRUD, 검색 필터, 좌석 선택 흐름의 회귀를 막는다.
그래서 문서를 읽을 때도 먼저 "변경한 기능의 성공 경로와 실패 경로를 모두 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

RAG와 비동기 처리 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `testing`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Unit, Integration, E2E를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `RAG 관련 API는 추천 도메인 또는 별도 AI 도메인으로 분리한다.`이다.
그 다음으로 `임베딩 생성과 인덱싱은 비동기 작업으로 분리하는 편이 안전하다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `캐시와 큐는 실패와 재시도 전략까지 함께 설계한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 후기 작성 버튼을 여러 번 눌러도 중복 저장되지 않는지 테스트할 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `testing`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.
