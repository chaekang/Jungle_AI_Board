# 009_rag_concepts

## 이 문서의 역할

이 문서는 `009 RAG` 단계에서 알아야 하는 검색 보강, 비동기 파이프라인, 데이터 재구성 개념을 정리한다.
RAG는 단순히 LLM에 질문을 던지는 기능이 아니라,
"어떤 데이터를 어떤 방식으로 찾아서 어떤 맥락으로 LLM에 넘길 것인가"를 설계하는 작업이다.

이 프로젝트에서 RAG는 좌석 후기 원문과 구조화 메타데이터를 활용해
사용자의 좌석 질문에 더 맥락 있는 답변을 만드는 역할을 한다.
즉 핵심은 모델 자체보다 "데이터를 어떻게 준비하고 전달하느냐"에 있다.

## RAG를 왜 별도 단계로 두나

많은 프로젝트가 AI 기능을 너무 빨리 붙이려다가 흔들린다.
하지만 이 프로젝트에서는 먼저 후기, 메타데이터, 검색을 안정화한 뒤 RAG를 붙인다.
그 이유는 RAG의 품질이 결국 원천 데이터 품질에 의존하기 때문이다.

후기 데이터가 약하면:

- 검색 결과가 부정확하고
- 질문 의도와 관련 없는 문서가 섞이고
- 답변이 그럴듯하지만 근거가 약해진다

즉 RAG는 마법이 아니라, 좋은 검색과 좋은 데이터 구조 위에 올라가는 보강 계층이다.

## 핵심 키워드

### `asynchronous processing`

정의:

비동기 처리는 시간이 걸리는 작업을 요청-응답 흐름과 분리하거나, 기다리는 동안 다른 작업을 계속 처리하는 방식이다.

RAG에서 왜 중요한가:

후기 저장과 임베딩 생성을 한 요청 안에서 모두 처리하면 느려질 수 있다.
예를 들어 사용자가 후기를 저장할 때:

1. 후기 DB 저장
2. 검색용 문서 생성
3. 임베딩 생성
4. 벡터 저장

을 모두 동기적으로 하면 응답이 무거워진다.

그래서 보통은:

- 핵심 저장은 먼저 끝내고
- 임베딩 생성은 비동기로 넘기는 전략

을 많이 쓴다.

장점:

- 사용자 체감 응답 속도가 좋아진다
- 긴 작업을 따로 관리할 수 있다

문제 / 주의점:

- 최종 일관성이 즉시 맞지 않을 수 있다
- 후기가 저장됐지만 검색 인덱스 반영은 잠시 늦을 수 있다

면접 포인트:

"RAG 파이프라인에서는 사용자 요청 응답 속도와 색인 최신성 사이 균형이 중요합니다. 그래서 저는 저장과 임베딩 생성을 분리하는 비동기 구조를 선호합니다."

### `Job Queue`

정의:

잡 큐는 지금 바로 끝낼 필요가 없는 작업을 큐에 넣고, 워커가 나중에 처리하게 만드는 구조다.

RAG에서 언제 쓰는가:

- 임베딩 생성
- 문서 재색인
- 대량 문서 백필
- 실패한 임베딩 재시도

장점:

- 작업을 안정적으로 분리할 수 있다
- 재시도 전략을 둘 수 있다
- 사용자 요청 경로를 가볍게 유지할 수 있다

문제 / 주의점:

- 큐 지연으로 최신 데이터 반영이 늦을 수 있다
- 작업 중복 방지와 재시도 정책이 필요하다

면접 포인트:

"RAG에서는 모델 호출보다 파이프라인 안정성이 더 중요할 때가 많습니다. 그래서 임베딩 생성처럼 무거운 작업은 잡 큐로 분리하는 접근이 실무적으로 유리합니다."

### `RabbitMQ`, `Kafka`

정의:

둘 다 비동기 메시징과 이벤트 전달에 자주 쓰이는 시스템이다.

- `RabbitMQ`: 작업 큐, 라우팅, 재시도에 익숙한 메시지 브로커
- `Kafka`: 대용량 이벤트 스트림과 로그성 데이터 흐름에 강한 플랫폼

언제 각각 적합한가:

- `RabbitMQ`: "이 작업을 처리해라" 같은 명령성 작업
- `Kafka`: "이 이벤트가 발생했다"를 여러 소비자가 받아 처리하는 구조

RAG 관점에서의 차이:

후기 저장 후 임베딩 생성 같은 작업은 RabbitMQ식 사고가 잘 맞을 수 있고,
후기 생성 이벤트를 여러 파이프라인이 구독하는 구조는 Kafka식 사고가 잘 맞을 수 있다.

면접 포인트:

"작업 큐 중심이면 RabbitMQ, 이벤트 스트림 중심이면 Kafka가 더 자연스러울 수 있습니다. 중요한 건 도구보다 데이터 흐름 성격이라고 생각합니다."

### `caching`, `Redis`

정의:

캐시는 반복되는 결과를 빠르게 재사용하는 전략이고, Redis는 그 구현 수단으로 많이 쓰인다.

RAG에서 언제 유용한가:

- 자주 들어오는 질문
- 문서 메타데이터 조회
- 중간 검색 결과 재사용

장점:

- 응답 속도 향상
- 모델 / 검색 비용 절감

문제 / 주의점:

- 질문이 조금만 달라져도 캐시 효율이 떨어질 수 있다
- 문서가 갱신되면 캐시 무효화가 필요하다

면접 포인트:

"RAG 캐시는 모든 질문에 적용하기보다 반복률이 높은 질문 패턴이나 메타데이터 조회부터 적용하는 편이 현실적입니다."

### `data layer`

정의:

RAG도 결국 데이터 레이어 위에 선다.
원문 후기, 구조화 메타데이터, 검색용 문서, 임베딩 저장소가 모두 데이터 관리 대상이다.

왜 중요한가:

많은 경우 RAG 문제는 LLM이 아니라 데이터 레이어 문제다.

- 어떤 필드를 문서화했는가
- 어떤 메타데이터를 검색 필터로 썼는가
- 어떤 문서를 최신 상태로 유지했는가

가 성능을 크게 좌우한다.

면접 포인트:

"RAG 품질은 프롬프트보다 데이터 레이어 설계에 더 크게 좌우된다고 생각합니다. 특히 원문과 메타데이터를 어떻게 결합해 문서화할지가 중요합니다."

### `api design`

정의:

RAG API는 질문 입력과 답변 반환 구조를 설계하는 일이다.

좋은 RAG 응답이 갖추면 좋은 요소:

- `answer`
- `sources`
- `reasons`
- 필요 시 필터에 사용된 조건

왜 중요한가:

답변 텍스트만 반환하면 디버깅도 어렵고 사용자 신뢰도도 낮아진다.
근거 후기와 함께 보여주는 구조가 훨씬 좋다.

면접 포인트:

"RAG 응답은 텍스트 한 줄보다 근거와 함께 나와야 운영과 사용자 신뢰 측면에서 유리하다고 생각합니다."

### `testing`

정의:

RAG 테스트는 단순 문자열 일치보다 검색과 근거 구조를 검증하는 테스트다.

언제 중요한가:

- 검색 조건 추출이 맞는지
- 작품명이 있을 때 해당 작품 후기를 우선 찾는지
- 배우 관련 질문에서 좌우 / 동선 관련 근거가 섞이는지

를 확인해야 한다.

문제 / 주의점:

- 생성형 응답은 항상 똑같지 않을 수 있다
- 그래서 테스트 포인트를 "정답 문장"보다 "구조와 근거"에 맞추는 편이 낫다

면접 포인트:

"RAG 테스트는 exact match보다 retrieval quality와 source attribution을 더 중요하게 봅니다."

## 면접처럼 정리하는 한 문단 요약

RAG는 LLM 기능이 아니라 검색과 데이터 재구성을 포함한 파이프라인 기능이다.
그래서 비동기 처리, 잡 큐, 메시징, 캐시, 문서화 방식, 응답 설계를 함께 봐야 하고,
실무적으로는 "모델이 똑똑하냐"보다 "관련 문서를 얼마나 정확하게, 빠르게, 안정적으로 가져오느냐"가 더 중요하다.
