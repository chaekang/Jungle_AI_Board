# 007_search_filter_sort_pagination_concepts

## 이 문서의 역할

이 문서는 `007 검색 / 필터 / 정렬 / 페이징` 단계에서 필요한 API, 성능, UX 개념을 깊게 설명한다.
이 단계는 사용자가 AI 없이도 원하는 좌석 정보를 직접 찾을 수 있게 만드는 핵심 단계다.

좋은 검색은 단순히 문자열 찾기가 아니다.
검색, 필터, 정렬, 페이지네이션, 인덱스, 프론트 상태 관리가 함께 맞아야 체감 품질이 나온다.
즉 이 단계는 "DB 조회 기능 추가"가 아니라 "정보 탐색 경험 설계"에 가깝다.

## 검색 기능을 왜 별도 단계로 보나

후기 CRUD가 된다고 해서 사용자가 좋은 데이터를 쉽게 찾는 것은 아니다.
후기 수가 늘어날수록 "찾을 수 있는가"가 "쓸 수 있는가"만큼 중요해진다.

좌석 후기 서비스에서는 특히 다음 질문이 잦다.

- 특정 극장 후기만 보고 싶다
- 2층 3열만 보고 싶다
- 시야 방해가 없는 후기만 보고 싶다
- 표정 보기 좋은 자리 위주로 보고 싶다

즉 검색 품질은 서비스 핵심 가치와 직접 연결된다.

## 핵심 키워드

### `api design`

정의:

검색 API 설계는 조건을 어떤 파라미터로 받을지, 응답 메타데이터를 어떻게 줄지 결정하는 작업이다.

언제 중요한가:

- 필터 조합이 많아질 때
- 프론트에서 URL 기반 상태 복원을 하고 싶을 때
- 검색 조건을 공유 링크로 만들고 싶을 때

어떻게 적용하는가:

예를 들어:

- `?theater=블루스퀘어`
- `?floor=2&row=3`
- `?tag=시야좋음`
- `?sort=latest&page=2&limit=10`

처럼 읽히는 쿼리 구조가 좋다.

장점:

- 프론트와 백엔드가 이해하기 쉽다
- 테스트가 쉽다
- URL만 봐도 조건이 드러난다

문제 / 주의점:

- 검색과 필터의 경계가 흐려질 수 있다
- 파라미터 이름이 제각각이면 유지보수가 어렵다

면접 포인트:

"검색 API는 기능을 많이 넣는 것보다 규칙이 예측 가능해야 한다고 생각합니다. 특히 검색 조건이 URL에 자연스럽게 드러나는 구조를 선호합니다."

### `Rest`, `GraphQL`, `GRPC`

정의:

세 가지는 서로 다른 API 접근 방식이다.

- `Rest`: 리소스 중심
- `GraphQL`: 필요한 필드를 클라이언트가 지정
- `GRPC`: 바이너리 기반 고성능 RPC

현재 프로젝트에서 Rest가 맞는 이유:

- 검색 조건이 쿼리 파라미터와 잘 맞는다
- 브라우저 / 프론트에서 다루기 쉽다
- MVP 단계에서 가장 단순하다

GraphQL이 더 맞는 경우:

- 화면마다 필요한 필드가 크게 다를 때
- 중첩 조회 최적화가 중요할 때

GRPC가 더 맞는 경우:

- 내부 서비스 간 고성능 통신
- 외부 브라우저 클라이언트보다는 서버 간 통신

면접 포인트:

"현재 같은 검색 중심 웹 앱에서는 REST가 가장 현실적이라고 봅니다. 다만 클라이언트마다 필요한 필드 구성이 크게 갈리면 GraphQL을 검토할 수 있습니다."

### `Index`

정의:

인덱스는 검색과 정렬을 빠르게 만드는 DB 자료구조다.

언제 중요한가:

검색 기능이 늘어날수록 사실상 필수다.

후기 검색에서 인덱스 후보:

- `theater_id`
- `musical_id`
- `seat_floor`
- `seat_section`
- `has_obstruction`
- `created_at`

장점:

- 조회 속도가 빨라진다
- 정렬 성능이 개선된다

문제 / 주의점:

- 쓰기 비용이 늘어난다
- 실제 검색 패턴과 안 맞는 인덱스는 거의 소용이 없다

면접 포인트:

"인덱스는 많이 두는 게 아니라, 실제 where / order by 패턴을 기준으로 잡아야 의미가 있습니다."

### `caching`, `Redis`

정의:

- `caching`: 반복 조회 결과를 재사용하는 전략
- `Redis`: 캐싱에 자주 쓰이는 빠른 메모리 저장소

검색에서 언제 유용한가:

- 인기 검색 결과
- 자주 쓰는 필터 조합
- 극장별 후기 첫 페이지

장점:

- 응답 속도 향상
- DB 부하 감소

문제 / 주의점:

- 검색 조건 조합이 너무 많으면 캐시 효율이 떨어질 수 있다
- 캐시 무효화 전략이 필요하다

면접 포인트:

"검색 캐시는 무조건 모든 쿼리에 적용하기보다, 반복률이 높은 인기 조건부터 적용하는 편이 효율적입니다."

### `CSR`, `SSR`, `SSG`, `ISR`, `RSC`

정의:

- `CSR`: 클라이언트 렌더링
- `SSR`: 서버 렌더링
- `SSG`: 정적 생성
- `ISR`: 정적 재생성
- `RSC`: React Server Components

검색 UI에서 왜 중요할 수 있는가:

검색 페이지는 SEO, 초기 로딩, 필터 반응성에 따라 렌더링 전략이 달라질 수 있다.

현재 프로젝트에서는:

초기에는 CSR만으로 충분할 가능성이 크다.
검색 페이지의 인터랙션이 많고, 내부 서비스 성격이 강하기 때문이다.

다만 공개형 서비스로 커지고 SEO가 중요해지면 SSR 계열 전략도 검토할 수 있다.

면접 포인트:

"검색 페이지는 실시간 상호작용이 많아서 CSR이 잘 맞지만, 공개 유입과 SEO가 중요해지는 순간 SSR 전략도 검토할 가치가 있습니다."

### `Redux Toolkit`, `Zustand`, `Pinia`

정의:

상태 관리 도구들이다.

- `Redux Toolkit`, `Zustand`: React 생태계
- `Pinia`: Vue 생태계

언제 쓰는가:

- 검색 조건이 여러 화면에 걸쳐 공유될 때
- URL 상태와 내부 상태를 같이 관리할 때
- 전역 필터 상태가 복잡해질 때

현재 프로젝트에서는:

초기에는 로컬 state + URL 쿼리만으로도 충분할 수 있다.
전역 상태 관리 도구는 정말 복잡도가 올라갈 때 도입하는 편이 좋다.

면접 포인트:

"상태 관리 도구는 미리 넣기보다 상태 공유 범위가 넓어질 때 도입하는 편을 선호합니다. 검색 UI는 먼저 URL 상태와 로컬 상태만으로 충분한지 보는 편입니다."

### `UI`, `a11y`

정의:

- `UI`: 사용자가 필터와 결과를 다루는 시각적 인터페이스
- `a11y`: 접근성, 즉 다양한 사용자가 불편 없이 이용할 수 있게 만드는 품질

검색 UI에서 왜 중요한가:

검색 기능은 많다고 좋은 게 아니라, 사용자가 이해하고 조합할 수 있어야 좋다.

예:

- 필터 그룹이 너무 많으면 사용자가 길을 잃는다
- 라벨과 포커스 흐름이 약하면 접근성이 떨어진다

면접 포인트:

"검색 UX에서는 서버 성능 못지않게 필터를 사용자가 이해할 수 있는 구조로 배치하는 것이 중요하다고 생각합니다. 접근성도 그 안에 포함됩니다."

## 면접처럼 정리하는 한 문단 요약

검색 단계의 핵심은 데이터를 저장해두는 것이 아니라 사용자가 탐색할 수 있게 만드는 것이다.
그래서 API 설계, 인덱스, 캐시, 렌더링 전략, 상태 관리, 접근성을 함께 봐야 한다.
실무적으로는 "필터가 많다"보다 "사용자가 원하는 결과를 빠르고 예측 가능하게 찾을 수 있다"가 더 중요하다.
