# 007 검색, 필터, 정렬, 페이지네이션 개념

## 문서 목적

게시판 검색과 좌석 필터가 화면 기능처럼 보여도 API 설계, 인덱스, 캐시, 렌더링 방식, 상태 관리와 연결된다는 점을 정리한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- React 게시판 화면은 `apps/web-react/src/features/reviews/ReviewBoardPage.tsx`에 둔다.
- 후기 목록 훅은 `apps/web-react/src/features/reviews/hooks`에 둔다.
- 서버 목록 조회는 query parameter와 인덱스 설계를 함께 고려한다.
- 필터 UI는 접근성과 모바일 레이아웃을 함께 확인한다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `api design`

#### 개념

API 설계는 클라이언트와 서버가 어떤 자원과 규칙으로 대화할지 정하는 일이다.

여기서 중요한 건 `api design`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 검색, 필터, 정렬, 페이지네이션 개념에서 후기 목록 검색 조건, 좌석 필터, 태그 선택, 댓글 작성 규칙을 엔드포인트로 표현한다.
그래서 문서를 읽을 때도 먼저 "요청과 응답이 화면 하나가 아니라 도메인 자원 중심으로 설계됐는지 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

검색, 필터, 정렬, 페이지네이션 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `api design`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 REST, DTO, Pagination를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/web-react/src/features/reviews/ReviewBoardPage.tsx`이다.
그 다음으로 `apps/web-react/src/features/reviews/hooks`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `서버 목록 조회는 query parameter와 인덱스 설계를 함께 고려한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 검색어와 극장 필터는 query parameter로 표현하면 목록 조회 의미가 자연스럽다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `api design`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// api design를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `api design`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| REST | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| DTO | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Pagination | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

프론트 화면 구조 그대로 API를 만들면 다른 화면이나 기능에서 재사용하기 어렵다.
또 하나의 실수는 `api design`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 요청과 응답이 화면 하나가 아니라 도메인 자원 중심으로 설계됐는지 본다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`api design`는 검색, 필터, 정렬, 페이지네이션 개념에서 후기 목록 검색 조건, 좌석 필터, 태그 선택, 댓글 작성 규칙을 엔드포인트로 표현한다.

### `gRPC`

#### 개념

### `caching`

#### 개념

캐싱은 자주 쓰거나 계산 비용이 큰 데이터를 임시 저장해 재사용하는 기법이다.

#### 동작 방식

4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.

### `SSR`

#### 개념

SSR은 서버에서 HTML을 먼저 만들어 보내고 브라우저가 이후 상호작용을 연결하는 방식이다.

여기서 중요한 건 `SSR`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 검색, 필터, 정렬, 페이지네이션 개념에서 공개 후기 페이지의 초기 로딩과 검색 노출을 강화할 때 고려할 수 있다.
그래서 문서를 읽을 때도 먼저 "서버 렌더링 비용과 사용자별 데이터 처리 방식을 함께 설계한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

검색, 필터, 정렬, 페이지네이션 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `SSR`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Hydration, CSR, SSG를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

예를 들어 서버가 후기 목록 HTML을 미리 렌더링해 보내면 첫 화면이 빨리 보일 수 있다.

#### 동작 방식

7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 정리

예시는 authSlice로 로그인 토큰과 사용자 정보를 전역에서 관리할 수 있다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 작은 지역 상태까지 전부 Redux에 넣으면 코드가 과하게 무거워진다는 점이다.
Store, Slice, Reducer와 비교하면 책임 범위가 더 선명해진다.

### `Zustand`

#### 개념

Zustand는 React에서 간단한 전역 상태 저장소를 만들 수 있는 가벼운 상태 관리 라이브러리다.

여기서 중요한 건 `Zustand`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 검색, 필터, 정렬, 페이지네이션 개념에서 로그인 사용자, 선택한 극장, 모달 상태처럼 작은 전역 상태에 어울린다.
그래서 문서를 읽을 때도 먼저 "전역 상태가 필요한 이유와 생명주기를 분명히 한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

검색, 필터, 정렬, 페이지네이션 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Zustand`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Redux Toolkit, Store, Hook를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/web-react/src/features/reviews/ReviewBoardPage.tsx`이다.
그 다음으로 `apps/web-react/src/features/reviews/hooks`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `서버 목록 조회는 query parameter와 인덱스 설계를 함께 고려한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 useAuthStore가 token과 logout 액션을 제공하게 만들 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Zustand`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

```ts
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Zustand`를 읽을 때의 사고 순서다.

| Store | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |

예쁘게 보이는 것만 UI라고 생각하면 상태, 접근성, 오류 안내를 놓친다.
또 하나의 실수는 `UI`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.

### `a11y`

#### 개념

a11y는 accessibility의 축약 표현으로 더 많은 사람이 서비스를 사용할 수 있게 만드는 접근성이다.

여기서 중요한 건 `a11y`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 검색, 필터, 정렬, 페이지네이션 개념에서 버튼 의미, 키보드 이동, aria-label, 색 대비, 폼 라벨에 적용된다.
그래서 문서를 읽을 때도 먼저 "마우스 없이도 주요 기능을 사용할 수 있는지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

검색, 필터, 정렬, 페이지네이션 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `a11y`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Semantic HTML, Keyboard, Screen Reader를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/web-react/src/features/reviews/ReviewBoardPage.tsx`이다.
그 다음으로 `apps/web-react/src/features/reviews/hooks`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `서버 목록 조회는 query parameter와 인덱스 설계를 함께 고려한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 필터 아이콘 버튼에는 화면에 안 보이더라도 스크린 리더용 이름이 필요하다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.
