# 012_final_integration_and_demo_concepts

## 현재 파일 경로 규칙

이 문서에서 코드를 추가하거나 예시 경로를 적을 때는 아래 규칙을 따른다.

- Nest 새 기능은 `apps/nest-api/src/<domain>` 형태의 도메인 폴더에 둔다.
- React 새 기능은 `apps/web-react/src/features/<feature>` 형태의 기능 폴더에 둔다.
- React 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- React 화면, 컴포넌트, 스타일, 타입, 요청 코드는 기능 폴더 안에서 역할별로 나눈다.
- DTO는 Nest 도메인 폴더 아래 `dto`, 타입/인터페이스는 해당 기능 폴더의 `types.ts` 또는 `interfaces`에 둔다.

## 문서 목적

이 문서는 마지막 단계에서 자주 함께 등장하는 원래 키워드들을 초보자 기준으로 다시 정리한다.
여기서는 설계 지침보다 각 용어의 뜻과 맥락을 이해하는 데 집중한다.
특정 프로젝트나 레포 설명은 넣지 않고 개념 자체만 다룬다.

## 핵심 키워드

### `Full Stack Technology Roadmap`

#### `Full Stack Technology Roadmap`
- `Full Stack Technology Roadmap`은 웹 서비스를 만들 때 만나게 되는 기술 영역을 큰 흐름으로 정리한 지도 개념이다이다.
- 이 용어는 학습 지도 문맥에서 특히 자주 등장한다.
- 처음에는 낯설어 보여도 핵심은 `Full Stack Technology Roadmap`이 무엇을 설명하는 말인지 붙잡는 데 있다.
- 세부 구현보다 먼저 `Full Stack Technology Roadmap`이 어떤 역할의 개념인지 이해하면 훨씬 덜 헷갈린다.
#### 기본 뜻
- `Full Stack Technology Roadmap`이라는 이름은 보통 하나의 기술 이름이면서 동시에 하나의 개념 이름으로도 쓰일 수 있다.
- 여기서 중요한 것은 특정 제품 이름을 외우는 것보다 `Full Stack Technology Roadmap`이 가리키는 생각을 이해하는 일이다.
- `Full Stack Technology Roadmap`은 보통 학습 지도를 설명할 때 공통어처럼 쓰인다.
- 그래서 이 단어를 이해하면 주변 용어도 함께 읽히기 쉬워진다.
#### 이 용어가 가리키는 것
| 관점 | 뜻 |
| --- | --- |
| 핵심 | 웹 서비스를 만들 때 만나게 되는 기술 영역을 큰 흐름으로 정리한 지도 개념이다이다. |
| 문맥 | `Full Stack Technology Roadmap`은 학습 지도를 설명하는 자리에서 자주 나온다. |
| 감각 | 이름보다 역할을 붙잡으면 이해가 빨라진다. |
| 학습 포인트 | 비슷한 용어와 무엇이 다른지 같이 보는 것이 중요하다. |
#### 왜 자주 보이는가
- 소프트웨어를 배우다 보면 복잡한 기능도 결국 몇 개의 핵심 개념어로 설명되는 경우가 많다.
- `Full Stack Technology Roadmap`도 그런 핵심 개념어 중 하나라서 반복해서 등장한다.
- 이 용어는 단독으로만 나타나지 않고 다른 개념과 관계를 이루며 함께 설명되는 경우가 많다.
- 그래서 한 번 이해해 두면 문서나 강의, 코드 설명을 읽을 때 반복해서 도움이 된다.
- 초보자에게는 “이 단어를 어디서 다시 만나도 기본 감각은 유지된다”는 점이 특히 중요하다.
#### 비슷한 말과 차이
| 표현 | 차이 |
| --- | --- |
| `Full Stack Technology Roadmap` | 지금 설명하고 있는 중심 개념이다. |
| `Framework` | 함께 자주 나오지만 같은 뜻으로 보면 안 되는 연관 개념이다. |
| `API` | 비슷한 맥락에서 쓰이지만 강조점이 조금 다른 개념이다. |
| `data modeling` | 더 넓거나 다른 층위에서 연결될 수 있는 주변 개념이다. |
| `testing` | `Full Stack Technology Roadmap`을 이해할 때 같이 보면 흐름이 또렷해지는 용어다. |
#### 처음 배울 때 헷갈리는 점
| 헷갈리는 지점 | 정리 |
| --- | --- |
| 이름만 외우는 것 | `Full Stack Technology Roadmap`은 암기보다 역할 이해가 더 중요한 개념이다. |
| 비슷한 말과 같은 뜻으로 보는 것 | 주변 용어와 겹쳐 보여도 보통 강조점이나 범위가 다르다. |
| 구현과 개념을 섞어 보는 것 | 구현 방식은 여러 가지일 수 있지만 개념의 중심 뜻은 비교적 안정적이다. |
| 예시 하나로만 이해하는 것 | 하나의 예시보다 여러 맥락에서 같은 개념이 어떻게 쓰이는지 보는 편이 좋다. |
#### 예시로 떠올리기
- 처음 배울 때는 `Full Stack Technology Roadmap`을 아주 짧은 한 문장으로 다시 말해 보는 것이 도움이 된다.
- 예를 들어 “`Full Stack Technology Roadmap`은 웹 서비스를 만들 때 만나게 되는 기술 영역을 큰 흐름으로 정리한 지도 개념이다”처럼 풀어 읽으면 단어가 훨씬 덜 딱딱해진다.
- 그다음에는 이 개념이 문서 제목인지, 구조 이름인지, 기술 이름인지 구분해 보는 연습을 할 수 있다.
- 같은 단어가 기술 이름으로도 쓰이고 개념 이름으로도 쓰일 수 있다는 점을 받아들이면 더 자연스럽다.
- 결국 중요한 것은 이름보다 “무엇을 설명하는 말인가”라는 질문이다.
- 이 질문을 유지하면 `Full Stack Technology Roadmap`이 다른 파일이나 강의에서 다시 나와도 덜 낯설다.
#### 개념적으로 기억할 점
- `Full Stack Technology Roadmap`은 보통 세부 문법보다 더 큰 설명 단위에 속한다.
- 그래서 코드를 보기 전에도 개념 설명만으로 어느 정도 이해를 시작할 수 있다.
- 이 용어를 안다는 것은 정답을 외운다는 뜻이 아니라 사고의 손잡이를 하나 갖는다는 뜻에 가깝다.
- 용어를 손잡이처럼 잡고 나면 주변 구조가 훨씬 덜 흩어져 보인다.
- 초보자는 처음부터 완벽히 구분하려 하기보다 큰 뜻을 먼저 잡는 편이 좋다.
- 그 큰 뜻 위에 예외와 세부 차이가 나중에 얹힌다고 생각하면 부담이 줄어든다.
#### 함께 들리는 말
- `Full Stack Technology Roadmap`은 단독으로보다 주변 개념과 함께 묶여 설명되는 경우가 많다.
- 그래서 한 용어를 알면 주변 용어를 따라 읽을 수 있는 발판이 생긴다.
- `Framework`과의 관계를 떠올리면 문맥이 조금 더 분명해질 수 있다.
- `API`와 구분해 보면 범위와 초점 차이를 잡기 좋다.
- `data modeling`를 같이 보면 더 큰 지도 안에서 `Full Stack Technology Roadmap`의 위치를 보기 쉽다.
- `testing`는 같은 장이나 같은 수업에서 같이 나올 가능성이 높은 연관 개념이다.
#### 초보자 메모
- 처음에는 `Full Stack Technology Roadmap`을 한 줄 정의로만 기억해도 충분하다.
- 그다음 단계에서는 비슷한 용어와의 차이를 붙이면 이해가 훨씬 단단해진다.
- 아직 구현 경험이 없어도 이 용어의 큰 뜻을 잡는 데는 문제가 없다.
- 오히려 구현 전에 뜻을 먼저 잡아 두면 나중에 코드를 볼 때 훨씬 편하다.
- `Full Stack Technology Roadmap`이 어렵게 느껴지는 이유는 보통 이름이 낯설어서이지, 핵심 생각이 특별히 복잡해서만은 아니다.
- 그래서 낯선 이름을 쉬운 말로 다시 바꾸어 읽는 습관이 도움이 된다.
#### 한 번 더 풀어 보기
- `Full Stack Technology Roadmap`을 배운다는 것은 학습 지도 쪽 공통어 하나를 익히는 것과 같다.
- 공통어를 익히면 문서를 읽을 때 같은 설명을 더 짧게 이해할 수 있다.
- 같은 단어가 반복될수록 “이 분야에서 중요한 개념이구나”라는 감각도 생긴다.
- `Full Stack Technology Roadmap`이 자주 보인다는 사실 자체가 이 개념의 중요성을 보여 주는 신호일 수 있다.
- 또한 하나의 용어를 여러 맥락에서 다시 보면 뜻이 더 안정적으로 머릿속에 남는다.
- 따라서 정의, 비교, 예시, 오해 정리를 함께 보는 방식이 초보자에게 특히 유용하다.
#### 기억할 문장
- `Full Stack Technology Roadmap`은 웹 서비스를 만들 때 만나게 되는 기술 영역을 큰 흐름으로 정리한 지도 개념이다이다.
- 이 용어는 학습 지도를 설명할 때 자주 나온다.
- 구현보다 먼저 역할과 위치를 이해하는 것이 좋다.
- 비슷한 말과의 차이를 함께 보면 훨씬 덜 헷갈린다.
- 초보자는 한 줄 정의부터 잡고 나중에 세부 차이를 얹으면 된다.
- 결국 `Full Stack Technology Roadmap`은 이름보다 의미를 붙잡는 쪽이 더 중요하다.
#### 요약
- `Full Stack Technology Roadmap`은 이 문맥에서 빠지기 어려운 핵심 개념어다.
- 이 단어의 정의, 문맥, 비슷한 말과의 차이를 같이 기억하면 이해가 오래간다.
- 한 번 뜻을 잡아 두면 이후의 더 어려운 설명도 훨씬 읽기 쉬워진다.
- 그래서 `Full Stack Technology Roadmap`은 개념 공부의 출발점 역할을 한다.

- 그래서 코드를 보기 전에도 개념 설명만으로 어느 정도 이해를 시작할 수 있다.
- 이 용어를 안다는 것은 정답을 외운다는 뜻이 아니라 사고의 손잡이를 하나 갖는다는 뜻에 가깝다.
- 용어를 손잡이처럼 잡고 나면 주변 구조가 훨씬 덜 흩어져 보인다.
- 초보자는 처음부터 완벽히 구분하려 하기보다 큰 뜻을 먼저 잡는 편이 좋다.
- 그 큰 뜻 위에 예외와 세부 차이가 나중에 얹힌다고 생각하면 부담이 줄어든다.
#### 함께 들리는 말
- `testing`은 단독으로보다 주변 개념과 함께 묶여 설명되는 경우가 많다.
- 그래서 한 용어를 알면 주변 용어를 따라 읽을 수 있는 발판이 생긴다.
- `Unit`과의 관계를 떠올리면 문맥이 조금 더 분명해질 수 있다.
- `Integration`와 구분해 보면 범위와 초점 차이를 잡기 좋다.
- `E2E`를 같이 보면 더 큰 지도 안에서 `testing`의 위치를 보기 쉽다.
- `Framework`는 같은 장이나 같은 수업에서 같이 나올 가능성이 높은 연관 개념이다.
#### 초보자 메모
- 처음에는 `testing`을 한 줄 정의로만 기억해도 충분하다.
- 그다음 단계에서는 비슷한 용어와의 차이를 붙이면 이해가 훨씬 단단해진다.
- 아직 구현 경험이 없어도 이 용어의 큰 뜻을 잡는 데는 문제가 없다.
- 오히려 구현 전에 뜻을 먼저 잡아 두면 나중에 코드를 볼 때 훨씬 편하다.
- `testing`이 어렵게 느껴지는 이유는 보통 이름이 낯설어서이지, 핵심 생각이 특별히 복잡해서만은 아니다.
- 그래서 낯선 이름을 쉬운 말로 다시 바꾸어 읽는 습관이 도움이 된다.
#### 한 번 더 풀어 보기
- `testing`을 배운다는 것은 품질 확인 쪽 공통어 하나를 익히는 것과 같다.
- 공통어를 익히면 문서를 읽을 때 같은 설명을 더 짧게 이해할 수 있다.
- 같은 단어가 반복될수록 “이 분야에서 중요한 개념이구나”라는 감각도 생긴다.
- `testing`이 자주 보인다는 사실 자체가 이 개념의 중요성을 보여 주는 신호일 수 있다.
- 또한 하나의 용어를 여러 맥락에서 다시 보면 뜻이 더 안정적으로 머릿속에 남는다.
- 따라서 정의, 비교, 예시, 오해 정리를 함께 보는 방식이 초보자에게 특히 유용하다.
#### 기억할 문장
- `testing`은 프로그램이 기대한 대로 동작하는지 확인하는 개념이다이다.
- 이 용어는 품질 확인를 설명할 때 자주 나온다.
- 구현보다 먼저 역할과 위치를 이해하는 것이 좋다.
- 비슷한 말과의 차이를 함께 보면 훨씬 덜 헷갈린다.
- 초보자는 한 줄 정의부터 잡고 나중에 세부 차이를 얹으면 된다.
- 결국 `testing`은 이름보다 의미를 붙잡는 쪽이 더 중요하다.
#### 요약
- `testing`은 이 문맥에서 빠지기 어려운 핵심 개념어다.
- 이 단어의 정의, 문맥, 비슷한 말과의 차이를 같이 기억하면 이해가 오래간다.
- 한 번 뜻을 잡아 두면 이후의 더 어려운 설명도 훨씬 읽기 쉬워진다.
- 그래서 `testing`은 개념 공부의 출발점 역할을 한다.

#### `Unit`
- `Unit`은 아주 작은 코드 단위를 따로 떼어 확인하는 테스트 개념이다이다.
- 이 용어는 테스트 종류 문맥에서 특히 자주 등장한다.
- 처음에는 낯설어 보여도 핵심은 `Unit`이 무엇을 설명하는 말인지 붙잡는 데 있다.
- 세부 구현보다 먼저 `Unit`이 어떤 역할의 개념인지 이해하면 훨씬 덜 헷갈린다.
#### 기본 뜻
- `Unit`이라는 이름은 보통 하나의 기술 이름이면서 동시에 하나의 개념 이름으로도 쓰일 수 있다.
- 여기서 중요한 것은 특정 제품 이름을 외우는 것보다 `Unit`이 가리키는 생각을 이해하는 일이다.
- `Unit`은 보통 테스트 종류를 설명할 때 공통어처럼 쓰인다.
- 그래서 이 단어를 이해하면 주변 용어도 함께 읽히기 쉬워진다.

| 학습 포인트 | 비슷한 용어와 무엇이 다른지 같이 보는 것이 중요하다. |
#### 왜 자주 보이는가
- 소프트웨어를 배우다 보면 복잡한 기능도 결국 몇 개의 핵심 개념어로 설명되는 경우가 많다.
- `Unit`도 그런 핵심 개념어 중 하나라서 반복해서 등장한다.
- 이 용어는 단독으로만 나타나지 않고 다른 개념과 관계를 이루며 함께 설명되는 경우가 많다.
- 그래서 한 번 이해해 두면 문서나 강의, 코드 설명을 읽을 때 반복해서 도움이 된다.
- 초보자에게는 “이 단어를 어디서 다시 만나도 기본 감각은 유지된다”는 점이 특히 중요하다.

#### 개념적으로 기억할 점
- `HTTP 3xx`은 보통 세부 문법보다 더 큰 설명 단위에 속한다.
- 그래서 코드를 보기 전에도 개념 설명만으로 어느 정도 이해를 시작할 수 있다.

- 따라서 정의, 비교, 예시, 오해 정리를 함께 보는 방식이 초보자에게 특히 유용하다.
#### 기억할 문장
- `HTTP 4xx`은 클라이언트 쪽 요청 문제를 나타낼 때 쓰이는 HTTP 상태 코드 범주 개념이다이다.
- 이 용어는 HTTP 상태 코드를 설명할 때 자주 나온다.
- 구현보다 먼저 역할과 위치를 이해하는 것이 좋다.
- 비슷한 말과의 차이를 함께 보면 훨씬 덜 헷갈린다.
- 초보자는 한 줄 정의부터 잡고 나중에 세부 차이를 얹으면 된다.
- 결국 `HTTP 4xx`은 이름보다 의미를 붙잡는 쪽이 더 중요하다.
#### 요약
- `HTTP 4xx`은 이 문맥에서 빠지기 어려운 핵심 개념어다.
- 이 단어의 정의, 문맥, 비슷한 말과의 차이를 같이 기억하면 이해가 오래간다.
- 한 번 뜻을 잡아 두면 이후의 더 어려운 설명도 훨씬 읽기 쉬워진다.
- 그래서 `HTTP 4xx`은 개념 공부의 출발점 역할을 한다.

#### `HTTP 5xx`
- `HTTP 5xx`은 서버 쪽 문제를 나타낼 때 쓰이는 HTTP 상태 코드 범주 개념이다이다.
- 이 용어는 HTTP 상태 코드 문맥에서 특히 자주 등장한다.
- 처음에는 낯설어 보여도 핵심은 `HTTP 5xx`이 무엇을 설명하는 말인지 붙잡는 데 있다.
- 세부 구현보다 먼저 `HTTP 5xx`이 어떤 역할의 개념인지 이해하면 훨씬 덜 헷갈린다.
#### 기본 뜻
- `HTTP 5xx`이라는 이름은 보통 하나의 기술 이름이면서 동시에 하나의 개념 이름으로도 쓰일 수 있다.
- 여기서 중요한 것은 특정 제품 이름을 외우는 것보다 `HTTP 5xx`이 가리키는 생각을 이해하는 일이다.
- `HTTP 5xx`은 보통 HTTP 상태 코드를 설명할 때 공통어처럼 쓰인다.
- 그래서 이 단어를 이해하면 주변 용어도 함께 읽히기 쉬워진다.

| 학습 포인트 | 비슷한 용어와 무엇이 다른지 같이 보는 것이 중요하다. |
#### 왜 자주 보이는가
- 소프트웨어를 배우다 보면 복잡한 기능도 결국 몇 개의 핵심 개념어로 설명되는 경우가 많다.
- `HTTP 5xx`도 그런 핵심 개념어 중 하나라서 반복해서 등장한다.
- 이 용어는 단독으로만 나타나지 않고 다른 개념과 관계를 이루며 함께 설명되는 경우가 많다.
- 그래서 한 번 이해해 두면 문서나 강의, 코드 설명을 읽을 때 반복해서 도움이 된다.
- 초보자에게는 “이 단어를 어디서 다시 만나도 기본 감각은 유지된다”는 점이 특히 중요하다.

| `HTTP 5xx` | 지금 설명하고 있는 중심 개념이다. |

| `HTTP 3xx` | 비슷한 맥락에서 쓰이지만 강조점이 조금 다른 개념이다. |
| `HTTP 4xx` | 더 넓거나 다른 층위에서 연결될 수 있는 주변 개념이다. |
| `state` | `HTTP 5xx`을 이해할 때 같이 보면 흐름이 또렷해지는 용어다. |

#### 예시로 떠올리기
- 처음 배울 때는 `HTTP 5xx`을 아주 짧은 한 문장으로 다시 말해 보는 것이 도움이 된다.
- 예를 들어 “`HTTP 5xx`은 서버 쪽 문제를 나타낼 때 쓰이는 HTTP 상태 코드 범주 개념이다”처럼 풀어 읽으면 단어가 훨씬 덜 딱딱해진다.
- 그다음에는 이 개념이 문서 제목인지, 구조 이름인지, 기술 이름인지 구분해 보는 연습을 할 수 있다.
- 같은 단어가 기술 이름으로도 쓰이고 개념 이름으로도 쓰일 수 있다는 점을 받아들이면 더 자연스럽다.
- 결국 중요한 것은 이름보다 “무엇을 설명하는 말인가”라는 질문이다.
- 이 질문을 유지하면 `HTTP 5xx`이 다른 파일이나 강의에서 다시 나와도 덜 낯설다.
#### 개념적으로 기억할 점
- `HTTP 5xx`은 보통 세부 문법보다 더 큰 설명 단위에 속한다.
- 그래서 코드를 보기 전에도 개념 설명만으로 어느 정도 이해를 시작할 수 있다.
- 이 용어를 안다는 것은 정답을 외운다는 뜻이 아니라 사고의 손잡이를 하나 갖는다는 뜻에 가깝다.
- 용어를 손잡이처럼 잡고 나면 주변 구조가 훨씬 덜 흩어져 보인다.
- 초보자는 처음부터 완벽히 구분하려 하기보다 큰 뜻을 먼저 잡는 편이 좋다.
- 그 큰 뜻 위에 예외와 세부 차이가 나중에 얹힌다고 생각하면 부담이 줄어든다.
#### 함께 들리는 말
- `HTTP 5xx`은 단독으로보다 주변 개념과 함께 묶여 설명되는 경우가 많다.
- 그래서 한 용어를 알면 주변 용어를 따라 읽을 수 있는 발판이 생긴다.
- `error handling`과의 관계를 떠올리면 문맥이 조금 더 분명해질 수 있다.
- `HTTP 3xx`와 구분해 보면 범위와 초점 차이를 잡기 좋다.
- `HTTP 4xx`를 같이 보면 더 큰 지도 안에서 `HTTP 5xx`의 위치를 보기 쉽다.
- `state`는 같은 장이나 같은 수업에서 같이 나올 가능성이 높은 연관 개념이다.
#### 초보자 메모
- 처음에는 `HTTP 5xx`을 한 줄 정의로만 기억해도 충분하다.
- 그다음 단계에서는 비슷한 용어와의 차이를 붙이면 이해가 훨씬 단단해진다.
- 아직 구현 경험이 없어도 이 용어의 큰 뜻을 잡는 데는 문제가 없다.
- 오히려 구현 전에 뜻을 먼저 잡아 두면 나중에 코드를 볼 때 훨씬 편하다.
- `HTTP 5xx`이 어렵게 느껴지는 이유는 보통 이름이 낯설어서이지, 핵심 생각이 특별히 복잡해서만은 아니다.

#### 개념적으로 기억할 점

- 그래서 코드를 보기 전에도 개념 설명만으로 어느 정도 이해를 시작할 수 있다.

#### 함께 들리는 말

#### `Bootstrap`
- `Bootstrap`은 미리 준비된 스타일과 컴포넌트를 제공하는 프론트엔드 UI 프레임워크 이름이다이다.

#### 기본 뜻
- `Bootstrap`이라는 이름은 보통 하나의 기술 이름이면서 동시에 하나의 개념 이름으로도 쓰일 수 있다.
- 여기서 중요한 것은 특정 제품 이름을 외우는 것보다 `Bootstrap`이 가리키는 생각을 이해하는 일이다.
- `Bootstrap`은 보통 UI 프레임워크를 설명할 때 공통어처럼 쓰인다.
- 그래서 이 단어를 이해하면 주변 용어도 함께 읽히기 쉬워진다.
