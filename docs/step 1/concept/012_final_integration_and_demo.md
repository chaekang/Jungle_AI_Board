# 012 최종 통합과 데모 개념

## 문서 목적

최종 데모 단계에서 기술 로드맵, UI, 접근성, 테스트, 로깅, 에러 처리, UI 라이브러리 선택을 기능 완성도와 연결한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- 데모는 사용자가 실제로 겪는 흐름을 기준으로 준비한다.
- 로그인 없이 게시판 보기, 로그인 후 후기 작성, 좌석배치도 확인, 내 글 수정/삭제를 핵심 여정으로 둔다.
- 테스트와 로그는 데모 실패를 줄이는 안전장치다.
- UI 라이브러리는 도입 속도와 디자인 통일성을 함께 평가한다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `Full Stack Technology Roadmap`

#### 개념

풀스택 기술 로드맵은 화면, 서버, DB, 배포, 테스트, 운영까지 배워야 할 범위를 정리한 지도다.

여기서 중요한 건 `Full Stack Technology Roadmap`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 최종 통합과 데모 개념에서 현재 프로젝트에서 React, Nest, PostgreSQL, 인증, 검색, AI 추천을 단계별로 연결한다.
그래서 문서를 읽을 때도 먼저 "각 단계가 실제 기능 하나와 연결되어 있는지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

최종 통합과 데모 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Full Stack Technology Roadmap`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Frontend, Backend, Database를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `데모는 사용자가 실제로 겪는 흐름을 기준으로 준비한다.`이다.
그 다음으로 `로그인 없이 게시판 보기, 로그인 후 후기 작성, 좌석배치도 확인, 내 글 수정/삭제를 핵심 여정으로 둔다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `테스트와 로그는 데모 실패를 줄이는 안전장치다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 먼저 CRUD와 인증을 안정화하고, 그다음 검색과 좌석 지도, 마지막으로 AI 추천을 붙인다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Full Stack Technology Roadmap`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Full Stack Technology Roadmap를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Full Stack Technology Roadmap`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Frontend | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Backend | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Database | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

기술 이름을 많이 외우는 것을 로드맵 완성으로 착각하면 실제 구현력이 늘지 않는다.
또 하나의 실수는 `Full Stack Technology Roadmap`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 각 단계가 실제 기능 하나와 연결되어 있는지 확인한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?

### `a11y`

#### 개념

a11y는 accessibility의 축약 표현으로 더 많은 사람이 서비스를 사용할 수 있게 만드는 접근성이다.

여기서 중요한 건 `a11y`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 최종 통합과 데모 개념에서 버튼 의미, 키보드 이동, aria-label, 색 대비, 폼 라벨에 적용된다.
그래서 문서를 읽을 때도 먼저 "마우스 없이도 주요 기능을 사용할 수 있는지 확인한다"라는 질문을 붙여 두면 좋다.

#### 구현할 때 확인할 질문

#### 정리

예시는 rating sort 함수가 높은순과 낮은순을 올바르게 정렬하는지 검사한다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 외부 API와 DB까지 모두 붙이면 단위 테스트가 느리고 불안정해진다는 점이다.
Mock, Pure Function, Jest와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Unit test`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `Integration test`

#### 개념

통합 테스트는 여러 모듈이 함께 연결되었을 때 흐름이 맞는지 확인한다.

여기서 중요한 건 `Integration test`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 최종 통합과 데모 개념에서 Nest controller, service, Prisma가 함께 후기 작성 요청을 처리하는지 검증한다.
그래서 문서를 읽을 때도 먼저 "어떤 경계를 실제로 붙이고 어떤 경계를 대체할지 정한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

최종 통합과 데모 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Integration test`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 API Test, Database, Fixture를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `데모는 사용자가 실제로 겪는 흐름을 기준으로 준비한다.`이다.
그 다음으로 `로그인 없이 게시판 보기, 로그인 후 후기 작성, 좌석배치도 확인, 내 글 수정/삭제를 핵심 여정으로 둔다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `테스트와 로그는 데모 실패를 줄이는 안전장치다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 로그인 후 POST /seat-reviews가 DB에 후기를 만들고 응답을 돌려주는지 확인한다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Integration test`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

}

updateScreen(result.data);
```

모든 통합 테스트가 실제 외부 서비스에 의존하면 테스트가 쉽게 깨진다.

### `error handling (HTTP 3xx, 4xx, 5xx)`

#### 개념

HTTP 에러 처리는 실패 이유를 상태 코드와 메시지로 일관되게 전달하는 방식이다.

여기서 중요한 건 `error handling (HTTP 3xx, 4xx, 5xx)`를 단어 뜻으로만 외우지 않는 것이다.

### `MUI`

#### 개념

MUI는 Material Design 기반의 React UI 컴포넌트 라이브러리다.

여기서 중요한 건 `MUI`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 최종 통합과 데모 개념에서 폼, 모달, 버튼, 테이블을 빠르게 만들 수 있지만 현재 디자인 톤과 맞는지 검토해야 한다.
그래서 문서를 읽을 때도 먼저 "커스터마이징 비용과 개발 속도 이득을 비교한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

최종 통합과 데모 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `MUI`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Component Library, Theme, Material Design를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `데모는 사용자가 실제로 겪는 흐름을 기준으로 준비한다.`이다.
그 다음으로 `로그인 없이 게시판 보기, 로그인 후 후기 작성, 좌석배치도 확인, 내 글 수정/삭제를 핵심 여정으로 둔다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `테스트와 로그는 데모 실패를 줄이는 안전장치다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 Dialog와 Button을 사용하면 접근성 기본값이 있는 모달을 빠르게 만들 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `MUI`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

  showErrorMessage(result.reason);
}

```

필요하면 `테스트와 로그는 데모 실패를 줄이는 안전장치다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 container, row, col 클래스로 반응형 레이아웃을 빠르게 잡을 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Bootstrap`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Bootstrap를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Bootstrap`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| CSS Framework | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Grid System | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Component | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

기본 Bootstrap 느낌이 강해져 서비스 고유의 디자인이 흐려질 수 있다.
또 하나의 실수는 `Bootstrap`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 속도와 독자적인 디자인 사이에서 어느 쪽이 중요한지 결정한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`Bootstrap`는 최종 통합과 데모 개념에서 빠른 프로토타입에는 좋지만 현재 프로젝트의 커스텀 감성과 충돌할 수 있다.
예시는 container, row, col 클래스로 반응형 레이아웃을 빠르게 잡을 수 있다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 기본 Bootstrap 느낌이 강해져 서비스 고유의 디자인이 흐려질 수 있다는 점이다.
CSS Framework, Grid System, Component와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Bootstrap`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.
