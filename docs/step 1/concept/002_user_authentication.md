# 002 사용자 인증 개념

## 문서 목적

로그인과 회원가입을 구현할 때 나오는 인증, 토큰, 보안, HTTP 에러, React 폼 처리 개념을 구분한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- Nest 인증 코드는 `apps/nest-api/src/auth`에 둔다.
- 사용자 조회와 사용자 모델 코드는 `apps/nest-api/src/users`에 둔다.
- React 인증 화면과 스타일은 `apps/web-react/src/features/auth`에 둔다.
- 인증 토큰을 API 요청에 붙이는 공통 로직은 shared API 계층에서 관리한다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `Authentication`

#### 개념

인증은 요청을 보낸 사용자가 누구인지 확인하는 과정이다.

여기서 중요한 건 `Authentication`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 사용자 인증 개념에서 로그인한 사용자만 후기를 작성하고, 토큰으로 현재 사용자를 식별한다.
그래서 문서를 읽을 때도 먼저 "요청마다 사용자를 어떻게 식별하고 신뢰하는지 추적한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

사용자 인증 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Authentication`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Authorization, JWT, Session를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/auth`이다.
그 다음으로 `apps/nest-api/src/users`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/features/auth`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 이메일과 비밀번호가 맞으면 서버가 사용자 정보를 확인하고 토큰을 발급한다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Authentication`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Authentication를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Authentication`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Authorization | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| JWT | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Session | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

인증과 권한을 같은 말로 보면 로그인만 했다고 모든 작업이 허용된다고 착각한다.
또 하나의 실수는 `Authentication`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 요청마다 사용자를 어떻게 식별하고 신뢰하는지 추적한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`Authentication`는 사용자 인증 개념에서 로그인한 사용자만 후기를 작성하고, 토큰으로 현재 사용자를 식별한다.
예시는 이메일과 비밀번호가 맞으면 서버가 사용자 정보를 확인하고 토큰을 발급한다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 인증과 권한을 같은 말로 보면 로그인만 했다고 모든 작업이 허용된다고 착각한다는 점이다.
Authorization, JWT, Session와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Authentication`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `JWT`

#### 개념

JWT는 JSON 형태의 정보를 서명해 클라이언트가 들고 다닐 수 있게 만든 토큰 형식이다.

여기서 중요한 건 `JWT`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 사용자 인증 개념에서 로그인 후 access token을 저장하고 API 요청 Authorization 헤더에 붙인다.
그래서 문서를 읽을 때도 먼저 "토큰 만료 시간, 저장 위치, 갱신 전략, 로그아웃 처리를 함께 본다"라는 질문을 붙여 두면 좋다.

### `OAuth2`

#### 개념

OAuth2는 사용자가 비밀번호를 직접 넘기지 않고 외부 서비스 권한을 위임하는 표준 흐름이다.

그래서 문서를 읽을 때도 먼저 "권한 위임 대상이 무엇인지, 사용자 식별 정보는 어디서 얻는지 분리해서 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

사용자 인증 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `OAuth2`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Authorization Code, Scope, Access Token를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

### `HTTPS`

#### 개념

HTTPS는 HTTP 통신을 TLS로 암호화해 중간에서 내용을 훔쳐보거나 바꾸기 어렵게 만든다.

여기서 중요한 건 `HTTPS`를 단어 뜻으로만 외우지 않는 것이다.

그래서 문서를 읽을 때도 먼저 "배포 환경의 리다이렉트, 인증서, mixed content 문제를 점검한다"라는 질문을 붙여 두면 좋다.

### `CORS`

#### 개념

CORS는 브라우저가 다른 출처의 리소스 요청을 허용할지 제한하는 보안 정책이다.

여기서 중요한 건 `CORS`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 사용자 인증 개념에서 React 개발 서버와 Nest API 서버가 포트가 다를 때 CORS 설정이 필요하다.
그래서 문서를 읽을 때도 먼저 "허용 origin, method, header, credentials 옵션을 필요한 만큼만 연다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

사용자 인증 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `CORS`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Origin, Preflight, Credentials를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/auth`이다.
그 다음으로 `apps/nest-api/src/users`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/features/auth`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 http://localhost:5173에서 http://localhost:3000 API를 호출하면 출처가 다르다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `CORS`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts

const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `CORS`를 읽을 때의 사고 순서다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Origin | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Preflight | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Credentials | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

또 하나의 실수는 `CORS`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.

따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

### `REST`

#### 개념

REST는 자원을 URI로 표현하고 HTTP 메서드로 행위를 나타내는 API 설계 스타일이다.

여기서 중요한 건 `REST`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 사용자 인증 개념에서 후기 목록은 GET, 생성은 POST, 수정은 PATCH, 삭제는 DELETE로 표현한다.
그래서 문서를 읽을 때도 먼저 "URI는 명사 중심인지, 메서드가 동작을 표현하는지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

사용자 인증 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `REST`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 HTTP Method, Resource, Status Code를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/auth`이다.
그 다음으로 `apps/nest-api/src/users`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/features/auth`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 PATCH /seat-reviews/:id는 특정 후기 자원을 일부 수정한다는 의미가 분명하다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `REST`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts

const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| HTTP Method | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Resource | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |

3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts

const userInput = "화면에서 들어온 값";

  showErrorMessage(result.reason);

```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `React Controlled Input Pattern`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| state | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| event.target.value | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Form | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

value 없이 defaultValue만 쓰다가 나중에 상태와 화면 값이 어긋날 수 있다.
또 하나의 실수는 `React Controlled Input Pattern`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 입력값의 단일 출처가 React state인지 확인한다?

- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`React Controlled Input Pattern`는 사용자 인증 개념에서 로그인 폼과 후기 작성 폼에서 input value를 state와 연결한다.
예시는 value={email}과 onChange={setEmail} 구조가 controlled input의 기본이다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 value 없이 defaultValue만 쓰다가 나중에 상태와 화면 값이 어긋날 수 있다는 점이다.
state, event.target.value, Form와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `React Controlled Input Pattern`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `event.target.value`

#### 개념

event.target.value는 입력 이벤트가 발생한 요소의 현재 문자열 값을 읽는 속성이다.

여기서 중요한 건 `event.target.value`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 사용자 인증 개념에서 검색창, 로그인 이메일, 후기 내용 입력 값을 state로 옮길 때 사용한다.
그래서 문서를 읽을 때도 먼저 "이 이벤트가 어떤 요소에서 왔는지와 값 타입이 문자열인지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

사용자 인증 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `event.target.value`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 SyntheticEvent, Controlled Input, Form State를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/auth`이다.
그 다음으로 `apps/nest-api/src/users`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/features/auth`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 onChange={(event) => setSearchText(event.target.value)}처럼 쓴다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.
