# 002_user_authentication_concepts

## 이 문서의 역할

이 문서는 `002 사용자 인증` 단계에서 꼭 알아야 하는 인증 / 보안 개념을 자세히 정리한 문서다.
실제 구현은 [docs/steps/002_user_authentication.md](../steps/002_user_authentication.md)에서 따라가고,
이 문서는 "왜 이렇게 하는지"와 "다른 선택지는 무엇인지"를 이해하는 데 초점을 둔다.

이 프로젝트에서 인증은 단순 로그인 기능이 아니다.
후기 작성자, 댓글 작성자, 수정 권한, 삭제 권한을 사용자와 연결하는 기반이다.
즉 인증이 흔들리면 게시판의 데이터 신뢰성과 권한 체계가 모두 흔들린다.

## 인증을 보는 기본 시각

인증과 권한은 같은 말처럼 들리지만 다르다.

- 인증(Authentication): "누구인가?"
- 인가 / 권한(Authorization): "무엇을 할 수 있는가?"

예를 들어:

- 로그인 성공은 인증 문제
- 내가 쓴 후기만 수정 가능은 권한 문제

이 구분을 초반부터 명확히 하고 가면 이후 Guard, Strategy, Service 책임도 자연스럽게 정리된다.

## 핵심 키워드

### `Authentication`

정의:

Authentication은 사용자의 신원을 확인하는 과정이다.
로그인 폼을 제출하고 비밀번호를 비교하고, 성공 시 인증 상태를 부여하는 흐름 전체가 여기에 포함된다.

언제 쓰는가:

- 로그인
- 로그아웃
- 보호된 API 접근
- 현재 로그인 사용자 확인

어떻게 적용하는가:

이 프로젝트에서는 이메일 / 비밀번호 로그인 후 JWT Access Token을 발급하는 형태로 구현한다.
즉 사용자가 로그인하면 서버가 토큰을 발급하고, 이후 요청마다 그 토큰을 헤더로 전송해 인증 상태를 증명한다.

장점:

- 작성자 식별이 가능해진다
- 보호된 리소스를 만들 수 있다
- 사용자별 데이터 경험을 제공할 수 있다

문제 / 주의점:

- 인증만 붙이고 권한 체크를 빼먹기 쉽다
- 로그인은 되지만 토큰 만료, 저장, 복원 같은 운영 이슈가 생긴다

면접 포인트:

"인증은 단순 로그인 성공 여부가 아니라, 이후 모든 보호된 리소스 접근의 출발점입니다. 저는 항상 인증과 권한을 분리해서 설계하려고 합니다."

### `Session` vs `JWT`

정의:

- `Session`: 서버가 로그인 상태를 저장하고, 클라이언트는 세션 식별자만 보관
- `JWT`: 서버가 서명한 토큰 자체에 인증 정보를 담고, 클라이언트가 이를 보관

언제 Session을 쓰는가:

- 전통적인 서버 렌더링 웹 앱
- 서버가 로그인 상태를 직접 제어하고 싶을 때
- 세션 무효화가 매우 중요할 때

언제 JWT를 쓰는가:

- 프론트와 API가 분리된 구조
- 모바일 앱, 웹 앱 등 여러 클라이언트가 같은 인증 체계를 쓸 때
- 서버가 상태를 덜 들고 가는 구조가 필요할 때

현재 프로젝트에서 JWT를 택한 이유:

- React와 Nest API가 분리돼 있다
- 초기 MVP에서 구현이 단순하다
- `Authorization` 헤더 기반 접근 제어가 직관적이다

JWT의 장점:

- 프론트 분리 구조와 잘 맞는다
- 서버 세션 저장소 없이 시작하기 쉽다
- API 서버 확장 시 구조가 단순해진다

JWT의 문제 / 주의점:

- 토큰 탈취 시 만료 전까지 악용될 수 있다
- 로그아웃 / 강제 만료 처리가 세션보다 까다롭다
- 토큰 저장 위치에 따라 보안 고려가 달라진다

면접 포인트:

"JWT는 서버가 상태를 덜 가지는 장점이 있지만, 토큰 탈취와 무효화 전략을 같이 봐야 합니다. 그래서 저는 단순히 JWT가 좋다고 말하기보다, 서비스 구조와 보안 요구를 같이 봅니다."

### `JWT`

정의:

JWT는 JSON Web Token의 약자이며, 서명된 토큰 형식이다.
중요한 점은 "암호화된 보관함"이 아니라 "위변조 검증이 가능한 서명된 데이터"라는 점이다.

보통 담는 값:

- `sub`: 사용자 식별자
- `email`: 보조 식별 정보
- `iat`, `exp`: 발급 / 만료 관련 정보

언제 쓰는가:

- 로그인 후 인증 상태 전달
- 보호된 API 호출
- 현재 사용자 식별

어떻게 사용하는가:

1. 로그인 성공
2. 서버가 JWT 발급
3. 클라이언트가 저장
4. 이후 요청 헤더에 `Authorization: Bearer <token>` 전달
5. 서버가 서명과 만료를 검증

장점:

- 서버가 매 요청마다 세션 조회를 하지 않아도 된다
- 클라이언트 분리 구조에서 다루기 쉽다

문제 / 주의점:

- 민감한 개인정보를 많이 넣으면 안 된다
- `BigInt` 같은 비직렬화 값은 넣으면 안 된다
- 너무 긴 만료 시간은 위험하다

면접 포인트:

"JWT에는 최소한의 식별 정보만 넣는 것이 좋습니다. 저는 보통 `sub`와 몇 개의 보조 정보만 담고, 상세 사용자 정보는 서버에서 다시 조회하는 방식을 선호합니다."

### `OAuth2`, `OIDC`

정의:

- `OAuth2`: 사용자의 비밀번호를 직접 다루지 않고도 외부 서비스 권한을 위임받는 표준
- `OIDC`: OAuth2 위에서 인증 정보까지 표준화한 계층

언제 쓰는가:

- Google 로그인
- GitHub 로그인
- 회사 계정 기반 SSO

현재 단계에서의 위치:

이번 MVP에서는 직접 구현하지 않지만, "나중에 확장될 가능성이 높은 인증 방식"으로 이해하면 된다.

장점:

- 회원가입 / 비밀번호 관리 부담 감소
- 사용자 진입 장벽 감소

문제 / 주의점:

- 외부 제공자 의존성이 생긴다
- 계정 연결 정책이 필요하다
- 콜백 URL, 토큰 교환, 사용자 정보 동기화까지 고려해야 한다

면접 포인트:

"OAuth2는 권한 위임, OIDC는 신원 확인까지 포함한다고 설명할 수 있어야 합니다. 실무에서는 둘을 함께 쓰는 경우가 많습니다."

### `security`

정의:

보안은 인증 기능을 안전하게 운영하기 위해 필요한 전반적인 원칙이다.
로그인 성공만 되는 것은 보안이 아니다.

이 단계에서 최소한 지켜야 하는 것:

- 비밀번호 평문 저장 금지
- `JWT_SECRET` 코드 하드코딩 금지
- 민감 정보 응답 노출 금지
- 인증 실패와 권한 실패 분리

언제 중요해지는가:

항상 중요하지만, 인증 단계에서 특히 중요하다.
초기 구현이 그대로 운영 구조가 되는 경우가 많기 때문이다.

면접 포인트:

"인증 기능은 돌아가는 것보다 안전하게 돌아가는 것이 중요합니다. 평문 저장 금지, 비밀키 분리, 응답 최소화는 MVP라도 양보하면 안 되는 기준이라고 생각합니다."

### `HTTPS`

정의:

브라우저와 서버 사이 통신을 암호화하는 프로토콜이다.

언제 쓰는가:

- 배포 환경에서는 사실상 항상
- 로그인, 결제, 개인정보 전송이 있는 모든 환경

왜 중요한가:

JWT든 Session이든 인증 정보가 네트워크를 타고 이동한다.
HTTPS가 없으면 중간 탈취 위험이 커진다.

장점:

- 통신 구간 암호화
- 토큰 / 비밀번호 노출 위험 감소

문제 / 주의점:

- 로컬 개발에서는 체감이 적어서 무시하기 쉽다
- 운영 배포 시 인증서와 프록시 설정을 함께 봐야 한다

면접 포인트:

"인증 설계에서 저장 위치만큼 중요한 것이 전송 구간 보안입니다. HTTPS는 선택이 아니라 기본 전제라고 봅니다."

### `CSRF/CORS`

정의:

- `CORS`: 브라우저가 다른 출처의 자원 요청을 제어하는 정책
- `CSRF`: 사용자가 의도하지 않은 요청을 보내게 만드는 공격

언제 중요한가:

프론트와 백엔드가 분리된 순간 CORS는 바로 중요해진다.
CSRF는 특히 쿠키 기반 인증에서 더 자주 강조되지만, 인증 구조 전체에서 이해가 필요하다.

현재 프로젝트에서의 적용:

- React가 `localhost:5173`
- Nest API가 `localhost:3000`

이므로 CORS 허용 설정이 필요하다.

장점:

- CORS를 정확히 설정하면 허용된 프론트만 API를 호출하도록 통제할 수 있다

문제 / 주의점:

- `origin: *`처럼 너무 넓게 열면 운영 보안이 약해질 수 있다
- CORS는 서버 보안 기능이라기보다 브라우저 정책 제어에 가깝다는 점을 알아야 한다

면접 포인트:

"CORS는 서버 간 통신을 막는 게 아니라 브라우저 출처 정책을 제어하는 것입니다. 그래서 API가 열려 있는지와는 별개로 브라우저 UX와 보안에 영향을 줍니다."

### `Rate limit`

정의:

짧은 시간 동안 특정 사용자가 너무 많은 요청을 보내지 못하게 제한하는 전략이다.

언제 쓰는가:

- 로그인
- 회원가입
- 비밀번호 재설정
- 공개 검색 API

왜 중요한가:

로그인 API는 브루트포스 공격 대상이 되기 쉽다.
따라서 운영 시점에는 rate limit이 거의 필수에 가깝다.

장점:

- 무차별 대입 공격 완화
- 서버 부하 완화

문제 / 주의점:

- 너무 빡빡하면 정상 사용자도 막힌다
- IP 기반 제한만으로는 프록시 환경이나 공유망에서 부작용이 있을 수 있다

면접 포인트:

"Rate limit은 보안과 안정성을 동시에 위한 장치입니다. 다만 사용자 경험을 해치지 않도록 어떤 기준으로 제한할지 설계가 필요합니다."

### `OWASP Top 10`

정의:

웹 애플리케이션에서 자주 발생하는 주요 보안 위험 목록이다.

인증 단계와 가장 관련 깊은 항목:

- 취약한 인증
- 민감 정보 노출
- 접근 제어 실패

언제 유용한가:

보안 체크리스트를 만들 때
구현 후 위험 지점을 점검할 때

장점:

- 보안을 막연한 감이 아니라 체크리스트로 볼 수 있다

문제 / 주의점:

- 목록만 외운다고 보안이 해결되진 않는다
- 서비스 맥락에 맞는 적용이 필요하다

면접 포인트:

"OWASP Top 10은 정답집이라기보다 위험 감수성 체크리스트로 활용합니다. 인증 기능을 만들 때는 특히 취약한 인증과 접근 제어 실패를 먼저 봅니다."

### `Rest`, `error handling (HTTP 3xx, 4xx, 5xx)`

정의:

REST는 리소스 중심으로 URL과 HTTP 메서드를 설계하는 방식이다.
에러 핸들링은 실패를 상태 코드와 메시지로 일관되게 표현하는 것이다.

이 프로젝트의 인증 API 예:

- `POST /auth/register`
- `POST /auth/login`
- `GET /auth/me`

중요한 상태 코드:

- `400`: 입력값 검증 실패
- `401`: 인증 실패
- `403`: 권한 없음
- `409`: 중복 리소스 충돌 가능
- `500`: 서버 내부 오류

왜 중요한가:

프론트는 상태 코드에 따라 다른 UX를 보여줄 수 있다.
예를 들어 `401`이면 로그인 유도, `403`이면 권한 안내가 가능하다.

면접 포인트:

"저는 인증 API에서 401과 403을 특히 구분합니다. 인증 자체가 실패한 것과, 인증은 됐지만 허용되지 않은 작업은 사용자 경험과 디버깅 관점에서 전혀 다른 상황이기 때문입니다."

## 면접처럼 정리하는 한 문단 요약

이 프로젝트의 인증은 단순 로그인 기능이 아니라 사용자와 리소스를 연결하는 기반이다.
그래서 Session과 JWT의 차이, 토큰 저장과 검증 방식, CORS와 HTTPS, 최소 보안 원칙, 상태 코드 설계까지 함께 봐야 한다.
실무적으로는 "JWT를 쓴다"보다 "왜 지금 구조에서는 JWT가 맞는지, 어떤 한계가 있고 어떻게 보완할지"를 설명할 수 있어야 한다.
