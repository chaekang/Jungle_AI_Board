# 006 태그 개념

## 문서 목적

좌석 후기 태그가 왜 N:M 관계인지, 검색과 필터를 위해 어떤 인덱스와 캐시 전략이 필요한지 이해한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- Nest 태그 도메인은 `apps/nest-api/src/tags` 또는 후기 도메인과 연결해 둔다.
- React 태그 선택 UI는 `apps/web-react/src/features/reviews`에 둔다.
- 태그 이름과 태그 타입은 UI 표시용 값과 DB 식별자를 구분한다.
- 후기와 태그 연결은 중간 테이블로 관리한다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `normalization`

#### 개념

정규화는 데이터 중복과 이상 현상을 줄이기 위해 테이블을 목적별로 나누는 원칙이다.

여기서 중요한 건 `normalization`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 태그 개념에서 극장 이름을 후기마다 복사하지 않고 Theater 테이블로 분리하는 이유가 된다.
그래서 문서를 읽을 때도 먼저 "중복 제거 이득과 JOIN 비용 사이에서 서비스 요구에 맞는 균형을 찾는다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

태그 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `normalization`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Functional Dependency, Duplication, Join를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/tags`이다.
그 다음으로 `apps/web-react/src/features/reviews`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `태그 이름과 태그 타입은 UI 표시용 값과 DB 식별자를 구분한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 극장명이 바뀌어도 Theater 한 행만 고치면 관련 후기가 같은 극장을 바라본다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `normalization`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// normalization를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `normalization`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Functional Dependency | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Duplication | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Join | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

정규화를 무조건 많이 하면 좋은 줄 알고 조회 성능과 개발 복잡도를 무시하기 쉽다.
또 하나의 실수는 `normalization`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 중복 제거 이득과 JOIN 비용 사이에서 서비스 요구에 맞는 균형을 찾는다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`normalization`는 태그 개념에서 극장 이름을 후기마다 복사하지 않고 Theater 테이블로 분리하는 이유가 된다.
예시는 극장명이 바뀌어도 Theater 한 행만 고치면 관련 후기가 같은 극장을 바라본다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 정규화를 무조건 많이 하면 좋은 줄 알고 조회 성능과 개발 복잡도를 무시하기 쉽다는 점이다.
Functional Dependency, Duplication, Join와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `normalization`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `Index`

#### 개념

인덱스는 테이블에서 특정 조건의 행을 빠르게 찾기 위해 따로 만든 탐색 구조다.

여기서 중요한 건 `Index`를 단어 뜻으로만 외우지 않는 것이다.

그래서 문서를 읽을 때도 먼저 "실제 많이 쓰는 검색 조건과 정렬 조건을 기준으로 인덱스를 설계한다"라는 질문을 붙여 두면 좋다.

### `Join`

#### 개념

JOIN은 관계형 데이터베이스에서 여러 테이블의 관련 행을 연결해 조회하는 연산이다.

여기서 중요한 건 `Join`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 태그 개념에서 후기를 가져올 때 작성자, 극장, 작품, 공연 정보를 함께 보여주기 위해 필요하다.
그래서 문서를 읽을 때도 먼저 "JOIN 기준 컬럼에 인덱스가 있고 결과 행 수가 의도와 맞는지 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Inner Join, Outer Join, Foreign Key를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 왜 필요한가

태그 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Redis`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Cache, TTL, Pub/Sub를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/tags`이다.
그 다음으로 `apps/web-react/src/features/reviews`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `태그 이름과 태그 타입은 UI 표시용 값과 DB 식별자를 구분한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 사용자별 로그인 실패 횟수를 Redis에 TTL과 함께 저장할 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Redis`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Redis를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Redis`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Cache | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| TTL | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Pub/Sub | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

Redis가 빠르다고 영구 보존이 필요한 핵심 데이터를 무조건 맡기면 위험하다.
또 하나의 실수는 `Redis`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 데이터가 사라져도 복구 가능한지, 만료 정책이 있는지 확인한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`Redis`는 태그 개념에서 rate limit 카운터, 세션 저장, 자주 쓰는 검색 결과 캐시에 활용할 수 있다.
예시는 사용자별 로그인 실패 횟수를 Redis에 TTL과 함께 저장할 수 있다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 Redis가 빠르다고 영구 보존이 필요한 핵심 데이터를 무조건 맡기면 위험하다는 점이다.
Cache, TTL, Pub/Sub와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Redis`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.
