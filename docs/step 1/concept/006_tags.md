# 006_tags_concepts

## 이 문서의 역할

이 문서는 `006 태그 기능` 단계에서 중요한 분류 / 검색 / 데이터 품질 개념을 정리한다.
태그는 단순 라벨처럼 보이지만, 실제로는 검색성과 분류 품질을 빠르게 끌어올리는 매우 실용적인 구조다.

좌석 후기 서비스에서 태그는 본문을 모두 읽지 않아도,
후기의 성격을 짧은 신호로 압축해서 보여주는 역할을 한다.
예를 들어 `시야좋음`, `표정위주`, `가성비`, `첫관람추천` 같은 태그는 본문을 읽기 전 필터 역할을 수행한다.

## 태그를 왜 별도 기능으로 보나

태그는 "본문의 보조 정보" 같지만, 실제로는 검색 UX와 데이터 구조 양쪽에 영향을 준다.

- 사용자는 태그로 빠르게 후기를 거를 수 있다
- 서버는 태그를 기반으로 목록을 묶을 수 있다
- AI는 태그를 약한 구조화 신호로 활용할 수 있다

즉 태그는 저비용 구조화 수단이다.

## 핵심 키워드

### `normalization`

정의:

정규화는 같은 의미의 정보를 중복 없이 일관되게 다루도록 구조화하는 원칙이다.

태그에서 왜 중요한가:

태그를 본문 문자열에만 남기면 검색과 통계가 어렵다.
반대로 `tags` 테이블로 분리하면 같은 의미를 같은 단위로 모을 수 있다.

예:

- `시야좋음`
- `시야 좋음`
- `시야-좋음`

처럼 비슷한 표현이 제각각 생기면 검색 품질이 떨어진다.
그래서 태그 구조는 정규화의 대표 사례가 된다.

면접 포인트:

"태그는 본문보다 가볍지만 검색과 분석 관점에서는 구조화된 데이터로 다루는 편이 좋습니다. 그래서 별도 테이블과 연결 구조를 두는 쪽을 선호합니다."

### `N:M`

정의:

다대다 관계는 양쪽 모두 여러 개가 연결될 수 있는 관계다.

태그에서의 적용:

- 후기 1개는 태그 여러 개를 가진다
- 태그 1개는 후기 여러 개에 붙는다

즉 태그는 대표적인 `N:M` 관계이며, 보통 조인 테이블이 필요하다.

어떻게 구현하는가:

- `tags`
- `seat_review_tags`

같은 구조로 구현한다.

장점:

- 유연하다
- 태그 재사용이 쉽다
- 필터와 집계가 가능하다

문제 / 주의점:

- 연결 중복을 막지 않으면 데이터 품질이 망가진다
- 태그 이름 규칙이 없으면 비슷한 태그가 난립한다

면접 포인트:

"태그는 본질적으로 다대다 관계이기 때문에, 조인 테이블 설계와 중복 방지 전략이 핵심입니다."

### `Index`

정의:

인덱스는 특정 컬럼을 빠르게 조회할 수 있도록 도와주는 구조다.

태그에서 왜 중요한가:

사용자가 "시야좋음 후기만 보기" 같은 요청을 자주 하면,
조인 테이블과 태그 테이블 조회 성능이 중요해진다.

인덱스 후보:

- `tags.name`
- `seat_review_tags.tag_id`
- `seat_review_tags.seat_review_id`

장점:

- 태그 기반 필터 성능 향상

문제 / 주의점:

- 쓰기 비용 증가
- 필요 없는 인덱스 남발 위험

면접 포인트:

"태그는 작은 기능처럼 보여도 필터 트래픽이 몰리면 성능 병목이 되기 쉽습니다. 그래서 조인 테이블 인덱스를 꼭 검토하는 편입니다."

### `Join`

정의:

조인은 관계형 테이블을 함께 읽는 방식이다.

태그에서의 적용:

- 후기 상세에 태그를 포함
- 특정 태그에 해당하는 후기 목록 조회

왜 중요한가:

태그를 분리 저장하면 결국 읽을 때 다시 조인해야 한다.
즉 분리 저장과 조회 복잡도는 세트로 이해해야 한다.

면접 포인트:

"태그 설계는 저장 구조만 보면 안 되고, 실제로 후기 목록과 상세에서 어떻게 다시 조합할지까지 같이 봐야 합니다."

### `caching`

정의:

자주 읽는 데이터를 메모리나 빠른 저장소에 임시 저장해 응답 속도를 높이는 전략이다.

태그에서 왜 유용한가:

- 인기 태그 목록
- 자주 쓰는 태그 추천
- 태그 자동완성 후보

같은 값은 읽기 비중이 높고 자주 바뀌지 않을 수 있다.

장점:

- 빠른 응답
- DB 부하 감소

문제 / 주의점:

- 캐시 무효화가 필요하다
- 실제 데이터와 캐시 데이터가 잠시 어긋날 수 있다

면접 포인트:

"태그는 읽기 비중이 높아서 캐시 친화적인 데이터입니다. 다만 캐시 적용은 태그 구조가 먼저 안정된 뒤에 하는 것이 맞다고 생각합니다."

### `Redis`

정의:

Redis는 메모리 기반의 빠른 저장소로, 캐시와 세션, 임시 데이터 저장에 자주 쓰인다.

태그 기능에서의 활용 예:

- 인기 태그 캐시
- 자동완성 후보 캐시
- 자주 조회되는 태그 목록 캐시

장점:

- 빠르다
- 단순 키-값 캐시에 적합하다

문제 / 주의점:

- 영속 DB를 대체하는 용도가 아니다
- 캐시 일관성 관리가 필요하다

면접 포인트:

"Redis는 태그 데이터를 저장하는 주 저장소가 아니라, 반복 조회되는 태그 정보를 가속하는 보조 계층으로 보는 편이 맞습니다."

## 면접처럼 정리하는 한 문단 요약

태그는 적은 비용으로 검색성과 분류 품질을 높여주는 구조화 장치다.
기술적으로는 `N:M` 관계와 조인 테이블, 중복 방지, 인덱스, 캐시 전략이 핵심이고,
실무적으로는 "태그를 자유롭게 많이 받는 것"보다 "일관된 태그 체계를 유지하는 것"이 더 중요하다.
