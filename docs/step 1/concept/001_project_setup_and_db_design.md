# 001 프로젝트 세팅과 DB 설계 개념

## 문서 목적

프로젝트를 처음 열었을 때 보이는 프론트엔드, 백엔드, 프레임워크, 데이터베이스, 언어 기초 키워드를 구현 순서와 연결해 이해한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- Nest 기능 코드는 `apps/nest-api/src/<domain>` 아래에 둔다.
- React 기능 코드는 `apps/web-react/src/features/<feature>` 아래에 둔다.
- 공통 HTTP 요청 함수는 `apps/web-react/src/shared/api.ts`에 둔다.
- 문서에서 코드 경로를 언급할 때는 실제 저장 위치와 역할을 함께 적는다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `Framework`

#### 개념

프레임워크는 애플리케이션을 만들 때 반복되는 구조, 흐름, 규칙을 미리 제공하는 뼈대다.

여기서 중요한 건 `Framework`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 프로젝트 세팅과 DB 설계 개념에서 NestJS와 React가 각각 서버와 화면 개발의 기본 흐름을 정해 주는 도구로 쓰인다.
그래서 문서를 읽을 때도 먼저 "새 파일을 만들 때 프레임워크가 기대하는 위치와 이름을 따르는지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

프로젝트 세팅과 DB 설계 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Framework`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Library, API, Architecture를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/<domain>`이다.
그 다음으로 `apps/web-react/src/features/<feature>`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/shared/api.ts`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 NestJS의 Controller, Service, Module 구조는 요청 처리 흐름을 프레임워크 규칙 안에 배치하게 만든다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Framework`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Framework를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Framework`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Library | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| API | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Architecture | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

라이브러리처럼 필요한 함수만 호출한다고 생각하면 제어 흐름을 왜 프레임워크가 가져가는지 헷갈린다.
또 하나의 실수는 `Framework`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 새 파일을 만들 때 프레임워크가 기대하는 위치와 이름을 따르는지 확인한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`Framework`는 프로젝트 세팅과 DB 설계 개념에서 NestJS와 React가 각각 서버와 화면 개발의 기본 흐름을 정해 주는 도구로 쓰인다.
예시는 NestJS의 Controller, Service, Module 구조는 요청 처리 흐름을 프레임워크 규칙 안에 배치하게 만든다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 라이브러리처럼 필요한 함수만 호출한다고 생각하면 제어 흐름을 왜 프레임워크가 가져가는지 헷갈린다는 점이다.
Library, API, Architecture와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Framework`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `Backend`

#### 개념

### `API`

#### 개념

그래서 문서를 읽을 때도 먼저 "요청 파라미터, 응답 형태, 실패 조건이 문서나 타입으로 드러나는지 확인한다"라는 질문을 붙여 두면 좋다.

### `Vue.js`

#### 개념

Vue.js는 선언형 템플릿과 반응형 상태를 중심으로 UI를 만드는 프론트엔드 프레임워크다.

여기서 중요한 건 `Vue.js`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 프로젝트 세팅과 DB 설계 개념에서 현재 프로젝트는 React를 쓰지만, Vue와 비교하면 상태와 화면 연결 방식을 이해하기 쉽다.
그래서 문서를 읽을 때도 먼저 "프레임워크가 달라도 화면을 상태의 결과로 본다는 원칙을 기억한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

프로젝트 세팅과 DB 설계 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Vue.js`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 React, Component, Reactive State를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/<domain>`이다.
그 다음으로 `apps/web-react/src/features/<feature>`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/shared/api.ts`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 Vue의 ref와 computed는 React의 useState와 useMemo를 비교해 볼 수 있는 지점이다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Vue.js`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Vue.js를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Vue.js`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| React | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Component | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Reactive State | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

React와 Vue를 문법 차이로만 보면 컴포넌트 기반 사고라는 공통점을 놓친다.
또 하나의 실수는 `Vue.js`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?

- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

### `JavaScript`

개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

### `props`

#### 개념

props는 부모 컴포넌트가 자식 컴포넌트에 내려주는 읽기 전용 입력값이다.

### `FastAPI`

#### 개념

여기서 중요한 건 `FastAPI`를 단어 뜻으로만 외우지 않는 것이다.

그래서 문서를 읽을 때도 먼저 "Nest와 FastAPI 중 어느 쪽이 팀의 언어, 배포, 테스트 흐름에 맞는지 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

프로젝트 세팅과 DB 설계 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `FastAPI`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.

#### 이 프로젝트에서 보는 위치

#### 동작 방식

2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.

}

### `PostgreSQL`

이런 도메인에서는 RDBMS, SQL, Prisma를 함께 놓고 차이를 보는 습관이 필요하다.

입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Foreign Key | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |

#### 흔한 실수

N쪽에 외래키를 두는 기본 원칙을 놓치면 관계 표현이 어색해진다.
또 하나의 실수는 `1:N`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.

#### 구현할 때 확인할 질문

- 삭제나 수정 시 자식 데이터가 어떻게 처리될지 미리 정한다?

### `N:M`

#### 개념

N:M 관계는 양쪽 모두 여러 행과 연결될 수 있는 관계다.

여기서 중요한 건 `N:M`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 프로젝트 세팅과 DB 설계 개념에서 한 후기에 여러 태그가 붙고, 한 태그도 여러 후기에 붙을 수 있다.
그래서 문서를 읽을 때도 먼저 "중간 테이블에 중복 방지 제약과 필요한 인덱스가 있는지 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

프로젝트 세팅과 DB 설계 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `N:M`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Join Table, Composite Unique, Normalization를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

그 다음으로 `apps/web-react/src/features/<feature>`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.

이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

### `Foreign Key`

#### 개념

외래키는 한 테이블의 값이 다른 테이블의 기본키를 참조하도록 보장하는 제약이다.

여기서 중요한 건 `Foreign Key`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 프로젝트 세팅과 DB 설계 개념에서 후기가 존재하는 사용자, 극장, 작품을 참조하게 만들어 데이터 정합성을 지킨다.
그래서 문서를 읽을 때도 먼저 "삭제 정책과 참조 무결성이 서비스 규칙과 충돌하지 않는지 확인한다"라는 질문을 붙여 두면 좋다.

### `Layered architecture`

#### 개념

계층형 아키텍처는 시스템을 표현, 비즈니스, 데이터 접근 같은 층으로 나누는 설계 방식이다.

여기서 중요한 건 `Layered architecture`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 프로젝트 세팅과 DB 설계 개념에서 Controller는 요청을 받고, Service는 규칙을 처리하고, Prisma는 DB와 연결된다.
그래서 문서를 읽을 때도 먼저 "상위 계층이 하위 계층의 세부 구현에 지나치게 의존하지 않는지 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

프로젝트 세팅과 DB 설계 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Layered architecture`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Controller, Service, Repository를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/<domain>`이다.
그 다음으로 `apps/web-react/src/features/<feature>`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/shared/api.ts`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 후기 삭제 권한 검사는 Service에 두고 Controller는 id와 사용자 정보를 전달한다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Layered architecture`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Layered architecture를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Layered architecture`를 읽을 때의 사고 순서다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Controller | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Service | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Repository | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 OOP, Spring Boot, Static Typing를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

}

### `functional programming`

예시는 후기 목록을 불러오는 동안 isLoading을 true로 두고 완료되면 화면을 갱신한다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 await를 빠뜨리면 Promise 자체를 데이터처럼 다루는 오류가 생긴다는 점이다.
Promise, Event Loop, Job Queue와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Async`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `Event loop`

#### 개념

이벤트 루프는 JavaScript가 호출 스택과 작업 큐를 오가며 비동기 콜백을 실행하는 구조다.

여기서 중요한 건 `Event loop`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 프로젝트 세팅과 DB 설계 개념에서 브라우저 이벤트, fetch 응답, setTimeout, React 렌더링 타이밍을 이해하는 배경이 된다.
그래서 문서를 읽을 때도 먼저 "동기 코드와 비동기 콜백의 실행 순서를 말로 설명해 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

프로젝트 세팅과 DB 설계 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Event loop`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Call Stack, Microtask, Macrotask를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/<domain>`이다.
그 다음으로 `apps/web-react/src/features/<feature>`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `apps/web-react/src/shared/api.ts`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 버튼 클릭 핸들러가 끝난 뒤 API 응답 콜백이 큐에서 실행된다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Event loop`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Event loop를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Event loop`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Call Stack | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Microtask | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Macrotask | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

비동기 코드가 작성 순서 그대로 즉시 실행된다고 생각하면 상태 업데이트 순서를 오해한다.
또 하나의 실수는 `Event loop`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 동기 코드와 비동기 콜백의 실행 순서를 말로 설명해 본다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`Event loop`는 프로젝트 세팅과 DB 설계 개념에서 브라우저 이벤트, fetch 응답, setTimeout, React 렌더링 타이밍을 이해하는 배경이 된다.
