# 001_project_setup_and_db_design_concepts

## 이 문서의 역할

이 문서는 `001 프로젝트 기본 세팅 / DB 설계` 단계에서 반드시 이해하고 넘어가야 하는 핵심 개념을 정리한 문서다.
단순히 키워드 뜻만 적는 문서가 아니라, 실제 프로젝트에서 언제 쓰는지, 왜 쓰는지, 어떤 장단점이 있는지, 면접에서 어떻게 설명하면 좋은지까지 포함한다.

이 단계는 "아직 기능을 만들기 전"처럼 보이지만, 사실 이후 모든 기능의 실패 원인을 미리 심는 단계이기도 하다.
프레임워크 선택이 흔들리면 폴더 구조가 흔들리고, DB 모델링이 약하면 검색과 RAG 품질이 흔들리고, 설정 관리가 약하면 로컬과 배포 환경이 계속 어긋난다.
그래서 이 문서는 초반 세팅 단계를 가볍게 넘기지 않도록 도와주는 참고서 역할을 한다.

## 이 단계에서 특히 중요한 이유

프로젝트 초반에는 "일단 돌아가게 만들자"는 유혹이 크다.
하지만 이 프로젝트는 단순 게시판이 아니라 다음을 모두 염두에 두고 있다.

- 인증
- 좌석 후기 CRUD
- 검색 / 필터
- 태그
- RAG
- MCP 연동
- Agent 추천

즉 처음부터 데이터 구조와 애플리케이션 구조가 어느 정도 확장 가능해야 한다.
너무 무거운 아키텍처를 초반에 도입할 필요는 없지만, 최소한 나중에 붙일 기능이 어디에서 깨질지 정도는 알고 설계해야 한다.

## 핵심 키워드

### `Framework`

정의:

프레임워크는 애플리케이션을 만들 때 반복적으로 필요한 구조와 실행 규칙을 제공하는 기반 도구다.
단순 라이브러리보다 제약이 강하고, 대신 생산성과 일관성을 얻기 쉽다.

언제 쓰는가:

- 서버 구조를 빠르게 세우고 싶을 때
- 폴더 구조와 실행 흐름을 팀 안에서 통일하고 싶을 때
- 인증, 설정, 라우팅, 의존성 주입 같은 반복 패턴을 표준화하고 싶을 때

어떻게 사용하는가:

현재 프로젝트에서는 백엔드에 `NestJS`, 프론트엔드에 `React`가 이미 존재한다.
즉 "무엇을 새로 도입할까"보다 "이미 깔린 프레임워크에 프로젝트를 얼마나 일관되게 맞출까"가 더 중요하다.

장점:

- 팀원 간 공통 문법이 생긴다
- 구조가 비슷해져서 유지보수가 쉬워진다
- 생태계 패키지와 예제가 풍부하다

문제 / 주의점:

- 러닝 커브가 있다
- 프레임워크 방식에 맞추다 보면 자유도가 줄어든다
- 작은 프로젝트에서는 과한 구조처럼 느껴질 수 있다

면접에서 말할 포인트:

"프레임워크는 단순 편의성 도구가 아니라 팀의 의사결정 비용을 줄이는 장치라고 봅니다. 특히 NestJS처럼 모듈 구조와 DI가 강한 프레임워크는 기능이 늘어날수록 장점이 커집니다."

### `Frontend`, `Backend`, `API`

정의:

- `Frontend`: 사용자가 직접 보는 화면과 상호작용을 담당하는 영역
- `Backend`: 데이터 저장, 인증, 비즈니스 규칙, 외부 연동을 담당하는 영역
- `API`: 프론트와 백엔드가 대화하는 계약

언제 중요한가:

이 구분은 프로젝트 시작부터 끝까지 중요하다.
특히 같은 기능이라도 "누가 책임질 것인가"를 정할 때 기준이 된다.

어떻게 적용하는가:

예를 들어 회원가입 기능을 보면:

- 프론트는 입력 폼과 에러 표시를 담당
- 백엔드는 이메일 중복 검사, 비밀번호 해시, 저장을 담당
- API는 어떤 요청을 보내고 어떤 응답을 받는지 정한다

장점:

- 책임이 분리되어 디버깅이 쉬워진다
- 프론트와 백엔드가 병렬 작업하기 좋다
- 특정 계층만 교체하거나 개선하기 쉬워진다

문제 / 주의점:

- 경계가 흐려지면 로직이 여기저기 섞인다
- API 계약이 느슨하면 프론트와 백엔드가 서로 다른 기대를 갖게 된다

면접 포인트:

"저는 기능을 구현할 때 항상 API를 경계로 책임을 나눕니다. 프론트는 사용자 경험, 백엔드는 데이터와 규칙, API는 그 사이의 계약이라는 관점으로 보면 설계가 많이 단순해집니다."

### `React`, `Vuejs`

정의:

둘 다 컴포넌트 기반 프론트엔드 프레임워크 / 라이브러리다.
UI를 작은 단위로 쪼개고 상태 변화에 따라 화면을 다시 그리는 방식이 핵심이다.

현재 프로젝트에서의 의미:

실제로는 둘 중 무엇이 더 좋은지를 논쟁하는 단계가 아니다.
이미 `apps/web-react`가 있기 때문에 React로 밀고 가는 것이 맞다.
즉 기술 선택보다 일관된 사용이 더 중요하다.

언제 쓰는가:

- 상태에 따라 화면이 자주 바뀌는 앱
- 컴포넌트 재사용이 중요한 앱
- 폼, 인증, 검색 UI처럼 상호작용이 많은 앱

어떻게 적용하는가:

이 프로젝트에서는 로그인 화면, 후기 작성 폼, 검색 패널, 후기 상세, Agent 질의 UI까지 대부분 React 컴포넌트 기반으로 구현할 수 있다.

장점:

- 컴포넌트 재사용이 쉽다
- 생태계가 풍부하다
- 상태 변화 기반 UI에 강하다

문제 / 주의점:

- 상태가 많아질수록 구조가 복잡해진다
- 전역 상태 관리, 라우팅, 데이터 fetching 전략을 별도로 고민해야 한다

면접 포인트:

"React를 쓸 때는 컴포넌트 재사용성보다 상태 경계를 먼저 봅니다. 상태를 어디에 둘지 잘못 정하면 나중에 UI보다 데이터 흐름이 더 복잡해집니다."

### `HTML/CSS/Javascript/TypeScript`, `state / props`

정의:

- `HTML`: 문서 구조
- `CSS`: 시각적 표현
- `Javascript`: 동작
- `TypeScript`: 타입 시스템이 추가된 JavaScript
- `state`: 컴포넌트가 내부적으로 기억하는 값
- `props`: 부모가 자식에게 전달하는 값

언제 중요한가:

프론트엔드는 결국 이 기본기를 조합하는 작업이다.
프레임워크가 있어도 이 개념이 약하면 디버깅이 어렵다.

어떻게 적용하는가:

- 로그인 입력값은 `state`
- 상위 레이아웃이 하위 컴포넌트에 내려주는 데이터는 `props`
- API 응답 타입은 `TypeScript` 타입으로 모델링

장점:

- 타입을 붙이면 API 응답과 화면 모델이 맞는지 빨리 확인할 수 있다
- state / props 경계를 명확히 하면 컴포넌트 책임이 분명해진다

문제 / 주의점:

- 모든 상태를 전역으로 올리면 오히려 복잡해진다
- 타입을 너무 세세하게 만들면 초기 개발 속도가 느려질 수 있다

면접 포인트:

"TypeScript를 쓰는 가장 큰 이유는 런타임 버그를 줄이기보다, 프론트와 API 사이의 데이터 계약을 빨리 어긋남 없이 맞추기 위해서라고 생각합니다."

### `NestJS`, `FastAPI`, `Spring Boot`

정의:

각각 대표적인 서버 프레임워크다.

- `NestJS`: TypeScript 기반, 모듈성과 DI가 강점
- `FastAPI`: Python 기반, 타입 힌트와 문서화가 강점
- `Spring Boot`: Java 기반, 기업 환경과 확장성이 강점

언제 비교하는가:

- 팀의 주력 언어가 무엇인지
- 운영 경험이 어느 스택에 있는지
- 인증, DB, 테스트, 배포 생태계를 어느 쪽에서 더 빠르게 구성할 수 있는지

현재 프로젝트에서는:

이미 Nest 기반 구조가 있으므로 Nest를 메인 API로 잡는 것이 가장 안전하다.
초기 프로젝트에서는 "좋은 프레임워크 선택"보다 "하나로 집중"이 더 중요하다.

장점과 주의점:

- NestJS는 구조가 잘 잡히지만 작은 프로젝트에서는 무겁게 느껴질 수 있다
- FastAPI는 빠르게 시작하기 쉽지만, 현재 레포와 주 언어 통일 관점에서는 맞지 않는다
- Spring Boot는 강력하지만 현재 레포 맥락에서는 도입 비용이 크다

면접 포인트:

"기술 선택은 절대 성능보다 팀의 일관성과 운영 난이도까지 같이 봐야 합니다. 이 프로젝트에서는 TypeScript를 프론트와 백엔드에 같이 쓰는 편익이 큽니다."

### `RDBMS`, `PostgreSQL`, `MySQL`, `SQL`

정의:

- `RDBMS`: 관계를 중심으로 데이터를 저장하는 DB
- `PostgreSQL`, `MySQL`: 대표적인 RDBMS 구현체
- `SQL`: 데이터를 조회, 저장, 수정, 삭제하는 언어

언제 쓰는가:

- 데이터 관계가 명확할 때
- 정합성이 중요할 때
- 조인과 필터가 많은 서비스일 때

왜 이 프로젝트와 잘 맞는가:

이 프로젝트는 후기, 댓글, 사용자, 태그, 극장, 작품, 공연처럼 관계가 분명한 데이터가 많다.
또 검색 / 필터 / 정렬이 중요하므로 RDBMS가 자연스럽다.

PostgreSQL의 장점:

- 기능이 풍부하다
- JSON, Full Text Search, 확장 기능이 강하다
- 추후 `pgvector` 연동도 고려 가능하다

문제 / 주의점:

- 스키마 설계를 잘못하면 마이그레이션 비용이 커진다
- 인덱스를 무턱대고 늘리면 쓰기 성능이 떨어질 수 있다

면접 포인트:

"관계형 DB를 고른 이유는 데이터 정합성과 검색 요구 때문입니다. 특히 이 프로젝트는 후기와 좌석 메타데이터의 조합이 중요해서 조인과 인덱스 설계가 핵심입니다."

### `data layer`

정의:

데이터 레이어는 애플리케이션 코드와 DB 사이를 연결하는 계층이다.
SQL을 직접 쓰든 ORM을 쓰든, 결국 "데이터를 어떻게 읽고 쓰는가"를 관리하는 영역이다.

언제 중요한가:

- 쿼리 로직이 서비스 코드에 흩어지는 것을 막고 싶을 때
- 테스트에서 DB 의존성을 분리하고 싶을 때
- 트랜잭션과 조회 최적화를 일관되게 다루고 싶을 때

현재 프로젝트에서는:

Prisma가 핵심 데이터 레이어 역할을 한다.
Nest의 Service 계층과 Prisma Client를 조합하면 컨트롤러가 DB를 직접 모르는 구조를 만들 수 있다.

장점:

- 역할이 분리된다
- 쿼리 관련 수정이 한곳에 모인다
- 테스트와 리팩터링이 쉬워진다

문제 / 주의점:

- ORM에만 의존하면 실제 SQL 비용을 놓치기 쉽다
- 데이터 레이어가 비대해지면 서비스 계층과 경계가 다시 흐려질 수 있다

면접 포인트:

"데이터 레이어를 따로 두는 이유는 재사용성보다 쿼리 책임 분리에 있습니다. 특히 나중에 검색 최적화나 트랜잭션 경계를 조정할 때 큰 도움이 됩니다."

### `data modeling`, `ERD (1:1, 1:N, N:M)`, `normalization`

정의:

- `data modeling`: 어떤 테이블과 관계가 필요한지 설계하는 과정
- `ERD`: 그 관계를 시각화한 문서
- `normalization`: 중복을 줄이고 정합성을 높이는 구조화 원칙

언제 중요한가:

프로젝트 초반부터 끝까지 중요하지만, 특히 확장 기능이 많을수록 초반 모델링이 중요하다.

어떻게 적용하는가:

- 후기와 댓글은 `1:N`
- 후기와 태그는 `N:M`
- 공연과 극장은 `N:1`

정규화를 하는 이유:

- 같은 극장명을 후기마다 문자열로 복붙하지 않기 위해
- 작품 정보 변경 시 한 곳만 수정하게 하기 위해
- 검색과 통계를 정확하게 하기 위해

문제 / 주의점:

- 과도한 정규화는 조회 쿼리를 복잡하게 만든다
- 반대로 비정규화가 과하면 데이터 불일치가 늘어난다

면접 포인트:

"정규화는 무조건 많이 하는 것이 아니라, 쓰기 정합성과 읽기 복잡도의 균형을 맞추는 작업이라고 생각합니다."

### `Primary Key, Foreign Key, Index`

정의:

- `Primary Key`: 각 레코드를 고유하게 식별
- `Foreign Key`: 다른 테이블의 레코드와 연결
- `Index`: 검색 성능 향상을 위한 자료구조

언제 중요한가:

항상 중요하지만, 검색 기능이 많은 프로젝트일수록 인덱스는 체감 차이가 크다.

어떻게 적용하는가:

- `users.id`, `seat_reviews.id`는 PK
- `seat_reviews.author_id`, `seat_reviews.theater_id`는 FK
- `created_at`, `theater_id`, `musical_id`, `seat_floor` 등은 인덱스 후보

장점:

- PK / FK는 데이터 정합성을 보장한다
- 인덱스는 조회 성능을 높인다

문제 / 주의점:

- FK가 많다고 무조건 느려지는 것은 아니지만, 쓰기 비용은 늘 수 있다
- 인덱스를 너무 많이 두면 insert / update 성능이 나빠질 수 있다

면접 포인트:

"인덱스는 읽기 성능을 올리지만 쓰기 비용을 동반합니다. 그래서 실제 검색 조건과 정렬 조건을 기준으로 후보를 고르는 편이 좋습니다."

### `Layered architecture`

정의:

표현 계층, 서비스 계층, 데이터 계층처럼 역할별로 구조를 나누는 방식이다.

언제 쓰는가:

- 비즈니스 로직이 점점 많아질 때
- 테스트와 유지보수를 쉽게 하고 싶을 때
- 프레임워크 코드와 도메인 로직을 분리하고 싶을 때

현재 프로젝트의 예:

- Controller: 요청 진입점
- Service: 비즈니스 규칙
- Prisma / Repository 역할: 데이터 접근

장점:

- 역할이 명확하다
- 코드 위치를 찾기 쉽다
- 리팩터링과 테스트가 쉬워진다

문제 / 주의점:

- 무조건 계층만 나눈다고 좋은 구조가 되진 않는다
- 모든 로직이 서비스 하나에 몰리면 "계층은 있는데 응집도는 낮은" 상태가 된다

면접 포인트:

"레이어드 아키텍처는 시작하기 쉬운 구조이지만, 서비스 계층이 비대해지는 순간부터 모듈 경계와 도메인 경계를 같이 봐야 합니다."

### `configuration`

정의:

포트, DB URL, 비밀키처럼 환경마다 달라지는 값을 코드 밖으로 분리해 관리하는 방식이다.

언제 중요한가:

- 로컬 / 테스트 / 배포 환경이 나뉠 때
- 민감 정보를 소스코드에 넣지 않아야 할 때

어떻게 적용하는가:

- `DATABASE_URL`
- `PORT`
- `JWT_SECRET`

같은 값을 `.env` 또는 설정 모듈로 분리한다.

장점:

- 환경별 차이를 쉽게 관리할 수 있다
- 민감 정보를 코드에서 분리할 수 있다

문제 / 주의점:

- 환경변수 이름이 제각각이면 오히려 혼란이 커진다
- 로컬에서만 동작하는 "숨은 설정값"이 생기지 않게 주의해야 한다

면접 포인트:

"설정 관리는 단순 편의성이 아니라 운영 안정성 문제입니다. 특히 비밀키와 외부 서비스 연결 정보는 코드와 수명 주기가 다르기 때문에 분리해야 합니다."

### `language basics`, `TypeScript`, `Java`, `python`, `OOP`, `functional programming`, `Async`, `Event loop`, `paradigm`

정의:

이 키워드들은 구현 전에 필요한 프로그래밍 기본기와 사고방식을 나타낸다.

현재 프로젝트에서 실제로 중요한 것:

- `TypeScript`: 주력 언어
- `OOP`: Nest 구조 이해에 도움
- `Async`, `Event loop`: API / DB / 외부 호출 비동기 흐름 이해
- `functional programming`: 작은 변환 함수, 순수 함수 작성 습관

언제 중요한가:

초반에는 프레임워크 문법보다 언어 기본기가 더 많은 문제를 만든다.
예를 들어 `await`를 빼먹거나, 타입을 대충 잡거나, 데이터 변환 책임을 섞어 두면 이후 디버깅이 어려워진다.

면접 포인트:

"저는 프레임워크 지식보다 언어 기본기가 더 오래 간다고 생각합니다. 특히 TypeScript에서는 타입 모델링과 비동기 흐름 이해가 생산성과 안정성에 직접 연결됩니다."

## 면접처럼 정리하는 한 문단 요약

이 단계는 단순 세팅 단계가 아니라, 앞으로 만들 인증, CRUD, 검색, RAG, Agent가 기대고 설 기반을 만드는 단계다.
그래서 프레임워크 선택은 일관성 관점에서, DB 선택은 관계와 검색 요구 관점에서, 데이터 모델링은 확장성과 정합성 관점에서 봐야 한다.
실무적으로는 "빨리 시작하는 것"보다 "나중에 흔들리지 않는 기본 구조를 만드는 것"이 더 중요하다.
