# 010_mcp_integration_concepts

## 이 문서의 역할

이 문서는 `010 MCP 연동` 단계에서 필요한 외부 연동, 어댑터 설계, 장애 격리 개념을 정리한다.
MCP는 이 프로젝트 바깥의 데이터를 끌어오는 확장 계층으로 이해하면 된다.

중요한 점은 MCP가 "기능의 중심"이 아니라 "기존 서비스의 정확도와 편의성을 높이는 보조 계층"이라는 것이다.
즉 외부 연동은 붙이면 좋아지지만, 붙지 않아도 핵심 서비스가 살아 있어야 한다.

## 왜 외부 연동은 나중에 붙이나

외부 데이터 연동을 너무 빨리 붙이면 내부 데이터 구조가 외부 제공자 형식에 끌려가기 쉽다.
그래서 먼저 내부 모델과 후기 작성 흐름을 안정화하고,
그 후에 외부 데이터를 매핑하는 편이 혼란이 적다.

예를 들어 극장 정보나 좌석 배치도를 가져오더라도,
내부 `theaters` 기준이 먼저 있어야 외부 값이 들어올 자리가 명확해진다.

## 핵심 키워드

### `api design`

정의:

외부 연동 API 설계는 외부 응답을 그대로 노출하지 않고, 내부 서비스가 쓰기 좋은 형태로 가공하는 계약을 설계하는 일이다.

언제 중요한가:

- 외부 응답이 불안정할 때
- 공급자가 바뀔 가능성이 있을 때
- 프론트가 내부 모델만 알게 하고 싶을 때

좋은 설계 방향:

- 외부 응답을 내부 DTO로 변환
- 외부 오류를 내부 오류 정책에 맞게 재해석
- 필요한 값만 추출

면접 포인트:

"외부 연동에서는 외부 API 형식을 내부 도메인 모델과 분리하는 것이 중요합니다. 그래야 제공자 변경이나 장애에 덜 끌려갑니다."

### `security`

정의:

외부 연동 보안은 인증 키, 접근 범위, 데이터 검증, 비밀 관리까지 포함한다.

언제 중요한가:

- API Key를 써야 할 때
- 외부 데이터 신뢰도가 완전하지 않을 때
- 관리자 기능과 연결될 때

문제 / 주의점:

- 외부 응답을 그대로 신뢰하면 안 된다
- 비밀키를 코드에 넣으면 안 된다
- 접근 권한이 필요한 도구라면 사용자별 권한도 고려해야 한다

면접 포인트:

"외부 연동 보안은 우리 시스템을 지키는 문제이기도 하지만, 외부 시스템을 어떻게 신뢰할지 정하는 문제이기도 하다고 생각합니다."

### `Rate limit`

정의:

외부 API는 호출 횟수 제한이 있는 경우가 많다.

왜 중요한가:

- 좌석 배치도를 매 요청마다 새로 가져오면 제한에 걸릴 수 있다
- 운영 비용이 커질 수 있다

대응 전략:

- 캐시
- 수동 갱신
- 주기적 동기화
- 배치 처리

면접 포인트:

"외부 API는 우리 서버처럼 무제한이 아니기 때문에, 매 요청 실시간 조회보다 캐시와 동기화 전략을 같이 설계하는 편이 현실적입니다."

### `caching`

정의:

외부에서 가져온 자주 안 바뀌는 데이터를 일정 시간 저장해 재사용하는 전략이다.

언제 쓰는가:

- 극장 메타데이터
- 층 / 구역 정보
- 좌석 배치도 URL 또는 요약 정보

장점:

- 응답 속도 향상
- 외부 API 비용 / 제한 완화

문제 / 주의점:

- 갱신 시점이 필요하다
- 최신성과 성능 사이 균형이 필요하다

면접 포인트:

"외부 연동 데이터는 완전 실시간보다 적절한 TTL과 수동 갱신 전략이 더 실용적인 경우가 많습니다."

### `Reliability`

정의:

외부 연동에서도 서비스가 예측 가능하게 동작하는 성질이다.

왜 중요한가:

외부 API는 언젠가 실패한다.
중요한 것은 실패 여부보다, 실패해도 우리 서비스 핵심 흐름이 유지되는가다.

좋은 기준:

- 외부 좌석 배치도 조회 실패
- 그래도 후기 작성 기본 흐름은 가능

처럼 fallback이 있어야 한다.

면접 포인트:

"외부 연동은 성공 경로보다 실패 경로 설계가 더 중요합니다. 핵심 기능이 외부 장애에 같이 죽지 않게 하는 것이 신뢰성의 핵심이라고 생각합니다."

### `clean architecture`, `hexagonal architecture`

정의:

둘 다 외부 의존성을 핵심 비즈니스 로직에서 분리하려는 사고방식이다.

- `clean architecture`: 안쪽 도메인이 바깥 구현 세부사항에 의존하지 않음
- `hexagonal architecture`: 포트와 어댑터를 통해 외부와 연결

MCP 연동에서 왜 유용한가:

외부 극장 정보 제공자가 바뀌어도,
내부 서비스는 "극장 정보 조회 포트"만 알면 되게 만들 수 있다.

장점:

- 제공자 교체가 쉽다
- 테스트가 쉬워진다
- 외부 장애 영향 범위를 줄일 수 있다

문제 / 주의점:

- 작은 프로젝트 초반에는 구조가 과해질 수 있다
- 추상화만 많고 실제 이점이 없는 경우도 있다

면접 포인트:

"외부 연동은 도메인보다 변동 가능성이 높기 때문에, 어댑터 계층으로 감싸는 것이 구조적으로 유리합니다."

### `event-driven architecture`

정의:

이벤트가 발생했을 때 이를 구독하는 여러 처리기가 반응하는 구조다.

언제 쓰는가:

- 외부 메타데이터 갱신 후 재색인
- 극장 정보 갱신 후 캐시 무효화
- 좌석 배치도 갱신 후 관리자 알림

장점:

- 결합도를 낮출 수 있다
- 후속 작업을 확장하기 쉽다

문제 / 주의점:

- 흐름 추적이 어려워질 수 있다
- 결국 일관성과 재처리 전략이 필요하다

면접 포인트:

"외부 연동 후속 작업이 많아질수록 이벤트 기반 구조가 유리하지만, 초반에는 너무 복잡하게 가지 않도록 경계를 잘 잡는 것이 중요합니다."

## 면접처럼 정리하는 한 문단 요약

MCP 연동은 외부 데이터를 그대로 붙이는 작업이 아니라, 내부 모델에 맞게 안전하게 흡수하는 작업이다.
그래서 API 설계, 보안, rate limit, 캐시, fallback, 어댑터 구조를 함께 봐야 하며,
실무적으로는 "연동 성공"보다 "외부 장애가 나도 핵심 서비스가 유지되도록 설계하는 것"이 더 중요하다.
