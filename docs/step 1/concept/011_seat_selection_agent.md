# 011_seat_selection_agent_concepts

## 이 문서의 역할

이 문서는 `011 좌석 선택 도우미 Agent` 단계와 연결되는 아키텍처 / 오케스트레이션 개념을 정리한다.
Agent는 단순 챗봇이 아니라, 검색, RAG, 메타데이터, 추천 로직을 조합해 사용자 질문에 답하는 상위 계층이다.

이 단계에서 중요한 것은 "모델이 말을 잘하느냐"보다
"어떤 하위 기능을 어떤 순서로 호출하고, 어떤 근거로 추천을 만들 것인가"를 설계하는 것이다.
즉 Agent는 프롬프트 기능보다 시스템 설계 기능에 더 가깝다.

## 왜 Agent는 가장 마지막 단계인가

Agent는 기존 기능이 없으면 존재할 수 없다.

- 검색이 없으면 후보를 찾기 어렵고
- RAG가 없으면 근거를 만들기 어렵고
- 메타데이터가 없으면 좌석 방향성과 극장 구조를 설명하기 어렵다

그래서 Agent는 항상 가장 마지막에 조합 계층으로 두는 편이 안전하다.

## 핵심 키워드

### `architecture`

정의:

여기서 아키텍처는 컴포넌트를 어떻게 나누고 연결할지에 대한 상위 구조를 뜻한다.

Agent 단계에서 왜 중요한가:

사용자 질문 하나가 여러 하위 시스템을 거칠 수 있다.

- 의도 분석
- 조건 추출
- 검색
- RAG
- 외부 메타데이터 확인
- 답변 생성

즉 흐름 오케스트레이션 자체가 아키텍처 문제다.

면접 포인트:

"Agent는 모델 호출 하나가 아니라 여러 하위 기능을 조합하는 상위 계층이라고 생각합니다. 그래서 아키텍처 관점이 중요합니다."

### `Modular monolith`

정의:

배포는 하나의 애플리케이션으로 하되, 내부 모듈 경계를 명확히 나누는 구조다.

왜 현재 프로젝트에 잘 맞는가:

아직 서비스 규모가 아주 크지 않고, 검색 / RAG / 연동 / 추천을 각각 독립 배포할 정도는 아닐 수 있다.
이때는 모듈형 모놀리식이 좋은 타협점이 된다.

장점:

- 배포가 단순하다
- 내부 모듈 분리는 유지할 수 있다
- 과도한 분산 시스템 복잡도를 피할 수 있다

문제 / 주의점:

- 모듈 경계가 약하면 결국 거대한 한 덩어리가 된다
- 성능 병목이 한 애플리케이션 안에 몰릴 수 있다

면접 포인트:

"현재 단계에서는 모듈형 모놀리식이 가장 현실적이라고 생각합니다. 도메인 경계는 유지하면서도 운영 복잡도는 낮출 수 있기 때문입니다."

### `Microservices`

정의:

기능 단위를 독립 서비스로 나누고 각기 배포 / 확장하는 구조다.

언제 고려하는가:

- 팀이 커졌을 때
- 기능별 트래픽 특성이 크게 다를 때
- 독립 배포 요구가 강할 때

현재 프로젝트에서 왜 바로 가지 않는가:

- 운영 복잡도가 높다
- 네트워크 통신, 배포, 모니터링, 장애 대응 비용이 커진다
- 지금은 내부 모듈 분리만으로도 충분할 가능성이 크다

면접 포인트:

"마이크로서비스는 구조적 정답이 아니라 조직과 운영 상황에 맞는 선택이라고 봅니다. 현재 단계에서는 오히려 과할 수 있습니다."

### `CQRS`

정의:

명령(Command)과 조회(Query)의 모델과 경로를 분리하는 패턴이다.

왜 Agent 단계에서 언급되는가:

후기 작성과 추천용 조회는 요구사항이 다르다.

- 쓰기 모델: 정확한 저장과 검증
- 읽기 모델: 빠른 검색과 추천용 집계

즉 Agent는 조회 모델 최적화 필요성과 잘 연결된다.

장점:

- 읽기 최적화가 쉬워진다
- 복잡한 조회 요구를 분리할 수 있다

문제 / 주의점:

- 구조가 복잡해진다
- eventual consistency를 받아들여야 할 수 있다

면접 포인트:

"완전한 CQRS를 바로 도입하기보다, 추천과 검색처럼 읽기 요구가 강한 영역을 별도 조회 모델로 분리하는 사고가 실용적이라고 생각합니다."

### `event-driven architecture`

정의:

사건 발생을 중심으로 여러 처리 흐름을 연결하는 구조다.

Agent와 왜 관련 있는가:

Agent는 이미 만들어진 데이터와 색인을 소비한다.
즉 후기 생성 이벤트가 발생하면:

- 검색 인덱스 갱신
- 임베딩 생성
- 추천 후보 업데이트

같은 후속 흐름이 이어질 수 있다.

면접 포인트:

"Agent는 독립 기능이라기보다 이전 이벤트 흐름의 산출물을 소비하는 계층이라고 볼 수 있습니다."

### `Scalability`

정의:

질문 수와 데이터가 늘어나도 서비스가 감당 가능한 구조인지 보는 관점이다.

Agent에서 왜 중요해지는가:

추천 기능은 일반 CRUD보다 계산 비용이 높다.
검색 + RAG + 외부 조회 + 응답 생성이 한 요청에 몰릴 수 있기 때문이다.

대응 전략:

- 캐시
- 중간 결과 재사용
- 비동기 전처리
- 추천 범위 제한

면접 포인트:

"Agent는 응답 품질뿐 아니라 비용과 지연 시간을 같이 봐야 합니다. 그래서 확장성 관점에서는 사전 계산과 캐시 전략이 중요합니다."

### `Async`

정의:

여러 시간이 걸리는 작업을 순차 / 병렬로 조합하는 방식이다.

Agent에서 왜 중요한가:

- 검색
- 외부 메타데이터 조회
- LLM 호출

중 일부는 병렬화할 수 있고, 일부는 순서를 지켜야 한다.

즉 어디를 병렬화하고 어디를 직렬화할지 감각이 필요하다.

면접 포인트:

"Agent 오케스트레이션에서는 비동기 흐름 제어가 중요합니다. 모든 작업을 병렬화하면 안 되고, 의존 관계에 따라 순서를 설계해야 합니다."

### `ADR (Architecture Decision Record)`

정의:

아키텍처 결정 이유와 대안을 짧게 기록하는 문서다.

언제 유용한가:

Agent 단계처럼 선택지가 많아지는 시점에 특히 유용하다.

예:

- 지금은 Modular monolith 유지
- Agent는 기존 Nest 앱 내부 모듈로 구현
- 추천 후보는 검색 + RAG 조합으로 생성

장점:

- 나중에 왜 그렇게 결정했는지 설명 가능
- 팀 내 합의와 회고에 도움

문제 / 주의점:

- 너무 형식적으로만 쓰면 살아 있는 문서가 되지 않는다

면접 포인트:

"ADR은 복잡한 구조 선택에서 특히 유용합니다. 선택 자체보다 선택 이유를 남기는 것이 장기 유지보수에 도움이 됩니다."

## 면접처럼 정리하는 한 문단 요약

Agent는 단순 LLM 기능이 아니라 검색, 메타데이터, RAG, 추천 로직을 오케스트레이션하는 상위 계층이다.
그래서 Modular monolith와 Microservices의 선택, CQRS적 조회 분리, 이벤트 기반 갱신, 비동기 흐름, 확장성, ADR까지 함께 봐야 한다.
실무적으로는 "똑똑한 답변"보다 "근거 있고 예측 가능한 추천 흐름"을 만드는 것이 더 중요하다.
