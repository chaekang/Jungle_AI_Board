# 011 좌석 선택 에이전트 개념

## 문서 목적

AI 좌석 추천 기능을 만들 때 모놀리스, 마이크로서비스, CQRS, 이벤트 기반 구조, 확장성, ADR을 어떤 기준으로 선택하는지 정리한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- 좌석 추천 도메인은 후기 데이터, 좌석 메타데이터, 사용자 요구조건을 함께 읽는다.
- 초기에는 모듈러 모놀리스로 시작하고, 병목이 분명해질 때 서비스를 나누는 편이 안전하다.
- 추천 결과는 설명 가능한 근거와 함께 저장하거나 응답한다.
- 중요한 아키텍처 선택은 ADR로 남긴다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `architecture`

#### 개념

아키텍처는 시스템의 큰 구조, 책임 분리, 의존 관계, 운영 방식을 결정하는 설계다.

여기서 중요한 건 `architecture`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 좌석 선택 에이전트 개념에서 React, Nest, DB, AI 추천, 외부 도구가 어떻게 연결되는지 정한다.
그래서 문서를 읽을 때도 먼저 "중요한 변경이 생겼을 때 어디를 고쳐야 하는지 예측 가능한지 본다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

좌석 선택 에이전트 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `architecture`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Module, Boundary, Dependency를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `좌석 추천 도메인은 후기 데이터, 좌석 메타데이터, 사용자 요구조건을 함께 읽는다.`이다.
그 다음으로 `초기에는 모듈러 모놀리스로 시작하고, 병목이 분명해질 때 서비스를 나누는 편이 안전하다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `추천 결과는 설명 가능한 근거와 함께 저장하거나 응답한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 후기 서비스와 추천 에이전트를 같은 서버에 둘지 별도 서비스로 둘지 결정한다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `architecture`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// architecture를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `architecture`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Module | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Boundary | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Dependency | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

폴더 구조만 아키텍처라고 생각하면 데이터 흐름과 장애 대응을 놓친다.
또 하나의 실수는 `architecture`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 중요한 변경이 생겼을 때 어디를 고쳐야 하는지 예측 가능한지 본다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`architecture`는 좌석 선택 에이전트 개념에서 React, Nest, DB, AI 추천, 외부 도구가 어떻게 연결되는지 정한다.
예시는 후기 서비스와 추천 에이전트를 같은 서버에 둘지 별도 서비스로 둘지 결정한다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 폴더 구조만 아키텍처라고 생각하면 데이터 흐름과 장애 대응을 놓친다는 점이다.
Module, Boundary, Dependency와 비교하면 책임 범위가 더 선명해진다.

### `Microservices`

#### 개념

마이크로서비스는 시스템을 독립 배포 가능한 작은 서비스들로 나누는 아키텍처 방식이다.

여기서 중요한 건 `Microservices`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 좌석 선택 에이전트 개념에서 AI 추천, 검색 인덱싱, 메인 API가 규모와 기술 요구가 달라질 때 고려할 수 있다.
그래서 문서를 읽을 때도 먼저 "독립 배포 이득이 서비스 간 통신 비용보다 큰지 판단한다"라는 질문을 붙여 두면 좋다.

좌석 선택 에이전트 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Microservices`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Distributed System, gRPC, Kafka를 함께 놓고 차이를 보는 습관이 필요하다.

### `CQRS`

#### 개념

CQRS는 명령(Command)과 조회(Query)의 모델과 책임을 분리하는 패턴이다.

여기서 중요한 건 `CQRS`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 좌석 선택 에이전트 개념에서 후기 작성/수정과 좌석별 빠른 조회 화면의 요구가 달라질 때 적용 후보가 된다.
그래서 문서를 읽을 때도 먼저 "읽기 성능 요구가 일반 CRUD 구조로 감당하기 어려운지 먼저 본다"라는 질문을 붙여 두면 좋다.

그 사이에서 `CQRS`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.

프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.

#### 구현할 때 확인할 질문

- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

마지막으로 `event-driven architecture`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `Scalability`

#### 개념

확장성은 데이터, 사용자, 요청량이 늘어도 시스템을 감당 가능하게 키울 수 있는 성질이다.

여기서 중요한 건 `Scalability`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 좌석 선택 에이전트 개념에서 후기와 좌석 데이터가 많아질 때 검색, 지도 표시, 추천 기능이 느려지지 않아야 한다.
그래서 문서를 읽을 때도 먼저 "데이터가 10배, 100배 늘었을 때 가장 먼저 느려질 지점을 상상한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

좌석 선택 에이전트 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Scalability`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Caching, Partition, Sharding를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `좌석 추천 도메인은 후기 데이터, 좌석 메타데이터, 사용자 요구조건을 함께 읽는다.`이다.
그 다음으로 `초기에는 모듈러 모놀리스로 시작하고, 병목이 분명해질 때 서비스를 나누는 편이 안전하다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `추천 결과는 설명 가능한 근거와 함께 저장하거나 응답한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 페이지네이션과 인덱스를 적용하면 전체 후기를 한 번에 내려보내지 않아도 된다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Scalability`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts

const result = await apiCall(requestBody);

  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Caching | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Partition | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Sharding | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Context, Decision, Consequence를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `좌석 추천 도메인은 후기 데이터, 좌석 메타데이터, 사용자 요구조건을 함께 읽는다.`이다.
그 다음으로 `초기에는 모듈러 모놀리스로 시작하고, 병목이 분명해질 때 서비스를 나누는 편이 안전하다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `추천 결과는 설명 가능한 근거와 함께 저장하거나 응답한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 왜 좌석 지도 데이터를 프론트 config로 들고 있는지 ADR로 남길 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `ADR (Architecture Decision Record)`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// ADR (Architecture Decision Record)를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `ADR (Architecture Decision Record)`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Context | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Decision | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Consequence | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

결과만 적고 왜 그런 선택을 했는지 빠뜨리면 나중에 되돌아보기 어렵다.
또 하나의 실수는 `ADR (Architecture Decision Record)`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 미래 팀원이 결정을 바꿔도 되는 조건을 이해할 수 있는지 본다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`ADR (Architecture Decision Record)`는 좌석 선택 에이전트 개념에서 JWT 선택, 좌석배치도 데이터 출처, AI 추천 구조 같은 결정을 기록한다.
예시는 왜 좌석 지도 데이터를 프론트 config로 들고 있는지 ADR로 남길 수 있다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 결과만 적고 왜 그런 선택을 했는지 빠뜨리면 나중에 되돌아보기 어렵다는 점이다.
Context, Decision, Consequence와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `ADR (Architecture Decision Record)`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.
