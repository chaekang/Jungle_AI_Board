# 008_core_feature_integration_concepts

## 이 문서의 역할

이 문서는 `008 기본 기능 통합 점검` 단계에서 필요한 품질 보증 개념을 정리한다.
이 단계는 기능을 새로 만드는 단계가 아니라, 지금까지 만든 기능이 서로 충돌 없이 맞물리는지 확인하는 단계다.

작은 프로젝트에서는 이 단계를 자주 건너뛰지만,
실제로는 이 단계가 있어야 이후 RAG, MCP, Agent 같은 상위 기능의 문제를 기초 기능과 분리해서 볼 수 있다.
즉 통합 점검은 "마지막 확인"이 아니라 "다음 단계 디버깅 비용을 줄이는 투자"다.

## 왜 통합 점검이 중요한가

예를 들어 다음은 모두 각 기능이 따로는 잘 될 수 있다.

- 로그인 가능
- 후기 작성 가능
- 댓글 작성 가능
- 태그 연결 가능

하지만 실제 사용자 흐름으로 연결하면:

- 로그인 후 후기 작성은 되는데 작성자 ID가 비어 있음
- 후기 삭제 후 댓글 조회가 이상하게 동작
- 태그 필터 결과와 상세 조회 응답 구조가 다름

같은 문제가 나타날 수 있다.
즉 통합 문제는 기능 단위 개발만 해서는 잘 안 보인다.

## 핵심 키워드

### `testing`

정의:

테스트는 코드가 기대한 방식으로 동작하는지 검증하는 활동이다.
단순히 버그를 찾는 것보다, 변경 이후 회귀를 빠르게 감지하는 장치에 가깝다.

언제 중요한가:

항상 중요하지만, 통합 점검 단계에서 특히 중요하다.
여러 기능을 붙이는 순간 "한 기능 수정이 다른 기능을 깨뜨리는" 일이 많아지기 때문이다.

면접 포인트:

"테스트는 품질 보증뿐 아니라 변경 비용을 줄이는 장치라고 생각합니다. 기능이 늘어날수록 통합 지점 테스트가 중요해집니다."

### `Unit, Integration, E2E`

정의:

- `Unit`: 작은 함수 / 모듈 단위
- `Integration`: 여러 모듈이 함께 동작하는지 확인
- `E2E`: 실제 사용자 흐름 전체를 테스트

언제 각각 쓰는가:

- Unit: DTO 검증 함수, 변환 함수, 작은 서비스 로직
- Integration: 인증 + 후기 저장 + DB 저장 흐름
- E2E: 회원가입 -> 로그인 -> 후기 작성 -> 조회

이 단계에서 특히 중요한 것:

Integration과 E2E다.
각 기능이 연결되는 지점을 봐야 하기 때문이다.

면접 포인트:

"Unit 테스트는 로직 단위 신뢰를 만들고, Integration 테스트는 접합부 신뢰를 만들고, E2E는 사용자 관점 신뢰를 만든다고 봅니다."

### `test framework`

정의:

테스트를 작성하고 실행하고 결과를 보여주는 도구다.

왜 중요한가:

프레임워크 선택 자체보다 중요한 것은 "어떤 시나리오를 자동화할 것인가"다.
특히 인증, 후기 CRUD, 검색처럼 사용자 핵심 흐름을 먼저 잡는 편이 좋다.

면접 포인트:

"테스트 프레임워크는 수단이고, 중요한 것은 회귀 위험이 큰 사용자 흐름을 먼저 자동화하는 전략이라고 생각합니다."

### `logging`

정의:

로깅은 요청, 처리 결과, 에러를 기록하는 행위다.

통합 점검에서 왜 중요한가:

테스트가 실패했을 때 어디서 틀어졌는지 빠르게 찾게 해준다.
예를 들어:

- 토큰 생성은 됐는지
- DB insert는 성공했는지
- 검색 쿼리 조건은 무엇이었는지

를 추적할 수 있다.

면접 포인트:

"통합 문제는 로그 없이는 추적 비용이 급격히 커집니다. 그래서 테스트와 로그는 같이 가야 한다고 생각합니다."

### `Reliability`

정의:

신뢰성은 시스템이 예측 가능한 방식으로 동작하는 성질이다.

언제 중요해지는가:

기능이 많아질수록 "대부분 된다"는 말이 의미가 없어지고,
"예외 상황에서도 일관되게 실패한다"가 더 중요해진다.

예:

- 잘못된 토큰은 항상 401
- 없는 리소스는 항상 404
- 권한 없는 수정은 항상 403

면접 포인트:

"신뢰성은 성공률만이 아니라 실패의 일관성까지 포함한다고 생각합니다. 통합 점검 단계에서는 실패 방식의 예측 가능성이 특히 중요합니다."

### `error handling`

정의:

에러 핸들링은 실패를 구조적으로 표현하는 방식이다.

통합 단계에서 왜 중요한가:

기능마다 에러 응답 형식이 다르면 프론트 처리도 어려워지고, 디버깅도 느려진다.
따라서 공통 에러 포맷과 상태 코드 기준이 필요하다.

면접 포인트:

"기능이 많아질수록 에러 응답 일관성이 UX와 운영 효율 모두에 영향을 줍니다. 그래서 공통 예외 정책이 중요합니다."

### `configuration`

정의:

환경별로 바뀌는 설정을 관리하는 체계다.

통합 단계에서 왜 다시 중요해지는가:

개별 기능 테스트는 로컬 기본값으로 돌아가더라도,
통합 흐름에서는 DB URL, JWT_SECRET, CORS origin, 프론트 API base URL 같은 설정 차이가 문제를 만들기 쉽다.

면접 포인트:

"통합 테스트에서 가장 흔한 문제 중 하나가 설정 불일치입니다. 그래서 설정은 기능 코드보다 먼저 의심하는 편입니다."

## 면접처럼 정리하는 한 문단 요약

통합 점검 단계는 기능을 더 만드는 단계가 아니라, 이미 만든 기능이 하나의 제품 경험으로 이어지는지 확인하는 단계다.
그래서 Unit보다 Integration / E2E 비중이 커지고, 로그, 설정, 에러 응답 일관성이 특히 중요해진다.
실무적으로는 이 단계를 거쳐야 이후 상위 기능 문제를 기초 기능 문제와 구분해서 볼 수 있다.
