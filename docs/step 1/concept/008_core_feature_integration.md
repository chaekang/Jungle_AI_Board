# 008 핵심 기능 통합 개념

## 문서 목적

기능을 각각 만들고 끝내는 것이 아니라 테스트, 로깅, 에러 처리, 설정 관리를 통해 하나의 서비스로 안정화하는 방법을 이해한다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- 프론트 기능 통합은 `apps/web-react/src/features`와 `apps/web-react/src/shared` 경계를 지킨다.
- 백엔드 기능 통합은 Nest module 간 의존성을 확인한다.
- 테스트는 위험도에 맞춰 unit, integration, E2E를 나눠 작성한다.
- 운영에 가까운 검증에서는 로그와 에러 메시지를 함께 본다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `testing`

#### 개념

테스트는 코드가 기대한 동작을 계속 만족하는지 자동 또는 수동으로 확인하는 활동이다.

여기서 중요한 건 `testing`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 핵심 기능 통합 개념에서 인증, 후기 CRUD, 검색 필터, 좌석 선택 흐름의 회귀를 막는다.
그래서 문서를 읽을 때도 먼저 "변경한 기능의 성공 경로와 실패 경로를 모두 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

핵심 기능 통합 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `testing`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Unit, Integration, E2E를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/web-react/src/features`이다.
그 다음으로 `백엔드 기능 통합은 Nest module 간 의존성을 확인한다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `테스트는 위험도에 맞춰 unit, integration, E2E를 나눠 작성한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 후기 작성 버튼을 여러 번 눌러도 중복 저장되지 않는지 테스트할 수 있다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `testing`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// testing를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `testing`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Unit | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Integration | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| E2E | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

빌드가 통과했다는 사실을 기능 테스트 완료로 착각하면 사용자 흐름 버그를 놓친다.
또 하나의 실수는 `testing`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 변경한 기능의 성공 경로와 실패 경로를 모두 확인한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`testing`는 핵심 기능 통합 개념에서 인증, 후기 CRUD, 검색 필터, 좌석 선택 흐름의 회귀를 막는다.
예시는 후기 작성 버튼을 여러 번 눌러도 중복 저장되지 않는지 테스트할 수 있다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 빌드가 통과했다는 사실을 기능 테스트 완료로 착각하면 사용자 흐름 버그를 놓친다는 점이다.
Unit, Integration, E2E와 비교하면 책임 범위가 더 선명해진다.

### `Integration test`

### `E2E test`

#### 개념

| Logging | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

catch에서 에러를 삼켜 버리면 사용자는 실패를 모르고 개발자는 원인을 잃는다.
또 하나의 실수는 `error handling`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 에러가 발생했을 때 누가 무엇을 할 수 있는지 응답이 알려주는지 본다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`error handling`는 핵심 기능 통합 개념에서 폼 검증 실패, 네트워크 실패, 권한 없음, 서버 예외를 각각 다르게 처리한다.
예시는 후기 삭제 실패 시 화면에 메시지를 보여주고 콘솔이 아닌 서버 로그에 원인을 남긴다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 catch에서 에러를 삼켜 버리면 사용자는 실패를 모르고 개발자는 원인을 잃는다는 점이다.
Exception, HTTP Status, Logging와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `error handling`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `configuration`

#### 개념

설정은 실행 환경에 따라 달라지는 값과 동작 옵션을 코드 밖에서 관리하는 방식이다.

여기서 중요한 건 `configuration`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 핵심 기능 통합 개념에서 DATABASE_URL, JWT_SECRET, API base URL 같은 값이 환경별 설정에 해당한다.
그래서 문서를 읽을 때도 먼저 "개발, 테스트, 운영 환경에서 필요한 설정 값이 분리되어 있는지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

핵심 기능 통합 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `configuration`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Environment Variable, Secret, Config Module를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/web-react/src/features`이다.
그 다음으로 `백엔드 기능 통합은 Nest module 간 의존성을 확인한다.`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `테스트는 위험도에 맞춰 unit, integration, E2E를 나눠 작성한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 로컬 DB 주소와 배포 DB 주소는 코드가 아니라 .env로 분리한다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `configuration`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// configuration를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `configuration`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
