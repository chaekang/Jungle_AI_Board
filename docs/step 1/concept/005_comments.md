# 005 댓글 개념

## 문서 목적

댓글 기능에서 실시간 반영이 정말 필요한지 판단하고, REST, WebSocket, SSE, 보안, 로깅의 역할을 나눈다.

참고한 형식: Velog 알고리즘 시리즈처럼 정의, 특징, 동작 방식, 예시, 주의점, 정리 순서로 키워드를 하나씩 풀어 쓴다.

## 현재 파일 경로 규칙

- Nest 댓글 도메인은 `apps/nest-api/src/comments`에 둔다.
- React 댓글 API, 훅, 컴포넌트는 `apps/web-react/src/features/comments`에 둔다.
- 댓글이 후기와 연결될 때는 후기 id와 작성자 id를 함께 검증한다.
- 실시간 기능은 기본 CRUD가 안정된 뒤 요구사항에 맞춰 붙인다.

## 읽는 방법

- 각 키워드는 Velog 알고리즘 글처럼 정의, 필요성, 동작 방식, 예시, 주의점, 정리 순서로 읽는다.
- 번호 문장을 억지로 외우지 말고, 소제목 아래의 흐름을 따라가며 개념을 이해한다.
- 비슷한 키워드는 저장 위치, 책임 주체, 데이터 흐름, 실패 방식을 기준으로 비교한다.
- 구현할 때는 각 키워드의 질문과 흔한 실수 항목을 체크리스트처럼 사용한다.

## 핵심 키워드

### `Realtime`

#### 개념

실시간은 데이터 변화가 사용자에게 거의 즉시 반영되는 성질이나 경험을 뜻한다.

여기서 중요한 건 `Realtime`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 댓글 개념에서 댓글이나 후기 반응이 다른 사용자 화면에 바로 보이게 할 때 필요하다.
그래서 문서를 읽을 때도 먼저 "사용자가 몇 초 안에 알아야 하는 변화인지 먼저 판단한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

댓글 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `Realtime`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 WebSocket, SSE, Polling를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/comments`이다.
그 다음으로 `apps/web-react/src/features/comments`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `댓글이 후기와 연결될 때는 후기 id와 작성자 id를 함께 검증한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 누군가 댓글을 작성하면 새로고침 없이 목록에 추가되는 UX가 실시간이다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `Realtime`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// Realtime를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `Realtime`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| WebSocket | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| SSE | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Polling | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

실시간이 필요 없는 기능까지 WebSocket으로 만들면 복잡도와 운영 비용이 커진다.
또 하나의 실수는 `Realtime`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 사용자가 몇 초 안에 알아야 하는 변화인지 먼저 판단한다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`Realtime`는 댓글 개념에서 댓글이나 후기 반응이 다른 사용자 화면에 바로 보이게 할 때 필요하다.
예시는 누군가 댓글을 작성하면 새로고침 없이 목록에 추가되는 UX가 실시간이다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 실시간이 필요 없는 기능까지 WebSocket으로 만들면 복잡도와 운영 비용이 커진다는 점이다.
WebSocket, SSE, Polling와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `Realtime`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

### `SSE`

### `REST`

#### 개념

REST는 자원을 URI로 표현하고 HTTP 메서드로 행위를 나타내는 API 설계 스타일이다.

여기서 중요한 건 `REST`를 단어 뜻으로만 외우지 않는 것이다.
이 개념은 댓글 개념에서 후기 목록은 GET, 생성은 POST, 수정은 PATCH, 삭제는 DELETE로 표현한다.
그래서 문서를 읽을 때도 먼저 "URI는 명사 중심인지, 메서드가 동작을 표현하는지 확인한다"라는 질문을 붙여 두면 좋다.

#### 왜 필요한가

그 사이에서 `REST`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 HTTP Method, Resource, Status Code를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

댓글 개념는 화면, API, 데이터베이스가 한 번에 이어지는 기능이다.
그 사이에서 `logging`가 흐려지면 어디서 값을 만들고, 어디서 검증하고, 어디서 저장하는지 헷갈린다.
특히 좌석 후기 서비스는 극장, 작품, 공연, 좌석, 사용자, 후기처럼 비슷해 보이는 말이 많다.
이런 도메인에서는 Observability, Error Handling, Audit를 함께 놓고 차이를 보는 습관이 필요하다.
개념을 분리해 두면 나중에 화면이 바뀌거나 API가 바뀌어도 수정 범위를 작게 잡을 수 있다.

#### 이 프로젝트에서 보는 위치

먼저 볼 파일은 `apps/nest-api/src/comments`이다.
그 다음으로 `apps/web-react/src/features/comments`를 보면 같은 개념이 다른 계층에서 어떤 이름으로 이어지는지 보인다.
필요하면 `댓글이 후기와 연결될 때는 후기 id와 작성자 id를 함께 검증한다.`까지 따라가며 화면, 요청, 저장 규칙을 한 번에 확인한다.
예를 들어 후기 삭제가 실패했을 때 reviewId와 userId를 함께 남기면 원인을 찾기 쉽다.
이 예시는 추상적인 개념이 아니라 실제 좌석 후기 기능에서 사용자가 겪는 흐름과 연결된다.

#### 동작 방식

1. 사용자가 화면에서 `logging`와 관련된 값을 입력하거나 선택한다.
2. React는 그 값을 임시 상태로 들고 있다가 API 요청에 맞는 모양으로 바꾼다.
3. API 함수는 URL, method, body, query string 중 어디에 값을 넣을지 결정한다.
4. Nest Controller는 요청이 들어왔을 때 필요한 DTO나 파라미터를 꺼낸다.
5. Service는 그 값이 실제 도메인 규칙에 맞는지 판단한다.
6. Prisma나 DB 계층은 저장, 조회, 수정, 삭제가 데이터 무결성을 깨지 않는지 확인한다.
7. 응답이 돌아오면 화면은 성공, 실패, 빈 결과, 권한 없음 같은 상태를 다르게 보여준다.

#### 코드로 보는 감각

```ts
// logging를 구현할 때 머릿속에 두는 최소 흐름
const userInput = "화면에서 들어온 값";
const requestBody = validateAndShape(userInput);
const result = await apiCall(requestBody);

if (!result.ok) {
  showErrorMessage(result.reason);
}

updateScreen(result.data);
```

이 코드는 실제 파일에 그대로 붙이는 코드라기보다 `logging`를 읽을 때의 사고 순서다.
입력값을 그대로 믿지 않고, 요청 모양을 만들고, 실패를 처리하고, 화면을 갱신한다는 점을 보면 된다.

#### 비슷한 개념과 비교

| 비교 대상 | 구분해서 볼 지점 |
| --- | --- |
| Observability | 책임 주체가 화면인지, API인지, 데이터베이스인지 확인한다. |
| Error Handling | 저장되는 값인지, 계산되는 값인지, 화면에서만 쓰는 값인지 확인한다. |
| Audit | 실패했을 때 사용자에게 보일 결과가 달라지는지 확인한다. |

#### 흔한 실수

비밀번호나 토큰 같은 민감 정보를 로그에 남기면 보안 사고가 된다.
또 하나의 실수는 `logging`를 모든 계층에서 같은 방식으로 처리한다고 착각하는 것이다.
프론트엔드에서는 사용자의 입력 경험이 중요하고, 백엔드에서는 신뢰할 수 없는 요청을 막는 것이 중요하다.
DB에서는 같은 데이터가 두 번 저장되거나 관계가 끊어지는 문제를 막아야 한다.
따라서 한 줄 정의를 외우는 것보다 계층마다 책임이 어떻게 달라지는지 보는 편이 더 정확하다.

#### 구현할 때 확인할 질문

- 문제를 재현하지 않아도 로그만으로 흐름을 따라갈 수 있는지 본다?
- 이 값은 사용자가 직접 보내는 값인가, 서버가 계산하는 값인가?
- 로그인하지 않은 사용자도 접근해도 되는가?
- 실패했을 때 화면에 어떤 문장을 보여줄 것인가?
- 나중에 수정하거나 삭제할 때 참조가 깨지지 않는가?

#### 정리

`logging`는 댓글 개념에서 로그인 실패, API 오류, AI 추천 실패, DB 예외를 분석하는 근거가 된다.
예시는 후기 삭제가 실패했을 때 reviewId와 userId를 함께 남기면 원인을 찾기 쉽다처럼 실제 기능 흐름에서 찾아야 한다.
주의점은 비밀번호나 토큰 같은 민감 정보를 로그에 남기면 보안 사고가 된다는 점이다.
Observability, Error Handling, Audit와 비교하면 책임 범위가 더 선명해진다.
마지막으로 `logging`를 설명할 수 있다는 건 정의, 예시, 실패 사례, 구현 위치를 같이 말할 수 있다는 뜻이다.

- 현재 프로젝트의 실제 파일 경로와 연결해 말할 수 있는가?
- 프론트엔드, 백엔드, 데이터베이스 중 어느 계층의 책임인지 구분할 수 있는가?
- 기능이 커졌을 때 이 개념이 어디에서 병목이나 기준점이 되는지 예상할 수 있는가?
