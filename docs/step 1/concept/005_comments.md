# 005_comments_concepts

## 이 문서의 역할

이 문서는 `005 댓글 기능` 단계와 연결되는 개념을 자세히 정리한다.
댓글은 부가 기능처럼 보이지만, 실제로는 사용자 상호작용, 권한, 정렬, 실시간성, 운영 이슈가 모두 모이는 기능이다.

좌석 후기 서비스에서 댓글은 단순 "한 줄 감상"이 아니라,
기존 후기를 보강하고 질문을 쌓고 맥락을 확장하는 장치가 된다.
그래서 댓글 설계가 약하면 후기 상세 페이지의 정보 밀도가 낮아지고, 사용자 커뮤니티성도 약해진다.

## 댓글을 설계할 때 봐야 할 관점

댓글은 CRUD만 보면 쉬워 보이지만 실제로는 다음을 같이 봐야 한다.

- 누구나 볼 수 있는가
- 누가 썼는가
- 수정 / 삭제 권한은 누구에게 있는가
- 정렬은 어떤 기준인가
- 대댓글을 둘 것인가
- 실시간성이 필요한가

즉 댓글은 게시판 경험을 완성하는 UX 기능이면서, 동시에 권한과 데이터 흐름을 검증하는 기능이다.

## 핵심 키워드

### `Realtime`

정의:

실시간성은 다른 사용자의 행동이 새로고침 없이 빠르게 화면에 반영되는 성질이다.

댓글에서 왜 자주 언급되는가:

댓글은 사용자 간 상호작용이 직접 드러나는 영역이라,
새 댓글이 생겼을 때 즉시 보이면 체감 품질이 올라간다.

언제 쓰는가:

- 커뮤니티
- 채팅
- 협업 도구
- 라이브 피드

이 프로젝트에서의 현실적인 판단:

초기 MVP에서는 반드시 실시간일 필요는 없다.
댓글 수동 새로고침 또는 페이지 재조회로도 충분하다.
하지만 나중에 사용자 활동성이 높아지면 실시간 전략을 붙이기 쉬운 대표 영역이다.

장점:

- 사용자 체감 반응성이 좋아진다
- 상호작용이 살아 있는 서비스처럼 느껴진다

문제 / 주의점:

- 구현과 운영이 복잡해진다
- 연결 수 관리와 예외 상황 처리가 필요하다

면접 포인트:

"실시간은 기능의 본질이 아니라 UX 향상 수단이라고 생각합니다. 그래서 초기 MVP에서는 CRUD를 먼저 안정화하고, 댓글처럼 체감 효과가 큰 영역부터 점진적으로 붙이는 전략이 현실적입니다."

### `WebSocket`

정의:

클라이언트와 서버가 연결을 계속 유지하며 양방향으로 메시지를 주고받는 통신 방식이다.

언제 쓰는가:

- 채팅
- 실시간 알림
- 다자간 협업
- 빠른 양방향 피드백이 필요한 기능

댓글 기능에서의 의미:

댓글이 달리면 서버가 연결된 사용자에게 바로 푸시할 수 있다.
즉 "새 댓글 확인"을 폴링 없이 구현할 수 있다.

장점:

- 양방향 통신에 강하다
- 지연이 적다
- 실시간 UX 구현에 적합하다

문제 / 주의점:

- 연결 유지 비용이 있다
- 인증과 재연결 로직이 필요하다
- 작은 프로젝트 초반에는 과한 선택일 수 있다

면접 포인트:

"WebSocket은 강력하지만 연결 관리와 운영 비용이 있습니다. 댓글 기능처럼 단순 알림 수준이면 SSE나 폴링이 더 실용적일 수도 있습니다."

### `SSE`

정의:

Server-Sent Events는 서버가 클라이언트에게 단방향으로 이벤트를 푸시하는 방식이다.

언제 쓰는가:

- 서버에서 클라이언트로만 알림을 보내면 충분할 때
- 구현을 WebSocket보다 단순하게 가져가고 싶을 때

댓글 기능에서의 의미:

새 댓글 알림이나 새 댓글 리스트 갱신 같은 단방향 업데이트에 잘 맞을 수 있다.

장점:

- WebSocket보다 단순한 경우가 많다
- 단방향 알림에는 충분하다

문제 / 주의점:

- 양방향 통신은 적합하지 않다
- 브라우저 / 인프라 환경에 따라 고려할 점이 있다

면접 포인트:

"댓글 실시간화가 꼭 양방향일 필요는 없기 때문에, 단방향 이벤트만 필요하면 SSE가 더 단순한 대안이 될 수 있다고 생각합니다."

### `Rest`

정의:

댓글 기본 기능은 여전히 REST API로 시작하는 것이 가장 단순하다.

예시:

- `POST /seat-reviews/:reviewId/comments`
- `GET /seat-reviews/:reviewId/comments`
- `PATCH /comments/:id`
- `DELETE /comments/:id`

언제 유용한가:

- 실시간화 이전 단계
- 서버 구조를 단순하게 유지하고 싶을 때

장점:

- 테스트가 쉽다
- 프론트 구현이 단순하다
- 권한 체크가 명확하다

문제 / 주의점:

- 새로고침 없는 즉시 반영은 별도 전략이 필요하다

면접 포인트:

"실시간 기능이 필요한 서비스라도 기본 도메인 조작은 먼저 REST로 안정화하는 편이 좋습니다. 댓글도 마찬가지라고 생각합니다."

### `security`

정의:

댓글 보안에서 핵심은 인증과 권한이다.

중요한 규칙:

- 로그인한 사용자만 작성
- 작성자 본인만 수정 / 삭제
- 필요하면 신고 / 블라인드 전략 별도 고려

문제 / 주의점:

- 후기 권한 패턴과 댓글 권한 패턴이 다르면 혼란이 생긴다
- 소유권 검사를 Guard에서 끝내려 하면 안 된다

면접 포인트:

"댓글은 후기보다 가볍게 보여도 권한 구조는 동일하게 엄격해야 합니다. 인증과 소유권 검사를 분리하는 것이 중요합니다."

### `logging`

정의:

로그는 시스템 내부에서 어떤 요청이 어떻게 처리되었는지 남기는 기록이다.

댓글에서 왜 중요한가:

- 수정 / 삭제 분쟁이 생길 수 있다
- 대량 스팸이나 이상 패턴을 추적할 수 있다

언제 쓰는가:

- 운영 이슈 분석
- 에러 추적
- 감사 로그

문제 / 주의점:

- 민감한 개인정보를 과도하게 로그로 남기면 안 된다
- 모든 것을 다 찍으면 노이즈가 심해진다

면접 포인트:

"로깅은 디버깅용과 운영 감사용을 나눠 생각하는 편입니다. 댓글 기능은 사용자 상호작용이 많아서 최소한 생성 / 수정 / 삭제 흐름은 추적 가능해야 한다고 봅니다."

## 면접처럼 정리하는 한 문단 요약

댓글 기능은 단순 서브 기능이 아니라, 인증 / 권한 구조를 반복 검증하고 사용자 상호작용을 확장하는 핵심 요소다.
초기에는 REST CRUD와 권한 체크를 안정적으로 만들고, 이후 필요에 따라 Realtime, WebSocket, SSE를 검토하는 접근이 현실적이다.
실무적으로는 "실시간 댓글"보다 "권한과 정렬이 일관된 댓글"을 먼저 만드는 것이 더 중요하다.
