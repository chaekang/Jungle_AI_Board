# 004_seat_review_crud_concepts

## 이 문서의 역할

이 문서는 `004 좌석 후기 CRUD` 단계에서 알아야 하는 API / 서버 설계 개념을 자세히 설명한다.
후기 CRUD는 단순한 게시글 추가 기능처럼 보이지만, 실제로는 인증, 권한, 데이터 모델링, 응답 구조, 예외 처리, 트랜잭션이 모두 만나는 지점이다.

이 단계가 안정적이어야 이후 댓글, 태그, 검색, RAG가 자연스럽게 붙는다.
즉 CRUD는 단순 기본기라기보다 이후 확장 기능의 기반 계층이라고 보는 편이 맞다.

## CRUD를 단순하게 보면 놓치기 쉬운 것

많은 초보 프로젝트에서 CRUD는 "데이터 넣고 읽고 고치고 지우면 끝"처럼 다뤄진다.
하지만 이 프로젝트에서는 다음이 같이 중요하다.

- 작성자 연결
- 수정 / 삭제 권한
- 구조화된 응답
- BigInt 직렬화
- FK 존재 여부 검증
- 선택 메타데이터와 필수 메타데이터의 균형

즉 후기 CRUD는 단순 DB 조작이 아니라 도메인 규칙을 담는 계층이다.

## 핵심 키워드

### `CRUD`

정의:

CRUD는 `Create`, `Read`, `Update`, `Delete`의 약자다.
대부분의 데이터 기반 서비스가 가장 먼저 갖추는 기본 기능 집합이다.

언제 쓰는가:

- 회원 관리
- 후기 관리
- 댓글 관리
- 태그 관리

이 프로젝트에서의 의미:

후기 CRUD는 서비스 핵심이다.
좋은 좌석 후기 데이터를 안정적으로 쌓는 기능이므로, 이후 AI 기능보다 오히려 더 중요할 수 있다.

장점:

- 기능 범위가 명확하다
- 테스트 시나리오를 만들기 쉽다
- 단계별 개발에 적합하다

문제 / 주의점:

- 너무 기계적으로 만들면 도메인 규칙이 빠진다
- Create / Update / Delete에 권한과 검증을 빼먹기 쉽다

면접 포인트:

"CRUD는 단순 기능이 아니라 도메인 규칙을 어디에 녹일지 보여주는 기본기라고 생각합니다. 특히 작성자 권한과 응답 구조 일관성이 중요합니다."

### `MVC`

정의:

MVC는 Model, View, Controller로 역할을 나누는 구조다.
NestJS가 전통적인 MVC와 완전히 같진 않지만, 역할 분리 관점에서 이해에 도움이 된다.

현재 프로젝트에 대응시키면:

- Controller: HTTP 요청 진입점
- Service: 비즈니스 로직
- Prisma / DB 모델: 데이터 접근

언제 중요한가:

CRUD를 구현할 때 "어떤 코드를 어디에 둘 것인가"가 계속 등장한다.
그때 MVC나 계층 분리 감각이 없으면 코드가 금방 섞인다.

장점:

- 역할이 명확하다
- 코드 찾기가 쉽다
- 테스트가 쉬워진다

문제 / 주의점:

- Controller에 로직이 많아지면 구조가 무너진다
- Service가 모든 것을 다 떠안으면 거대한 God object가 된다

면접 포인트:

"저는 컨트롤러는 얇게, 서비스에는 도메인 규칙을 모으는 쪽을 선호합니다. 그래야 CRUD가 늘어나도 구조가 덜 무너집니다."

### `Rest`

정의:

REST는 리소스를 중심으로 URL과 메서드를 설계하는 방식이다.

후기 CRUD의 대표 예:

- `POST /seat-reviews`
- `GET /seat-reviews`
- `GET /seat-reviews/:id`
- `PATCH /seat-reviews/:id`
- `DELETE /seat-reviews/:id`

언제 유용한가:

- 리소스가 명확할 때
- 프론트와 API 계약을 단순하게 유지하고 싶을 때

장점:

- 구조가 직관적이다
- 프론트와 백엔드가 예측 가능하게 협업할 수 있다

문제 / 주의점:

- 필터와 검색이 복잡해지면 URL 설계가 길어질 수 있다
- "REST처럼 보이지만 실제로는 규칙이 제각각"인 상태가 되기 쉽다

면접 포인트:

"REST 설계에서 중요한 것은 URL 모양보다도 리소스 책임과 응답 일관성이라고 생각합니다."

### `transaction`, `isolation`, `lock/deadlock`

정의:

- `transaction`: 여러 DB 작업을 하나의 논리적 묶음으로 처리
- `isolation`: 동시에 실행되는 작업 간 간섭 제어 수준
- `lock`: 충돌 방지를 위한 잠금
- `deadlock`: 서로 잠금을 기다리며 멈춘 상태

언제 중요한가:

후기 저장만 보면 단순해 보이지만, 실제로는:

- 후기 저장
- 태그 연결
- 문서 테이블 생성
- 임베딩 작업 예약

같은 작업이 묶일 수 있다.

즉 나중으로 갈수록 transaction 설계가 중요해진다.

장점:

- 부분 저장을 막을 수 있다
- 데이터 정합성을 유지한다

문제 / 주의점:

- 트랜잭션 범위를 너무 넓히면 성능이 나빠질 수 있다
- 동시성 높은 환경에서는 lock 경합이 커질 수 있다

면접 포인트:

"트랜잭션은 많을수록 좋은 게 아니라 꼭 같이 성공해야 하는 작업만 묶는 게 중요하다고 생각합니다."

### `error handling (HTTP 3xx, 4xx, 5xx)`

정의:

에러 핸들링은 실패 상황을 일관된 상태 코드와 메시지로 표현하는 것이다.

후기 CRUD에서 자주 쓰는 코드:

- `400`: 입력 검증 실패
- `401`: 로그인 필요
- `403`: 작성자 아님
- `404`: 후기 없음
- `500`: 서버 오류

왜 중요한가:

프론트는 상태 코드에 따라 다른 UI를 보여줄 수 있다.
예를 들어 `403`이면 "내 글만 수정 가능" 메시지를 띄울 수 있다.

면접 포인트:

"에러 핸들링은 디버깅 편의성뿐 아니라 UX 품질과도 연결됩니다. 특히 401, 403, 404를 명확히 나누는 게 중요합니다."

### `api design`

정의:

CRUD API 설계는 어떤 필드를 필수로 받고, 어떤 응답을 줄지 정하는 일이다.

이 프로젝트에서 중요한 포인트:

- 후기를 너무 무겁게 입력받지 않기
- 하지만 검색과 RAG에 필요한 핵심 메타데이터는 확보하기

즉 "입력 장벽"과 "데이터 품질" 사이 균형이 중요하다.

면접 포인트:

"좋은 CRUD API는 모든 것을 입력받는 API가 아니라, 지금 필요한 핵심 데이터만 받으면서도 나중 확장에 필요한 구조를 남겨두는 API라고 생각합니다."

## 면접처럼 정리하는 한 문단 요약

후기 CRUD는 단순 저장 기능이 아니라 도메인 규칙을 실제로 표현하는 핵심 계층이다.
REST API 구조, Controller-Service 분리, 작성자 권한, 예외 처리, BigInt 직렬화, 트랜잭션 가능성까지 함께 봐야 하고,
실무적으로는 "되는 CRUD"보다 "권한과 응답 일관성이 있는 CRUD"를 만드는 것이 훨씬 중요하다.
