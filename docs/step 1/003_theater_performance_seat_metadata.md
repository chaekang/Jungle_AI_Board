# 003_theater_performance_seat_metadata

## 목적

이 문서는 [implementation_order.md](../implementation_order.md)의
`4. 극장 / 공연 / 좌석 메타데이터 구조 설계`를 실제 구현 관점에서 풀어쓴 작업 가이드다.

이 단계의 목표는 좌석 후기 CRUD를 만들기 전에,
후기가 기대고 설 수 있는 극장, 공연, 좌석 메타데이터의 기준을 먼저 고정하는 것이다.

관련 개념 문서:

- [docs/concept/003_theater_performance_seat_metadata.md](../concept/003_theater_performance_seat_metadata.md)

## 이 단계가 끝나면 되는 것

- 극장, 작품, 공연, 좌석 위치의 데이터 기준이 고정된다
- 후기 작성 DTO와 DB 스키마가 같은 기준으로 움직인다
- 검색 / 필터 / RAG에서 재사용할 수 있는 좌석 필드가 정리된다
- 이후 단계에서 "층, 구역, 열, 번호"를 다르게 해석하는 일이 줄어든다

## 이번 단계의 핵심 결정

### 1. 엔티티를 어디까지 나눌지 먼저 정한다

초기 MVP에서는 아래 정도면 충분하다.

- `theaters`
- `musicals`
- `performances`
- `seat_reviews`

권장 관계:

- 극장 1개는 여러 공연을 가진다
- 작품 1개는 여러 공연 회차를 가진다
- 후기 1개는 극장, 작품, 필요하면 특정 공연 회차를 참조한다

### 2. 좌석 위치 표준을 먼저 고정한다

후기 본문만 저장하면 나중에 검색과 필터가 약해진다.
최소한 아래 필드는 구조화해서 저장하는 편이 좋다.

- `seat_floor`
- `seat_section`
- `seat_row`
- `seat_number`

이번 단계 기준 권장 규칙:

- 층: 문자열형으로 통일하고 `B1`, `1F`, `2F` 같은 값을 허용
- 구역: 우선 정규화된 문자열 1개만 저장
- 열: `"A"`, `"B"`, `"C"`처럼 문자열 허용
- 번호: 숫자형 사용

즉 지금은 "원본값 + 정규화값 이중 저장"까지 바로 가지 않더라도,
검색과 입력이 흔들리지 않도록 하나의 표준 문자열 규칙은 먼저 정해야 한다.

이번 단계에서 DB를 복잡하게 늘리기보다
입력 단계에 `normalizeSeatLocation()` 같은 정규화 함수를 구현한다.
예를 들어 `1층`, `1F`, `1관 1층`처럼 흔들릴 수 있는 표현을 저장 전에 한 번 정리해두면
검색과 필터, RAG 품질이 나중에 훨씬 안정적이다.

### 3. 후기 평가 필드를 어디까지 구조화할지 정한다

후기 본문만으로도 정보는 남길 수 있지만,
AI 검색과 추천 정확도를 높이려면 일부 평가는 분리 저장하는 편이 좋다.

우선 후보:

- `view_rating`
- `sound_rating`
- `comfort_rating`
- `expression_rating`
- `stage_visibility_rating`
- `has_obstruction`
- `obstruction_note`
- `side_preference`

## 권장 구현 순서

1. `theaters`, `musicals`, `performances` 스키마 확정
2. 좌석 위치 표준 필드 확정
3. 후기 평가 필드 초안 확정
4. seed 데이터 기준 극장 2~3개 정리
5. 공연 회차는 `performances` 테이블로 유지하되, 후기에서는 optional 연결 허용
6. 후기 작성 API가 기대는 FK 구조 확정

## Prisma 설계 체크포인트

### `theaters`

- 이름
- 지역 또는 주소 요약

지금 단계에서는 "좌석 메타데이터용 확장 필드"를 추상적으로 열어두기보다,
필요한 필드가 생기면 명시적으로 컬럼을 추가하는 편이 초보자 기준으로 더 안전하다.

### `musicals`

- 작품명
- 설명
- 상태값(선택)

### `performances`

- `theater_id`
- `musical_id`
- 공연 일시
- 시즌 / 회차 / 캐스트 요약 필드 여부

### 후기와의 연결

초기에는 후기에서 아래를 직접 참조하면 충분하다.

- `theater_id`
- `musical_id`
- `performance_id`는 optional 가능

## 완료 기준

아래가 되면 이 단계는 충분히 끝난 것으로 볼 수 있다.

- DB 스키마 초안만 봐도 후기 작성 입력값이 떠오른다
- 후기 CRUD에서 어떤 FK와 메타데이터를 써야 하는지 명확하다
- 검색 / 필터 / RAG 단계에서 재사용할 좌석 필드가 빠지지 않았다
